## 1. Purpose

This Database Design document defines the **database schema, migrations, and data architecture** for [PROJECT_NAME]. It implements DGA-01 (database as a product service — only long-term-value data persisted), LOCK-06 (database optional — tools function without DB), and supports DGA-05 (content isolation), DGA-07 (audit logs), and DGA-10 (marketplace readiness).

The database is NOT the primary storage for the platform — the browser is (per LOCK-02). The database serves specific product capabilities: authentication, user profile, history, favorites, content, admin, analytics, audit. Each of these is a product service, not just persistence. This document defines the schema for each, organized by bounded context (`03_DDD`), with Row-Level Security (RLS) policies enforcing access control at the database layer.

## 3. Architectural Decisions

### AD-01 — Schema per Bounded Context

**Context.** `03_DDD` AD-01 establishes six bounded contexts. Database schemas must align with these boundaries to maintain context isolation.

**Decision.** Each bounded context owns a Postgres schema (namespace): `identity`, `content`, `platform_ops`, `billing`, `analytics`. No cross-schema foreign keys. References are by string ID. Migrations scoped per context.

**Implements:** DGA-05 (Content Architecture — isolation), LOCK-04 (Modular), EC-02 (One Source of Truth).

### AD-02 — Row-Level Security (RLS) on Every Table

**Context.** Application-level access control is bypassable if a query is malformed. RLS enforces access at the database layer, providing defense in depth.

**Decision.** Every table has RLS policies. Policies use `auth.uid()` (Supabase Auth) to scope row access. Public tables (e.g., published articles) have `USING (is_published = true)` policy. User-scoped tables (e.g., history) have `USING (user_id = auth.uid())`. Admin tables require admin role check.

**Implements:** EC-08 (Security by Default), DGA-07 (Auditability — admin access controlled).

### AD-03 — Drizzle ORM for Schema and Migrations

**Context.** `04_TechStack` AD-04 selects Drizzle ORM. Schema definitions and migrations must use Drizzle's format.

**Decision.** Schema defined in TypeScript using Drizzle's schema builder. Migrations are plain SQL files (generated by Drizzle Kit), reviewed in PRs, scoped per context under `drizzle/[context]/`.

**Implements:** `04_TechStack` AD-04, EC-02 (One Source of Truth — schema is code).

### AD-04 — Only Long-Term-Value Data Persisted (DGA-01)

**Context.** DGA-01 mandates the database is a product service, not default persistence. Temporary data should stay browser-local.

**Decision.** The database stores only:
- Authentication (users, sessions).
- User profile, history, favorites.
- Content (articles, media).
- Admin (feature flags, settings, audit).
- Analytics (events, aggregates).
- Billing (subscriptions, invoices) — Phase 2+.

The database does NOT store:
- Tool execution inputs/outputs (ephemeral, browser-local).
- Search indexes (generated from manifests).
- SEO metadata (in manifest).
- Tool manifests (in code, not DB).

**Implements:** DGA-01, LOCK-02 (Browser-First), LOCK-06 (Database Optional).

### AD-05 — Audit Logs are Immutable (DGA-07)

**Context.** DGA-07 mandates audit logs are immutable. Mutable audit logs lose trust.

**Decision.** `platform_ops.audit_entries` table is append-only. No UPDATE or DELETE permissions granted. RLS policy allows INSERT only. Retention: 7 years (compliance).

**Implements:** DGA-07, EC-08.

### AD-06 — Marketplace Preparation (DGA-10)

**Context.** DGA-10 mandates the architecture anticipates a future plugin marketplace. Schema must not block this.

**Decision.** Phase 1 schema includes:
- `identity.users` table supports `publisher` role (Phase 4).
- `identity.users` has `is_verified_publisher` boolean (default false).
- No first-party tool assumptions in schema.
- Analytics events attribute to `toolSlug` (not `firstPartyToolId`), allowing third-party tools.

Phase 4 adds: `marketplace.publishers`, `marketplace.ratings`, `marketplace.reviews`, `marketplace.collections` — but Phase 1 schema doesn't block these.

**Implements:** DGA-10, LOCK-05 (Plugin-Ready).

## 4. Design Principles

### P1 — Schema per Context
Each bounded context owns its schema. No cross-schema foreign keys.

### P2 — RLS on Everything
Defense in depth. Application-level checks + database-level RLS.

### P3 — Migrations are Code
Migrations are version-controlled SQL, reviewed in PRs, scoped per context.

### P4 — Minimal PII
Store only necessary PII. Anonymize where possible. GDPR-compliant deletion.

### P5 — Indexes for Query Patterns
Indexes designed for actual query patterns, not speculative.

### P6 — Forward-Compatible
Schema doesn't block future capabilities (multi-tenancy, marketplace).

## 5. Schema Definitions

### 5.1 Identity Context

```typescript
// src/identity/infrastructure/schema.ts

import { pgSchema, uuid, text, timestamp, boolean, integer, jsonb } from 'drizzle-orm/pg-core';

export const identitySchema = pgSchema('identity');

// ─── Users ──────────────────────────────────────────────
export const users = identitySchema.table('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: text('email').notNull().unique(),
  emailVerified: timestamp('email_verified'),
  role: text('role').notNull().default('user'), // guest|user|premium|editor|admin|super_admin
  isVerifiedPublisher: boolean('is_verified_publisher').notNull().default(false), // DGA-10
  profile: jsonb('profile').$type<UserProfile>(),
  createdAt: timestamp('created_at').notNull().defaultNow(),
  updatedAt: timestamp('updated_at').notNull().defaultNow(),
  deletedAt: timestamp('deleted_at'), // Soft delete (GDPR)
});

// ─── Sessions (Supabase Auth manages, but we reference) ────
export const sessions = identitySchema.table('sessions', {
  id: uuid('id').primaryKey(),
  userId: uuid('user_id').notNull().references(() => users.id),
  expiresAt: timestamp('expires_at').notNull(),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});

// ─── History Entries ────────────────────────────────────
export const historyEntries = identitySchema.table('history_entries', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id').notNull().references(() => users.id),
  toolSlug: text('tool_slug').notNull(), // Reference, not FK (Tools Context is stateless)
  toolCategory: text('tool_category').notNull(),
  inputSummary: jsonb('input_summary'), // Anonymized summary
  outputSummary: jsonb('output_summary'),
  executedAt: timestamp('executed_at').notNull().defaultNow(),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});

// ─── Favorites ──────────────────────────────────────────
export const favorites = identitySchema.table('favorites', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id').notNull().references(() => users.id),
  toolSlug: text('tool_slug').notNull(),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});
```

**RLS Policies:**
```sql
-- users: users can read/update own row; admins can read all
CREATE POLICY "users_select_own" ON identity.users FOR SELECT USING (id = auth.uid());
CREATE POLICY "users_update_own" ON identity.users FOR UPDATE USING (id = auth.uid());

-- history_entries: users can CRUD own entries only
CREATE POLICY "history_select_own" ON identity.history_entries FOR SELECT USING (user_id = auth.uid());
CREATE POLICY "history_insert_own" ON identity.history_entries FOR INSERT WITH CHECK (user_id = auth.uid());
CREATE POLICY "history_delete_own" ON identity.history_entries FOR DELETE USING (user_id = auth.uid());

-- favorites: users can CRUD own favorites only
CREATE POLICY "favorites_select_own" ON identity.favorites FOR SELECT USING (user_id = auth.uid());
CREATE POLICY "favorites_insert_own" ON identity.favorites FOR INSERT WITH CHECK (user_id = auth.uid());
CREATE POLICY "favorites_delete_own" ON identity.favorites FOR DELETE USING (user_id = auth.uid());
```

### 6.1 Migration Files

```
├── analytics/
│   └── 0000_initial.sql
```

### 6.2 Migration Workflow

1. Modify schema in `src/[context]/infrastructure/schema.ts`.
2. Run `pnpm drizzle-kit generate` — produces SQL migration.
3. Review generated SQL.
4. Run `pnpm drizzle-kit migrate` locally to test.
5. Commit schema + migration files.
6. CI runs migrations on staging; PR review verifies.
7. Production migration runs on deploy.

### 6.3 Zero-Downtime Migrations

- **Additive changes** (new column, new table): safe, zero-downtime.
- **Breaking changes** (drop column, rename): multi-step:
  1. Add new column alongside old.
  2. Deploy code writing to both.
  3. Backfill new column from old.
  4. Deploy code reading from new only.
  5. Drop old column.

| Data Type | Retention | Reason |
|-----------|-----------|--------|
| Users | Until account deletion (GDPR) | User data ownership |
| History entries | 1 year | User utility; privacy |
| Favorites | Until user deletion | User data ownership |

| Analytics raw events | 90 days | Privacy; sufficient for analysis |
| Analytics aggregates | Indefinite | Growth tracking |
| Media assets | Indefinite (until orphaned) | Content |

## 8. Standards

### 8.1 Schema Standards
- Every table has `id` (UUID), `created_at`, `updated_at` (where applicable).
- Soft deletes via `deleted_at` for user data (GDPR).
- No cross-schema foreign keys.
- Snake_case table and column names.

### 8.2 RLS Standards
- Every table has RLS policies.
- Default deny; explicit allow.
- User-scoped tables use `auth.uid()`.
- Admin tables require role check.

### 8.3 Migration Standards
- Migrations scoped per context.
- Migrations are plain SQL.
- Migrations reviewed in PRs.
- Zero-downtime for breaking changes.

### 8.4 Indexing Standards
- Indexes for actual query patterns, not speculative.
- Composite indexes for multi-column filters.
- Monitor index usage; drop unused.

## 9. Best Practices

### 9.1 When Adding a Table
1. Identify which bounded context it belongs to.
2. Add to context's `schema.ts`.
3. Generate migration.
4. Add RLS policies.
5. Add appropriate indexes.
6. Update this document via PR.

### 9.2 When Modifying a Table
- Additive: safe, generate migration.
- Breaking: follow zero-downtime pattern (§6.3).

### 9.3 When Writing Queries
- Use Drizzle's query builder (type-safe).
- Never raw SQL string concatenation (SQL injection).
- Verify RLS policies don't block legitimate access.
