    11→    11→## 1. Purpose
    12→    12→
    13→    13→This Project Structure document defines the high-level layout of [PROJECT_NAME]'s codebase. It maps the bounded contexts from `03_DDD` and the architectural layers from `02_SAD` to physical directories. It defines the **Tool Registry Pattern** that enables plugin-ready architecture (LOCK-05), the **Module Boundaries** that enforce modularity (LOCK-04), and the **Stateless Tool Layout** that enables database-optional operation (LOCK-06).
    14→    14→
    15→    15→This document is the authoritative reference for "where does this code belong?" Every new file, every new module, every new tool must be placed according to this document. The structure is not a suggestion; it is enforced via ESLint boundary rules, code review, and CI checks.
    16→    16→
    17→    17→The structure scales from 30 tools (Phase 1) to 1,000+ tools (Phase 4) without reorganization. The unit of growth is the tool folder; adding a tool creates one new folder and modifies nothing outside the tools directory except auto-generated registry files. This linear scaling property is the single most important structural guarantee.

    40→    40→## 3. Architectural Decisions
    41→    41→
    42→    42→### AD-01 — Feature-Based Architecture at the Top Level
    43→    43→
    44→    44→**Context.** Two common patterns exist: type-based (folders for `components/`, `hooks/`, `utils/`) and feature-based (folders for `tools/`, `auth/`, `admin/`). Type-based breaks down at scale because related files scatter across many folders; changing one feature requires touching many directories.
    45→    45→
    46→    46→**Decision.** Adopt feature-based architecture at the top level. Each bounded context (`03_DDD`) is a top-level directory. Within each context, files are organized by architectural layer (`02_SAD` AD-01: Presentation, Application, Domain, Infrastructure).
    47→    47→
    48→    48→```
    49→    49→src/
    50→    50→├── app/                    # Next.js App Router (routing layer)
    51→    51→├── tools/                  # Tools Context (core domain)
    52→    52→├── identity/               # Identity Context
    53→    53→├── content/                # Content Context
    54→    54→├── platform-ops/           # Platform Operations Context (Admin)
    55→    55→├── billing/                # Billing Context (Phase 2+)
    56→    56→├── analytics/              # Analytics Context
    57→    57→├── shared/                 # Cross-cutting shared code
    58→    58→└── generated/              # Build-time generated files (registry, etc.)
    59→    59→```
    60→    60→
    61→    61→**Consequences.**
    62→    62→- ✅ Each context is self-contained; team can own a context end-to-end.
    63→    63→- ✅ Tool count scales linearly — each tool is a folder under `tools/`.
    64→    64→- ✅ Boundary enforcement is structural (folder-per-context) + tooling (ESLint).
    65→    65→- ⚠️ Some shared utilities needed across contexts; mitigated by `/shared` and `/packages`.
    66→    66→
    67→    67→**Implements:** LOCK-04 (Modular Architecture), EC-03 (Component Reuse First — shared code lives in `/packages` and `/src/shared`).
    68→    68→
    69→    69→### AD-02 — Layered Folders Within Each Context
    70→    70→
    71→    71→**Context.** `02_SAD` AD-01 mandates four layers (Presentation, Application, Domain, Infrastructure) with strict dependency rules. The folder structure must make these layers visible and enforceable.
    72→    72→
    73→    73→**Decision.** Within each bounded context directory, four subdirectories represent the layers:
    74→    74→
    75→    75→```
    76→    76→src/identity/
    77→    77→├── presentation/           # React components, pages (depends on Application)
    78→    78→├── application/            # Use cases, hooks, server actions (depends on Domain)
    79→    79→├── domain/                 # Entities, value objects, schemas (no external deps)
    80→    80→└── infrastructure/         # DB access, external APIs (depends on Domain)
    81→    81→```
    82→    82→
    83→    83→Dependency direction is enforced via ESLint `no-restricted-imports`:
    84→    84→- `presentation/` may import from `application/`, `domain/`, `shared/`.
    85→    85→- `application/` may import from `domain/`, `shared/`.
    86→    86→- `domain/` may import from `shared/` only.
    87→    87→- `infrastructure/` may import from `domain/`, `shared/`.
    88→    88→
    89→    89→**Consequences.**
    90→    90→- ✅ Layer boundaries are visible in any IDE file tree.
    91→    91→- ✅ ESLint catches violations at lint time, not review time.
    92→    92→- ✅ Domain logic is fully testable without React or DB setup.
    93→    93→- ⚠️ Slightly more nesting than flat structure; justified by enforceability.
    94→    94→
    95→    95→**Implements:** LOCK-04, `02_SAD` AD-01.
    96→    96→
    97→    97→### AD-03 — Tool Module Anatomy (One Folder Per Tool)
    98→    98→
    99→    99→**Context.** LOCK-04 mandates tools are independent modules. LOCK-05 mandates plugin-ready architecture. The structure must make adding a tool a self-contained operation: create a folder, define a manifest, implement stages.
   100→   100→
   101→   101→**Decision.** Each tool lives in `src/tools/[category]/[slug]/` and contains:
   102→   102→
   103→   103→```
   104→   104→src/tools/image/image-resize/
   105→   105→├── manifest.ts             # ToolManifest — the aggregate root (LOCK-05)
   106→   106→├── stages/
   107→   107→│   ├── input.ts            # InputStage implementation
   108→   108→│   ├── validation.ts       # ValidationStage (Zod schema)
   109→   109→│   ├── processing.ts       # ProcessingStage (browser-side or server-side)
   110→   110→│   ├── preview.ts          # PreviewStage (React component)
   111→   111→│   ├── download.ts         # DownloadStage (packaging logic)
   112→   112→│   ├── history.ts          # HistoryStage (optional; best-effort persistence)
   113→   113→│   └── share.ts            # ShareStage (optional; link/QR generation)
   114→   114→├── components/             # Tool-specific UI components (input forms, etc.)
   115→   115→├── hooks/                  # Tool-specific hooks
   116→   116→├── tests/
   117→   117→│   ├── stages.test.ts      # Unit tests per stage
   118→   118→│   └── e2e.test.ts         # E2E test for the tool workflow
   119→   119→├── README.md               # Tool-specific docs (optional, for complex tools)
   120→   120→└── index.ts                # Re-exports manifest as default
   121→   121→```
   122→   122→
   123→   123→**Consequences.**
   124→   124→- ✅ Adding a tool = adding a folder; nothing else changes (registry auto-discovers).
   125→   125→- ✅ Tool is fully isolated; deleting the folder removes the tool completely.
   126→   126→- ✅ Tests colocated with implementation.
   127→   127→- ✅ Plugin-ready (LOCK-05): a third-party tool follows the same structure.
   128→   128→- ⚠️ Many small folders; mitigated by clear naming and IDE support.
   129→   129→
   130→   130→**Implements:** LOCK-04, LOCK-05, LOCK-03 (Tool Engine stages map to files), EC-04 (Tool Template Standard — every tool has identical structure).
   131→   131→
   132→   132→### AD-04 — Tool Registry Pattern (Build-Time Codegen)
   133→   133→
   134→   134→**Context.** LOCK-05 mandates auto-discovery. Runtime registry lookups add latency and complicate Edge deployment. Build-time codegen produces static typed files that Edge can serve.
   135→   135→
   136→   136→**Decision.** A build script (`scripts/generate-registry.ts`) walks `src/tools/**/manifest.ts`, imports each manifest, validates it against the schema, and emits typed registry files:
   137→   137→
   138→   138→```
   139→   139→src/generated/
   140→   140→├── registry.ts             # Map<slug, ToolManifest> with helpers
   141→   141→├── navigation.ts           # Category → Tool[] for nav menus
   142→   142→├── sitemap.ts              # Sitemap entries for SEO
   143→   143→├── seo-meta.ts             # Per-route SEO metadata (LOCK-08)
   144→   144→├── admin-inventory.ts      # Tool list for admin panel
   145→   145→└── types.ts                # Generated TypeScript types
   146→   146→```
   147→   147→
   148→   148→The script runs:
   149→   149→- On every `pnpm dev` start (development).
   150→   150→- On every `pnpm build` (production).
   151→   151→- Via `pnpm gen:registry` (manual).
   152→   152→
   153→   153→CI verifies that generated files are committed and match the manifests; mismatched generations fail the build.
   154→   154→
   155→   155→**Consequences.**
   156→   156→- ✅ Zero runtime cost for registry lookups.
   157→   157→- ✅ Type safety: malformed manifests fail the build.
   158→   158→- ✅ Auto-discovery: new tools appear in nav, sitemap, admin without manual wiring.
   159→   159→- ✅ Plugin-ready: third-party manifests follow the same schema; can be merged into registry.
   160→   160→- ⚠️ Build time grows linearly with tool count; mitigated by parallel codegen at >500 tools.
   161→   161→
   162→   162→**Implements:** LOCK-05 (Plugin-Ready, auto-discovery), LOCK-08 (SEO metadata in registry).
   163→   163→
   164→   164→### AD-05 — Shared Code Split Between `/packages` and `/src/shared`
   165→   165→
   166→   166→**Context.** Some code is shared across bounded contexts (UI primitives, utility functions, types). Where it lives affects reusability and dependency direction.
   167→   167→
   168→   168→**Decision.** Two locations for shared code, with distinct purposes:
   169→   169→
   170→   170→**`/packages/*`** — published-style packages, fully self-contained, no dependencies on `src/`. Used for code that could theoretically be extracted to a separate npm package later. Examples:
   171→   171→- `/packages/tool-engine` — the `ToolEngine<TInput, TOutput>` abstraction (LOCK-03).
   172→   172→- `/packages/ui` — design system primitives (built on shadcn/ui, per `09_DesignSystem`).
   173→   173→- `/packages/utils` — pure utility functions (formatting, math, etc.).
   174→   174→- `/packages/types` — shared TypeScript types and Zod schemas.
   175→   175→
   176→   176→**`/src/shared/*`** — application-specific shared code that depends on `src/` structure but is used across contexts. Examples:
   177→   177→- `/src/shared/lib` — application utilities (auth client, supabase client).
   178→   178→- `/src/shared/config` — env vars, feature flags client.
   179→   179→- `/src/shared/hooks` — cross-context hooks (useUser, useLocale).
   180→   180→
   181→   181→**Consequences.**
   182→   182→- ✅ `/packages` code is highly reusable and could be open-sourced later.
   183→   183→- ✅ `/src/shared` keeps context-spanning app code organized.
   184→   184→- ✅ Clear ownership: `/packages` owned by platform team; `/src/shared` owned by whoever needs it.
   185→   185→- ⚠️ Discipline required to put the right code in the right place; documented in `07_FolderStructure`.
   186→   186→
   187→   187→**Implements:** LOCK-04 (modularity), DRY principle.
   188→   188→
   189→   189→### AD-06 — App Router Maps to Tool Registry
   190→   190→
   191→   191→**Context.** Next.js App Router uses file-based routing. With 1,000+ tools, having one route file per tool is unmanageable. But dynamic routes (`/tools/[category]/[slug]`) need to know which slugs are valid.
   192→   192→
   193→   193→**Decision.** Use Next.js dynamic routes that resolve against the generated registry:
   194→   194→
   195→   195→```
   196→   196→src/app/
   197→   197→├── (public)/
   198→   198→│   ├── tools/[category]/[slug]/
   199→   199→│   │   └── page.tsx         # Single dynamic route for all tools
   200→   200→│   ├── tools/[category]/
   201→   201→│   │   └── page.tsx         # Category landing page
   202→   202→│   └── tools/
   203→   203→│       └── page.tsx         # All-tools hub page
   204→   204→├── (auth)/
   205→   205→│   ├── login/page.tsx
   206→   206→│   ├── register/page.tsx
   207→   207→│   └── dashboard/page.tsx
   208→   208→├── (admin)/
   209→   209→│   └── admin/               # Per `24_AdminSpecification`
   210→   210→└── (marketing)/
   211→   211→    ├── page.tsx             # Homepage
   212→   212→    ├── about/page.tsx
   213→   213→    └── blog/page.tsx        # Content Context articles
   214→   214→```
   215→   215→
   216→   216→The `[slug]/page.tsx` route:
   217→   217→1. Reads `params.slug` and `params.category`.
   218→   218→2. Looks up the manifest in the generated registry.
   219→   219→3. If not found → 404.
   220→   220→4. If found → renders the Tool Engine UI, lazy-loads the tool's Processing stage.
   221→   221→5. Generates SEO metadata from the manifest (LOCK-08).
   222→   222→
   223→   223→**Consequences.**
   224→   224→- ✅ One route file handles all tools; no route explosion.
   225→   225→- ✅ Sitemap, SEO metadata, navigation all derived from registry.
   226→   226→- ✅ Static generation possible for known slugs (via `generateStaticParams`).
   227→   227→- ⚠️ Careful handling needed for dynamic params validation.
   228→   228→
   229→   229→**Implements:** LOCK-05 (registry-driven routing), LOCK-08 (SEO from manifest).
   230→   230→
   231→   231→### AD-07 — Database-Optional Layout
   232→   232→
   233→   233→**Context.** LOCK-06 mandates that core tools function without the database. The structure must make this obvious: the Tools Context directory must have no DB imports.
   234→   234→
   235→   235→**Decision.** The `src/tools/` directory is structurally forbidden from importing anything from `infrastructure/` layers of any context. This is enforced via ESLint `no-restricted-imports`:
   236→   236→
   237→   237→```javascript
   238→   238→// .eslintrc — tools context boundary
   239→   239→{
   240→   240→  files: ['src/tools/**/*'],
   241→   241→  rules: {
   242→   242→    'no-restricted-imports': ['error', {
   243→   243→      patterns: [
   244→   244→        '@/identity/infrastructure/*',
   245→   245→        '@/content/infrastructure/*',
   246→   246→        '@/platform-ops/infrastructure/*',
   247→   247→        '@/billing/infrastructure/*',
   248→   248→        '@/analytics/infrastructure/*',
   249→   249→        // Tools context has its OWN infrastructure only for server-side tools
   250→   250→        // but it cannot depend on OTHER contexts' infrastructure
   251→   251→      ]
   252→   252→    }]
   253→   253→  }
   254→   254→}
   255→   255→```
   256→   256→
   257→   257→Tools that need to persist data (e.g., history) do so via published interfaces (server actions in Identity Context), not direct DB calls.
   258→   258→
   259→   259→**Consequences.**
   260→   260→- ✅ Structural enforcement of LOCK-06; cannot be violated by accident.
   261→   261→- ✅ Tools are testable without DB setup.
   262→   262→- ✅ If DB is down, tools still work (server actions fail gracefully).
   263→   263→- ⚠️ Slightly more indirection for persistence; justified by LOCK-06.
   264→   264→
   265→   265→**Implements:** LOCK-06 (Database Optional), EC-05 (Progressive Enhancement — structural enforcement of graceful degradation).

   267→   267→## 4. Design Principles
   268→   268→
   269→   269→### P1 — One Tool, One Folder
   270→   270→Every tool is one folder. Adding a tool = adding a folder. Removing a tool = deleting a folder. No side effects on other tools.
   271→   271→
   272→   272→### P2 — Layers Are Visible
   273→   273→The four layers (Presentation, Application, Domain, Infrastructure) are visible as subdirectories within each context. This makes architectural decisions inspectable.
   274→   274→
   275→   275→### P3 — Generated Files Are Separate
   276→   276→Build-time generated files live in `/src/generated/` and are clearly marked. They are committed to the repo (for type safety across the team) but never manually edited.
   277→   277→
   278→   278→### P4 — Shared Code Has a Home
   279→   279→Cross-context code lives in `/packages` (if reusable) or `/src/shared` (if app-specific). Code is never duplicated across contexts (LOCK-09).
   280→   280→
   281→   281→### P5 — Boundaries Are Enforced
   282→   282→Folder structure alone is insufficient; ESLint boundary rules enforce dependency direction. Code review verifies boundary compliance.
   283→   283→
   284→   284→### P6 — Growth Is Linear
   285→   285→Adding tools, articles, admin features, or users scales linearly: one new folder per item, no reorganization of existing folders.

   389→   389→└── README.md                    # Project README (links to /docs)
   390→   390→```

   392→   392→## 6. Tool Module Anatomy (Detailed)
   393→   393→
   394→   394→This section expands AD-03. Every tool folder follows this exact structure.
   395→   395→
   396→   396→### 6.1 The Manifest File (`manifest.ts`)
   397→   397→
   398→   398→The manifest is the tool's identity and contract. It is the ONLY file the registry reads. Schema defined in `13_FBRD`.
   399→   399→
   400→   400→```typescript
   401→   401→// src/tools/image/image-resize/manifest.ts
   402→   402→import { z } from 'zod';
   403→   403→import { inputStage } from './stages/input';
   404→   404→import { validationStage } from './stages/validation';
   405→   405→import { processingStage } from './stages/processing';
   406→   406→import { previewStage } from './stages/preview';
   407→   407→import { downloadStage } from './stages/download';
   408→   408→import type { ToolManifest } from '@packages/tool-engine';
   409→   409→
   410→   410→export const manifest: ToolManifest = {
   411→   411→  slug: 'image-resize',
   412→   412→  category: 'image',
   413→   413→  title: 'Image Resizer',
   414→   414→  description: 'Resize images in your browser...',
   415→   415→  lifecycle: 'stable',                    // LOCK-12
   416→   416→  execution: 'browser',                   // LOCK-02
   417→   417→  inputSchema: z.object({...}),
   418→   418→  outputSchema: z.object({...}),
   419→   419→  stages: {
   420→   420→    input: inputStage,
   421→   421→    validation: validationStage,
   422→   422→    processing: processingStage,
   423→   423→    preview: previewStage,
   424→   424→    download: downloadStage,
   425→   425→    // history and share optional
   426→   426→  },
   427→   427→  seo: { /* per LOCK-08 */ },
   428→   428→  relatedTools: ['image-compress', 'image-crop'],
   429→   429→  faq: [ /* ... */ ],
   430→   430→};
   431→   431→
   432→   432→export default manifest;
   433→   433→```
   434→   434→
   435→   435→### 6.2 Stage Files (`stages/*.ts`)
   436→   436→
   437→   437→Each stage is a pure function following the contract in `02_SAD` §6.1. Stages have no React imports (except `preview.tsx` which renders UI).
   438→   438→
   439→   439→### 6.3 Components Folder (`components/`)
   440→   440→
   441→   441→Tool-specific React components (input forms, configuration panels). These import from `@packages/ui` for design system primitives, never from other tools.
   442→   442→
   443→   443→### 6.4 Hooks Folder (`hooks/`)
   444→   444→
   445→   445→Tool-specific hooks. Reusable hooks live in `@packages/tool-engine` or `@/shared/hooks`.
   446→   446→
   447→   447→### 6.5 Tests Folder (`tests/`)
   448→   448→
   449→   449→- `stages.test.ts` — unit test per stage; pure function tests.
   450→   450→- `e2e.test.ts` — Playwright test for the full tool workflow.
   451→   451→
   452→   452→### 6.6 README (optional)
   453→   453→
   454→   454→For complex tools, a tool-specific README documents edge cases, performance notes, and design decisions. Not required for simple tools.
   455→   455→
   456→   456→## 7. Tool Registry Pattern (Detailed)
   457→   457→
   458→   458→### 7.1 Codegen Script
   459→   459→
   460→   460→`scripts/generate-registry.ts`:
   461→   461→
   462→   462→1. Walks `src/tools/**/manifest.ts` (using glob).
   463→   463→2. Dynamic-imports each manifest.
   464→   464→3. Validates each manifest against the Zod schema from `@packages/tool-engine`.
   465→   465→4. Validates slug/category uniqueness.
   466→   466→5. Emits typed files to `src/generated/`.
   467→   467→
   468→   468→### 7.2 Generated Files
   469→   469→
   470→   470→**`src/generated/registry.ts`:**
   471→   471→
   472→   472→```typescript
   473→   473→// AUTO-GENERATED. Do not edit manually.
   474→   474→import type { ToolManifest } from '@packages/tool-engine';
   475→   475→import imageResize from '@/tools/image/image-resize/manifest';
   476→   476→import imageCompress from '@/tools/image/image-compress/manifest';
   477→   477→// ... all tool manifests
   478→   478→
   479→   479→export const registry = {
   480→   480→  bySlug: (slug: string): ToolManifest | undefined => { /* ... */ },
   481→   481→  byCategory: (category: string): ToolManifest[] => { /* ... */ },
   482→   482→  relatedTo: (slug: string): ToolManifest[] => { /* ... */ },
   483→   483→  all: (): ToolManifest[] => { /* ... */ },
   484→   484→};
   485→   485→
   486→   486→export const allManifests: ToolManifest[] = [/* ... */];
   487→   487→```
   488→   488→
   489→   489→**`src/generated/navigation.ts`:**
   490→   490→
   491→   491→```typescript
   492→   492→export const navigation = {
   493→   493→  image: [/* tool slugs */],
   494→   494→  pdf: [/* ... */],
   495→   495→  // ...
   496→   496→};
   497→   497→```
   498→   498→
   499→   499→**`src/generated/sitemap.ts`:**
   500→   500→
   501→   501→```typescript
   502→   502→export const sitemapEntries = [
   503→   503→  { url: '/tools/image/image-resize', lastModified: '2026-06-28', priority: 0.8 },
   504→   504→  // ...
   505→   505→];
   506→   506→```
   507→   507→
   508→   508→### 7.3 CI Verification
   509→   509→
   510→   510→`scripts/verify-registry.ts` runs in CI:
   511→   511→1. Regenerates the registry.
   512→   512→2. Diffs against committed files.
   513→   513→3. Fails if out of sync.
   514→   514→
   515→   515→This ensures the committed registry always matches the manifests.

   517→   517→## 8. Module Boundaries
   518→   518→
   519→   519→### 8.1 Boundary Enforcement
   520→   520→
   521→   521→Boundaries are enforced at three levels:
   522→   522→
   523→   523→1. **Folder structure** — visible in IDE; humans can see boundaries.
   524→   524→2. **ESLint `no-restricted-imports`** — automated; fails lint on violation.
   525→   525→3. **Code review** — final check; reviewers verify boundary compliance.
   526→   526→
   527→   527→### 8.2 Boundary Rules
   528→   528→
   529→   529→| Source → Target | Allowed? | Reason |
   530→   530→|-----------------|----------|--------|
   531→   531→| Tool → Tool (same tool) | ✅ | Same module |
   532→   532→| Tool → Tool (different tool) | ❌ | LOCK-04 isolation |
   533→   533→| Tool → Tool `_shared/` | ✅ | Tools-context-internal shared |
   534→   534→| Tool → `@packages/*` | ✅ | Reusable packages |
   535→   535→| Tool → `@/shared/*` | ✅ | App-specific shared |
   536→   536→| Tool → `@/identity/infrastructure/*` | ❌ | LOCK-06 |
   537→   537→| Tool → `@/content/infrastructure/*` | ❌ | LOCK-06 |
   538→   538→| Tool → Server Action (Identity) | ✅ | Published interface |
   539→   539→| `presentation/` → `infrastructure/` | ❌ | Layer violation |
   540→   540→| `domain/` → `react` | ❌ | Layer violation |
   541→   541→| Context A → Context B (internal) | ❌ | Use published API or events |
   542→   542→
   543→   543→### 8.3 Server Actions as Published Interfaces
   544→   544→
   545→   545→Server actions are the sanctioned way for tools (and other contexts) to invoke cross-context functionality:
   546→   546→
   547→   547→```typescript
   548→   548→// src/identity/application/actions/history.ts
   549→   549→'use server';
   550→   550→export async function saveHistoryEntry(entry: HistoryEntryInput) {
   551→   551→  // Identity Context owns this; tools call it
   552→   552→}
   553→   553→
   554→   554→// src/tools/image/image-resize/stages/history.ts
   555→   555→import { saveHistoryEntry } from '@/identity/application/actions/history';
   556→   556→// ✅ Allowed — server action is a published interface
   557→   557→```
   558→   558→
   559→   559→Server actions are typed, audited, and rate-limited at the platform level.

   582→   582→
   583→   583→Generated files ARE committed to the repo. Reasons:
   584→   584→- Type safety across the team; no need to run codegen before type-checking.
   585→   585→- Faster CI (no codegen step before lint).
   586→   586→- Diff visibility in PRs (new tool → registry changes visible).

   588→   588→## 10. Standards
   589→   589→
   590→   590→### 10.1 Directory Naming Standards
   591→   591→- All directories: `kebab-case`.
   592→   592→- Tool slugs: `kebab-case`, URL-safe, unique within category.
   593→   593→- Context directories: match bounded context name (`identity`, `content`, etc.).
   594→   594→
   595→   595→### 10.2 File Placement Standards
   596→   596→- A file belongs to exactly one layer within one context.
   597→   597→- Shared utilities go in `/packages` (if reusable) or `/src/shared` (if app-specific).
   598→   598→- Tests are colocated with the code they test (`tests/` subdirectory).
   599→   599→
   600→   600→### 10.3 Generated File Standards
   601→   601→- All generated files start with `// AUTO-GENERATED. Do not edit manually.`
   602→   602→- Generated files live in `/src/generated/` only.
   603→   603→- Manual edits to generated files are reverted by CI.
   604→   604→
   605→   605→### 10.4 Tool Folder Standards
   606→   606→- Every tool folder MUST contain `manifest.ts` and `index.ts`.
   607→   607→- Every tool folder MUST contain `stages/` with at least 5 stage files (Input, Validation, Processing, Preview, Download).
   608→   608→- Every tool folder MUST contain `tests/` with at least one stage test.

   610→   610→## 11. Best Practices
   611→   611→
   612→   612→### 11.1 When Adding a New Tool
   613→   613→1. Create folder: `src/tools/[category]/[slug]/`.
   614→   614→2. Copy the tool template (`scripts/tool-template/`) into the new folder.
   615→   615→3. Edit `manifest.ts` with tool metadata.
   616→   616→4. Implement each stage file.
   617→   617→5. Run `pnpm gen:registry`.
   618→   618→6. Verify the tool appears in navigation.
   619→   619→7. Run `pnpm test` and `pnpm lint`.
   620→   620→8. PR includes the new folder + updated generated files.
   621→   621→
   622→   622→### 11.2 When Refactoring a Tool
   623→   623→- Keep changes within the tool folder where possible.
   624→   624→- If changes affect `_shared/` or `@packages/*`, get Chief Architect approval (LOCK-09).
   625→   625→- Update tests in the same PR.
   626→   626→
   627→   627→### 11.3 When Adding a New Bounded Context
   628→   628→1. Get Chief Architect approval (requires DDD amendment per `03_DDD`).
   629→   629→2. Create `src/[context-name]/` with four layer subdirectories.
   630→   630→3. Create `drizzle/[context-name]/` for migrations.
   631→   631→4. Add context to ESLint boundary rules.
   632→   632→5. Update this document via PR.
   633→   633→
   634→   634→### 11.4 When Moving Code Between Layers
   635→   635→- Moving code DOWN (Presentation → Application) is usually safe.
   636→   636→- Moving code UP (Domain → Application) requires justification; Domain should be stable.
   637→   637→- Moving code between contexts requires DDD amendment.
   638→   638→
   639→   639→## 12. Future Scalability
   640→   640→
   641→   641→### 12.1 Scaling to 1,000+ Tools
   642→   642→- Folder structure scales linearly; no reorganization needed.
   643→   643→- Build time scales linearly; mitigations at >500 tools:
   644→   644→  - Parallel manifest import during codegen.
   645→   645→  - Cache codegen output; invalidate only on manifest change.
   646→   646→  - Consider moving to a single index file per category.
   647→   647→
   648→   648→### 12.2 Scaling to Multiple Teams
   649→   649→- Each team owns one or more bounded contexts.
   650→   650→- Cross-team changes via published interfaces (server actions, events).
   651→   651→- `/packages` owned by platform team; changes require their review.
   652→   652→
   653→   653→### 12.3 Preparing for Plugin Marketplace (Phase 4)
   654→   654→- Tool folder structure is the plugin contract.
   655→   655→- Phase 4 adds:
   656→   656→  - `/plugins/` directory for third-party tools (separate from `/src/tools/`).
   657→   657→  - Plugin manifest signing (verifies publisher).
   658→   658→  - Plugin sandboxing (limits APIs available to third-party code).
   659→   659→  - Plugin versioning (manifest declares supported platform version).
   660→   660→- No changes to first-party tool structure; plugins follow the same anatomy.
   661→   661→
   662→   662→### 12.4 Preparing for Multi-Tenancy (Phase 3+)
   663→   663→- Each context's domain layer gains `tenantId` on relevant aggregates.
   664→   664→- Infrastructure layer enforces tenant isolation in queries.
   665→   665→- No structural changes to top-level layout.
   666→   666→
   667→   667→### 12.5 Preparing for Mobile App (Phase 3+)
   668→   668→- `/packages` code (types, schemas, utilities) is portable to React Native.
   669→   669→- `/src/tools/` Processing stages are portable (pure TS).
   670→   670→- UI layers rebuilt natively; share types via `/packages/types`.
