    11тЖТ    11тЖТ## 1. Purpose
    12тЖТ    12тЖТ
    13тЖТ    13тЖТThis Product Constitution document expands the ten Product Constitution articles (PC-01 through PC-10) established in `00_Project_Charter` ┬з5 into operational detail. Where the charter establishes the articles as binding commitments, this document defines what each article means in practice: what it requires, what it forbids, how it's enforced, and how it interacts with the other governance layers.
    14тЖТ    14тЖТ
    15тЖТ    15тЖТThe Product Constitution is the third governance tier, sitting below the Architectural Locks (┬з3 of charter) and Engineering Constitution (┬з4 of charter) and above all technical documents. It binds every tool, present and future, with the same authority. The rationale for treating product decisions as architectural decisions is simple: at 1,000+ tools, inconsistent product behavior fragments the user experience and makes the platform feel like a patchwork rather than a cohesive ecosystem. A user who learns the layout on one tool should immediately understand every other tool.
    16тЖТ    16тЖТ
    17тЖТ    17тЖТ**Product decisions are architectural decisions.** A tool that violates PC-01 through PC-10 cannot be promoted to Stable status (PC-04) and may be blocked from Beta depending on severity. This document is the authoritative reference for what each article requires; ADR-054 through ADR-063 in `06_ArchitectureDecisionRecords` provide the formal architectural decision records.

    38тЖТ    38тЖТ## 3. Architectural Decisions
    39тЖТ    39тЖТ
    40тЖТ    40тЖТ### AD-01 тАФ Product Constitution as Governance Tier 3
    41тЖТ    41тЖТ
    42тЖТ    42тЖТ**Context.** Without a product-level governance layer, product decisions are made ad hoc per tool. Over time, tools diverge in how they handle errors, layout, analytics, monetization, and discoverability. The platform feels inconsistent.
    43тЖТ    43тЖТ
    44тЖТ    44тЖТ**Decision.** Establish the Product Constitution as the third governance tier, with binding authority over all product behavior. Priority order: Architectural Locks > Engineering Constitution > Product Constitution > Technical Documents. Product decisions require charter-level authority to amend.
    45тЖТ    45тЖТ
    46тЖТ    46тЖТ**Implements:** All PC articles.

    50тЖТ    50тЖТ**Context.** Each PC article is a binding commitment. Without a corresponding ADR, the article's rationale and alternatives are lost.
    51тЖТ    51тЖТ
    52тЖТ    52тЖТ**Decision.** Each PC article has a corresponding ADR (ADR-054 through ADR-063) that records context, decision, consequences, alternatives, and future review triggers.

    56тЖТ    56тЖТ## 4. Design Principles
    57тЖТ    57тЖТ
    58тЖТ    58тЖТ### P1 тАФ One Tool, One Problem
    59тЖТ    59тЖТEvery tool solves exactly one user problem. Multi-feature tools fragment intent and SEO.
    60тЖТ    60тЖТ
    61тЖТ    61тЖТ### P2 тАФ Contract-Driven
    62тЖТ    62тЖТEvery tool has a formal product contract encoded in its manifest. No contract, no tool.
    63тЖТ    63тЖТ
    64тЖТ    64тЖТ### P3 тАФ Consistency Enables Scale
    65тЖТ    65тЖТ1,000+ tools feel like one ecosystem because every tool follows the same layout, analytics, error patterns, and discoverability rules.
    66тЖТ    66тЖТ
    67тЖТ    67тЖТ### P4 тАФ Quality Is Earned, Not Assumed
    68тЖТ    68тЖТStable status requires passing 7 quality gates. Tools don't become Stable by aging; they become Stable by meeting objective criteria.
    69тЖТ    69тЖТ
    70тЖТ    70тЖТ### P5 тАФ User Value Before Revenue
    71тЖТ    71тЖТMonetization never interrupts task completion. Revenue follows value demonstration.
    72тЖТ    72тЖТ
    73тЖТ    73тЖТ### P6 тАФ Errors Are Recoverable
    74тЖТ    74тЖТEvery error explains what, why, how to fix. Users can self-recover.
    75тЖТ    75тЖТ
    76тЖТ    76тЖТ### P7 тАФ Discovery Is Mandatory
    77тЖТ    77тЖТEvery tool helps users discover more tools. The ecosystem grows engagement naturally.

    79тЖТ    79тЖТ## 5. Product Constitution Articles (Detailed)
    80тЖТ    80тЖТ
    81тЖТ    81тЖТ### 5.1 PC-01 тАФ Every Tool Solves One Problem
    82тЖТ    82тЖТ
    83тЖТ    83тЖТ**Binding text (from `00_Project_Charter` ┬з5):**
    84тЖТ    84тЖТ> Each tool MUST solve exactly one clearly defined user problem. Avoid feature creep. One page. One intent. One outcome.
    85тЖТ    85тЖТ
    86тЖТ    86тЖТ**Operational meaning:**
    87тЖТ    87тЖТ- A tool has ONE primary user intent, stated in its manifest's `purpose` field.
    88тЖТ    88тЖТ- A tool's `slug` and `title` clearly communicate that single intent (e.g., `image-resize` not `image-toolkit`).
    89тЖТ    89тЖТ- A tool's Processing stage produces ONE primary output type (with optional secondary outputs like metadata, but not multiple distinct transformations).
    90тЖТ    90тЖТ- If a tool naturally does multiple things (e.g., image resize + compress), it's split into multiple tools.
    91тЖТ    91тЖТ
    92тЖТ    92тЖТ**What it forbids:**
    93тЖТ    93тЖТ- "Swiss army knife" tools that combine unrelated functions.
    94тЖТ    94тЖТ- "Mode" selectors that change what a tool does.
    95тЖТ    95тЖТ- Tools whose name doesn't predict their behavior.
    96тЖТ    96тЖТ
    97тЖТ    97тЖТ**Enforcement:**
    98тЖТ    98тЖТ- Tool manifest validation: `purpose` field is a single sentence.
    99тЖТ    99тЖТ- Code review: reviewer verifies the tool does ONE thing.
   100тЖТ   100тЖТ- PC-04 quality gate: Functional review checks single-intent compliance.
   101тЖТ   101тЖТ
   102тЖТ   102тЖТ**Examples:**
   103тЖТ   103тЖТ- тЬЕ `image-resize` тАФ changes image dimensions. Single intent.
   104тЖТ   104тЖТ- тЬЕ `image-compress` тАФ reduces image file size. Single intent.
   105тЖТ   105тЖТ- тЭМ `image-toolkit` тАФ resizes, compresses, converts, watermarks. Multi-intent (split into 4 tools).
   106тЖТ   106тЖТ- тЬЕ `pdf-merge` тАФ combines multiple PDFs into one. Single intent.
   107тЖТ   107тЖТ- тЬЕ `pdf-split` тАФ splits one PDF into multiple. Single intent.
   108тЖТ   108тЖТ
   109тЖТ   109тЖТ**Interaction with other articles:**
   110тЖТ   110тЖТ- PC-09 (Discoverability): Multi-step workflows span multiple tools; each tool is single-intent but related tools are linked.
   111тЖТ   111тЖТ- LOCK-08 (SEO): One intent per page = higher search relevance.
   112тЖТ   112тЖТ
   113тЖТ   113тЖТ**Exceptions:** None. If a tool genuinely needs to do multiple things, propose a "workflow" concept via ADR (not a multi-feature tool).
   114тЖТ   114тЖТ

   117тЖТ   117тЖТ### 5.2 PC-02 тАФ Every Tool Has a Product Contract
   118тЖТ   118тЖТ
   119тЖТ   119тЖТ**Binding text:**
   120тЖТ   120тЖТ> Every tool MUST define: Purpose, User Problem, Inputs, Outputs, Validation Rules, Processing Steps, Success Criteria, Failure States, Empty States, Loading States, SEO Intent, Related Tools, Analytics Events. No tool exists without this contract.
   121тЖТ   121тЖТ
   122тЖТ   122тЖТ**Operational meaning:**
   123тЖТ   123тЖТEvery tool's manifest (per `12_ToolManifestSpecification`) MUST include:
   124тЖТ   124тЖТ
   125тЖТ   125тЖТ| Contract Field | Manifest Field | Description |
   126тЖТ   126тЖТ|---------------|----------------|-------------|
   127тЖТ   127тЖТ| Purpose | `purpose` | One-sentence statement of what the tool does. |
   128тЖТ   128тЖТ| User Problem | `userProblem` | The problem the user is trying to solve. |
   129тЖТ   129тЖТ| Inputs | `inputSchema` (Zod) | Schema of accepted inputs. |
   130тЖТ   130тЖТ| Outputs | `outputSchema` (Zod) | Schema of produced outputs. |
   131тЖТ   131тЖТ| Validation Rules | `validationRules` | Specific validation constraints (size limits, format restrictions). |
   132тЖТ   132тЖТ| Processing Steps | `stages.processing` | The Processing stage implementation. |
   133тЖТ   133тЖТ| Success Criteria | `successCriteria` | What constitutes a successful execution. |
   134тЖТ   134тЖТ| Failure States | `failureStates` | Enumerated failure modes with user-facing messages. |
   135тЖТ   135тЖТ| Empty States | `emptyStates` | UI when no input has been provided yet. |
   136тЖТ   136тЖТ| Loading States | `loadingStates` | UI during processing. |
   137тЖТ   137тЖТ| SEO Intent | `seo.searchIntent` | Informational / Transactional / Navigational. |
   138тЖТ   138тЖТ| Related Tools | `relatedTools` | Slugs of related tools for discoverability (PC-09). |
   139тЖТ   139тЖТ| Analytics Events | `analytics.events` | List of analytics events the tool emits (PC-07). |
   140тЖТ   140тЖТ
   141тЖТ   141тЖТ**What it forbids:**
   142тЖТ   142тЖТ- Tools with incomplete manifests.
   143тЖТ   143тЖТ- Tools where behavior is undocumented (if it's not in the contract, it doesn't exist).
   144тЖТ   144тЖТ- Tools with implicit inputs/outputs not in the schema.
   145тЖТ   145тЖТ
   146тЖТ   146тЖТ**Enforcement:**
   147тЖТ   147тЖТ- Build-time manifest validation against Zod schema.
   148тЖТ   148тЖТ- CI fails if manifest is incomplete.
   149тЖТ   149тЖТ- PC-04 quality gate: Documentation review verifies contract completeness.
   150тЖТ   150тЖТ
   151тЖТ   151тЖТ**Interaction with other articles:**
   152тЖТ   152тЖТ- PC-03 (Completion Standard): Contract is the foundation; completion standard verifies implementation matches contract.
   153тЖТ   153тЖТ- PC-07 (Analytics): Analytics events are part of the contract.
   154тЖТ   154тЖТ- PC-10 (Product Scalability): Manifest is the metadata source for codegen.

   158тЖТ   158тЖТ### 5.3 PC-03 тАФ Tool Completion Standard
   159тЖТ   159тЖТ
   160тЖТ   160тЖТ**Binding text:**
   161тЖТ   161тЖТ> A tool is only considered complete if it provides: Upload/Input, Validation, Processing, Preview, Download/Copy, Error Handling, Success Feedback, Accessibility, Mobile Support, SEO, Analytics, Documentation, Tests. All 13 items mandatory.
   162тЖТ   162тЖТ
   163тЖТ   163тЖТ**Operational meaning:**
   164тЖТ   164тЖТThe 13 mandatory items:
   165тЖТ   165тЖТ
   166тЖТ   166тЖТ| # | Item | Requirement |
   167тЖТ   167тЖТ|---|------|-------------|
   168тЖТ   168тЖТ| 1 | Upload/Input | Tool accepts user input via form, file upload, or URL. |
   169тЖТ   169тЖТ| 2 | Validation | Input validated with Zod; invalid input returns typed error. |
   170тЖТ   170тЖТ| 3 | Processing | Tool Engine Processing stage implemented (browser or server per manifest). |
   171тЖТ   171тЖТ| 4 | Preview | Result previewed before download; user can inspect. |
   172тЖТ   172тЖТ| 5 | Download/Copy | Result downloadable (file) or copyable (text); per LOCK-07, no registration required for download. |
   173тЖТ   173тЖТ| 6 | Error Handling | All error paths produce typed errors with user-facing messages (PC-08). |
   174тЖТ   174тЖТ| 7 | Success Feedback | Successful execution shows clear success state (toast, highlight, etc.). |
   175тЖТ   175тЖТ| 8 | Accessibility | WCAG 2.1 AA conformance; keyboard nav, screen reader, focus visibility, reduced motion (EC-06). |
   176тЖТ   176тЖТ| 9 | Mobile Support | Tool usable on 360px viewport; touch targets тЙе44x44px. |
   177тЖТ   177тЖТ| 10 | SEO | Per `18_SEOSpecification`: metadata, structured data, FAQ, related tools, breadcrumb. |
   178тЖТ   178тЖТ| 11 | Analytics | Emits all PC-07 required events. |
   179тЖТ   179тЖТ| 12 | Documentation | Tool README documents purpose, inputs, outputs, edge cases. |
   180тЖТ   180тЖТ| 13 | Tests | Unit tests for stages; E2E test for workflow; accessibility test. |
   181тЖТ   181тЖТ
   182тЖТ   182тЖТ**What it forbids:**
   183тЖТ   183тЖТ- Tools missing any of the 13 items.
   184тЖТ   184тЖТ- Tools with "TODO" or "coming soon" for any item.
   185тЖТ   185тЖТ- Tools that work on desktop but not mobile.
   186тЖТ   186тЖТ
   187тЖТ   187тЖТ**Enforcement:**
   188тЖТ   188тЖТ- PC-04 quality gates verify each item.
   189тЖТ   189тЖТ- CI checks for tests, SEO metadata, analytics events.
   190тЖТ   190тЖТ- Manual review for accessibility, mobile, UX.
   191тЖТ   191тЖТ
   192тЖТ   192тЖТ**Interaction with other articles:**
   193тЖТ   193тЖТ- PC-04 (Quality Gates): Completion standard is the baseline; quality gates verify it.
   194тЖТ   194тЖТ- EC-04 (Tool Template): Template scaffolds all 13 items.

   198тЖТ   198тЖТ### 5.4 PC-04 тАФ Product Quality Gates
   199тЖТ   199тЖТ
   200тЖТ   200тЖТ**Binding text:**
   201тЖТ   201тЖТ> A feature cannot reach Stable unless it passes: Functional review, Accessibility review, Performance review, SEO review, Security review, Documentation review, UX review. Quality gates are mandatory.
   202тЖТ   202тЖТ
   203тЖТ   203тЖТ**Operational meaning:**
   204тЖТ   204тЖТSeven quality gates, each with objective criteria:
   205тЖТ   205тЖТ
   206тЖТ   206тЖТ| # | Gate | Criteria | Reviewer |
   207тЖТ   207тЖТ|---|------|----------|----------|
   208тЖТ   208тЖТ| 1 | Functional | All manifest-declared features work as specified; tests pass. | Engineer (peer) |
   209тЖТ   209тЖТ| 2 | Accessibility | WCAG 2.1 AA; Lighthouse accessibility тЙе95; keyboard nav tested; screen reader tested. | Engineer (trained) |
   210тЖТ   210тЖТ| 3 | Performance | Lighthouse performance тЙе90; bundle size within budget; TTFB <500ms. | Engineer |
   211тЖТ   211тЖТ| 4 | SEO | All `18_SEOSpecification` requirements met; structured data valid; unique metadata. | SEO reviewer |
   212тЖТ   212тЖТ| 5 | Security | Input validation; no secrets in client; RLS policies; no XSS vectors. | Security reviewer |
   213тЖТ   213тЖТ| 6 | Documentation | Tool README complete; ADRs updated if architectural change; manifest accurate. | Tech writer or engineer |
   214тЖТ   214тЖТ| 7 | UX | Layout follows PC-05; error states follow PC-08; success feedback present; mobile usable. | UX reviewer or designer |
   215тЖТ   215тЖТ
   216тЖТ   216тЖТ**Promotion workflow:**
   217тЖТ   217тЖТ1. Tool is in `Testing` or `Beta` lifecycle state.
   218тЖТ   218тЖТ2. Author requests promotion to `Stable` via PR.
   219тЖТ   219тЖТ3. Each gate reviewer signs off (or rejects with feedback).
   220тЖТ   220тЖТ4. All 7 gates pass тЖТ promote to `Stable` (update manifest `lifecycle` field).
   221тЖТ   221тЖТ5. Any gate fails тЖТ address feedback; re-request.
   222тЖТ   222тЖТ
   223тЖТ   223тЖТ**What it forbids:**
   224тЖТ   224тЖТ- Self-promotion to Stable without all 7 sign-offs.
   225тЖТ   225тЖТ- Skipping gates "because it's a small change."
   226тЖТ   226тЖТ
   227тЖТ   227тЖТ**Enforcement:**
   228тЖТ   228тЖТ- Admin panel tracks lifecycle transitions; requires sign-off records.
   229тЖТ   229тЖТ- Audit trail (LOCK-11) records who signed off and when.
   230тЖТ   230тЖТ- Code review checklist includes gate verification.
   231тЖТ   231тЖТ
   232тЖТ   232тЖТ**Interaction with other articles:**
   233тЖТ   233тЖТ- PC-03 (Completion Standard): Completion is the baseline; gates verify completion plus additional quality.
   234тЖТ   234тЖТ- LOCK-12 (Feature Lifecycle): Gates are the criteria for promotion to Stable.

   283тЖТ   283тЖТ
   284тЖТ   284тЖТ**Binding text:**

   293тЖТ   293тЖТ
   294тЖТ   294тЖТ**What it forbids:**
   295тЖТ   295тЖТ- Ads before the result.
   296тЖТ   296тЖТ- Paywalls on core functionality.
   297тЖТ   297тЖТ- Forced registration to complete a task.
   298тЖТ   298тЖТ- Premium features that are necessary for basic use (e.g., "premium required to download full resolution").
   299тЖТ   299тЖТ
   300тЖТ   300тЖТ**Enforcement:**
   301тЖТ   301тЖТ- Code review: reviewer verifies no monetization in core workflow.
   302тЖТ   302тЖТ- PC-04 UX gate: checks monetization placement.
   303тЖТ   303тЖТ

   305тЖТ   305тЖТ- LOCK-07 (Guest-First): Reinforces no registration before value.

   310тЖТ   310тЖТ### 5.7 PC-07 тАФ Analytics Standard
   311тЖТ   311тЖТ
   312тЖТ   312тЖТ**Binding text:**
   313тЖТ   313тЖТ> Every tool MUST emit consistent events. Minimum events: Tool Viewed, Tool Started, Validation Failed, Processing Started, Processing Completed, Download Attempted, Download Completed, Registration Prompt Viewed, Registration Completed, Tool Shared.
   314тЖТ   314тЖТ
   315тЖТ   315тЖТ**Operational meaning:**
   316тЖТ   316тЖТMandatory analytics events (all tools emit these):
   317тЖТ   317тЖТ
   318тЖТ   318тЖТ| Event Name | Trigger | Payload |
   319тЖТ   319тЖТ|-----------|---------|---------|
   320тЖТ   320тЖТ| `tool_viewed` | User lands on tool page | `{ toolSlug, referrer, locale }` |
   321тЖТ   321тЖТ| `tool_started` | User begins input (uploads file, types text) | `{ toolSlug, inputType }` |
   322тЖТ   322тЖТ| `validation_failed` | Input validation fails | `{ toolSlug, field, errorKind }` |
   323тЖТ   323тЖТ| `processing_started` | Processing stage begins | `{ toolSlug, inputSummary }` |
   324тЖТ   324тЖТ| `processing_completed` | Processing stage completes successfully | `{ toolSlug, durationMs, outputSummary }` |
   325тЖТ   325тЖТ| `download_attempted` | User clicks download/copy button | `{ toolSlug, outputFormat }` |
   326тЖТ   326тЖТ| `download_completed` | Download/copy succeeds | `{ toolSlug, outputSize }` |
   327тЖТ   327тЖТ| `registration_prompt_viewed` | Registration prompt shown | `{ toolSlug, trigger }` |
   328тЖТ   328тЖТ| `registration_completed` | User completes registration | `{ toolSlug, method }` |
   329тЖТ   329тЖТ| `tool_shared` | User shares tool (link copy, social) | `{ toolSlug, shareMethod }` |
   330тЖТ   330тЖТ
   331тЖТ   331тЖТ**What it forbids:**
   332тЖТ   332тЖТ- Tools emitting custom event names instead of standard ones.
   333тЖТ   333тЖТ- Tools omitting any of the 10 mandatory events.
   334тЖТ   334тЖТ- Tools sending PII in event payloads (only userId if authenticated).
   335тЖТ   335тЖТ
   336тЖТ   336тЖТ**Enforcement:**
   337тЖТ   337тЖТ- Tool Engine emits events automatically; tools don't write custom analytics code.
   338тЖТ   338тЖТ- Manifest declares which events the tool emits (for tools with optional events like `tool_shared`).
   339тЖТ   339тЖТ- CI verifies manifest's analytics configuration.
   340тЖТ   340тЖТ
   341тЖТ   341тЖТ**Interaction with other articles:**
   342тЖТ   342тЖТ- PC-02 (Product Contract): Analytics events are part of the contract.
   343тЖТ   343тЖТ- PC-10 (Product Scalability): Analytics config derived from manifest.

   347тЖТ   347тЖТ### 5.8 PC-08 тАФ Error Experience
   348тЖТ   348тЖТ
   349тЖТ   349тЖТ**Binding text:**
   350тЖТ   350тЖТ> Every error MUST: explain what happened, explain why, explain how to fix it. Never expose technical stack traces.
   351тЖТ   351тЖТ
   352тЖТ   352тЖТ**Operational meaning:**
   353тЖТ   353тЖТEvery error has three components:
   354тЖТ   354тЖТ
   355тЖТ   355тЖТ1. **What happened** (plain language): "The image you uploaded is too large."
   356тЖТ   356тЖТ2. **Why** (if known): "The maximum file size is 10MB; your file is 25MB."
   357тЖТ   357тЖТ3. **How to fix it** (actionable): "Please upload a smaller image, or use the Image Compressor tool first."
   358тЖТ   358тЖТ
   359тЖТ   359тЖТ**Error types** (from `02_SAD` ┬з6.3):
   360тЖТ   360тЖТ- `validation` тАФ input validation failed.
   361тЖТ   361тЖТ- `processing` тАФ tool logic failed (recoverable or not).
   362тЖТ   362тЖТ- `quota_exceeded` тАФ user hit usage limit.
   363тЖТ   363тЖТ- `auth_required` тАФ feature requires registration.
   364тЖТ   364тЖТ- `server_unavailable` тАФ backend service down.
   365тЖТ   365тЖТ
   366тЖТ   366тЖТEach error type has a user-facing message template; tools fill in specifics.
   367тЖТ   367тЖТ
   368тЖТ   368тЖТ**What it forbids:**
   369тЖТ   369тЖТ- Generic "Something went wrong" messages.
   370тЖТ   370тЖТ- Raw stack traces or technical error codes visible to users.
   371тЖТ   371тЖТ- Errors without recovery guidance.
   372тЖТ   372тЖТ- Errors that blame the user ("You entered invalid input").
   373тЖТ   373тЖТ
   374тЖТ   374тЖТ**Enforcement:**
   375тЖТ   375тЖТ- Tool Engine maps errors to user-facing messages.
   376тЖТ   376тЖТ- Code review: reviewer verifies error messages meet the three-component standard.
   377тЖТ   377тЖТ- PC-04 UX gate: checks error experience.
   378тЖТ   378тЖТ
   379тЖТ   379тЖТ**Interaction with other articles:**
   380тЖТ   380тЖТ- EC-05 (Progressive Enhancement): Errors are graceful degradation points.
   381тЖТ   381тЖТ

   505тЖТ   505тЖТ**PC-02 (Contract):**
   506тЖТ   506тЖТ- Purpose: "Resize an image to specific width and height."
   507тЖТ   507тЖТ- User Problem: "I need an image at a specific size for a profile picture / thumbnail / print."

   509тЖТ   509тЖТ- Outputs: Resized image file (same format as input).
   510тЖТ   510тЖТ- Validation Rules: File тЙд10MB; dimensions 1-10000px; supported formats only.
   511тЖТ   511тЖТ- Processing Steps: Decode image тЖТ resize via Canvas тЖТ encode тЖТ return blob.
   512тЖТ   512тЖТ- Success Criteria: Output image has exact specified dimensions.
   513тЖТ   513тЖТ- Failure States: `validation` (invalid file), `processing` (decode failure), `quota_exceeded` (file too large).
   514тЖТ   514тЖТ- Empty States: "Upload an image to resize" with upload icon.
   515тЖТ   515тЖТ- Loading States: Progress bar during resize.
   516тЖТ   516тЖТ- SEO Intent: Transactional (user wants to DO something).
   517тЖТ   517тЖТ- Related Tools: `image-compress`, `image-crop`, `image-format-convert`.
   518тЖТ   518тЖТ- Analytics Events: All 10 standard events.
   519тЖТ   519тЖТ
   520тЖТ   520тЖТ**PC-03 (Completion):** All 13 items implemented.
   521тЖТ   521тЖТ
   522тЖТ   522тЖТ**PC-04 (Gates):** Passed all 7 gates; promoted to Stable.
   523тЖТ   523тЖТ
   524тЖТ   524тЖТ**PC-05 (Layout):** Uses `ToolLayout` with all 8 sections.
   525тЖТ   525тЖТ
   526тЖТ   526тЖТ**PC-06 (Monetization):** No ads before result. Ad appears after download. Premium upsell for batch resize (not blocking single resize).
   527тЖТ   527тЖТ
   528тЖТ   528тЖТ**PC-07 (Analytics):** Emits all 10 events via Tool Engine.
   529тЖТ   529тЖТ
   530тЖТ   530тЖТ**PC-08 (Errors):**
