    11‚Üí## 1. Purpose
    12‚Üí
    13‚ÜíThis Product Constitution document expands the ten Product Constitution articles (PC-01 through PC-10) established in `00_Project_Charter` ¬ß5 into operational detail. Where the charter establishes the articles as binding commitments, this document defines what each article means in practice: what it requires, what it forbids, how it's enforced, and how it interacts with the other governance layers.
    14‚Üí
    15‚ÜíThe Product Constitution is the third governance tier, sitting below the Architectural Locks (¬ß3 of charter) and Engineering Constitution (¬ß4 of charter) and above all technical documents. It binds every tool, present and future, with the same authority. The rationale for treating product decisions as architectural decisions is simple: at 1,000+ tools, inconsistent product behavior fragments the user experience and makes the platform feel like a patchwork rather than a cohesive ecosystem. A user who learns the layout on one tool should immediately understand every other tool.
    16‚Üí
    17‚Üí**Product decisions are architectural decisions.** A tool that violates PC-01 through PC-10 cannot be promoted to Stable status (PC-04) and may be blocked from Beta depending on severity. This document is the authoritative reference for what each article requires; ADR-054 through ADR-063 in `06_ArchitectureDecisionRecords` provide the formal architectural decision records.

    38‚Üí## 3. Architectural Decisions
    39‚Üí
    40‚Üí### AD-01 ‚Äî Product Constitution as Governance Tier 3
    41‚Üí
    42‚Üí**Context.** Without a product-level governance layer, product decisions are made ad hoc per tool. Over time, tools diverge in how they handle errors, layout, analytics, monetization, and discoverability. The platform feels inconsistent.
    43‚Üí
    44‚Üí**Decision.** Establish the Product Constitution as the third governance tier, with binding authority over all product behavior. Priority order: Architectural Locks > Engineering Constitution > Product Constitution > Technical Documents. Product decisions require charter-level authority to amend.
    45‚Üí
    46‚Üí**Implements:** All PC articles.
    47‚Üí
    48‚Üí### AD-02 ‚Äî Every Article Maps to Specific ADR
    49‚Üí
    50‚Üí**Context.** Each PC article is a binding commitment. Without a corresponding ADR, the article's rationale and alternatives are lost.
    51‚Üí
    52‚Üí**Decision.** Each PC article has a corresponding ADR (ADR-054 through ADR-063) that records context, decision, consequences, alternatives, and future review triggers.
    53‚Üí
    54‚Üí**Implements:** EC-01 (Documentation First).

    56‚Üí## 4. Design Principles
    57‚Üí
    58‚Üí### P1 ‚Äî One Tool, One Problem
    59‚ÜíEvery tool solves exactly one user problem. Multi-feature tools fragment intent and SEO.
    60‚Üí
    61‚Üí### P2 ‚Äî Contract-Driven
    62‚ÜíEvery tool has a formal product contract encoded in its manifest. No contract, no tool.
    63‚Üí
    64‚Üí### P3 ‚Äî Consistency Enables Scale
    65‚Üí1,000+ tools feel like one ecosystem because every tool follows the same layout, analytics, error patterns, and discoverability rules.
    66‚Üí
    67‚Üí### P4 ‚Äî Quality Is Earned, Not Assumed
    68‚ÜíStable status requires passing 7 quality gates. Tools don't become Stable by aging; they become Stable by meeting objective criteria.
    69‚Üí
    70‚Üí### P5 ‚Äî User Value Before Revenue
    71‚ÜíMonetization never interrupts task completion. Revenue follows value demonstration.
    72‚Üí
    73‚Üí### P6 ‚Äî Errors Are Recoverable
    74‚ÜíEvery error explains what, why, how to fix. Users can self-recover.
    75‚Üí
    76‚Üí### P7 ‚Äî Discovery Is Mandatory
    77‚ÜíEvery tool helps users discover more tools. The ecosystem grows engagement naturally.

    79‚Üí## 5. Product Constitution Articles (Detailed)
    80‚Üí
    81‚Üí### 5.1 PC-01 ‚Äî Every Tool Solves One Problem
    82‚Üí
    83‚Üí**Binding text (from `00_Project_Charter` ¬ß5):**
    84‚Üí> Each tool MUST solve exactly one clearly defined user problem. Avoid feature creep. One page. One intent. One outcome.
    85‚Üí
    86‚Üí**Operational meaning:**
    87‚Üí- A tool has ONE primary user intent, stated in its manifest's `purpose` field.
    88‚Üí- A tool's `slug` and `title` clearly communicate that single intent (e.g., `image-resize` not `image-toolkit`).
    89‚Üí- A tool's Processing stage produces ONE primary output type (with optional secondary outputs like metadata, but not multiple distinct transformations).
    90‚Üí- If a tool naturally does multiple things (e.g., image resize + compress), it's split into multiple tools.
    91‚Üí
    92‚Üí**What it forbids:**
    93‚Üí- "Swiss army knife" tools that combine unrelated functions.
    94‚Üí- "Mode" selectors that change what a tool does.
    95‚Üí- Tools whose name doesn't predict their behavior.
    96‚Üí
    97‚Üí**Enforcement:**
    98‚Üí- Tool manifest validation: `purpose` field is a single sentence.
    99‚Üí- Code review: reviewer verifies the tool does ONE thing.
   100‚Üí- PC-04 quality gate: Functional review checks single-intent compliance.
   101‚Üí
   102‚Üí**Examples:**
   103‚Üí- ‚úÖ `image-resize` ‚Äî changes image dimensions. Single intent.
   104‚Üí- ‚úÖ `image-compress` ‚Äî reduces image file size. Single intent.
   105‚Üí- ‚ùå `image-toolkit` ‚Äî resizes, compresses, converts, watermarks. Multi-intent (split into 4 tools).
   106‚Üí- ‚úÖ `pdf-merge` ‚Äî combines multiple PDFs into one. Single intent.
   107‚Üí- ‚úÖ `pdf-split` ‚Äî splits one PDF into multiple. Single intent.
   108‚Üí
   109‚Üí**Interaction with other articles:**
   110‚Üí- PC-09 (Discoverability): Multi-step workflows span multiple tools; each tool is single-intent but related tools are linked.
   111‚Üí- LOCK-08 (SEO): One intent per page = higher search relevance.
   112‚Üí
   113‚Üí**Exceptions:** None. If a tool genuinely needs to do multiple things, propose a "workflow" concept via ADR (not a multi-feature tool).

   117‚Üí### 5.2 PC-02 ‚Äî Every Tool Has a Product Contract
   118‚Üí
   119‚Üí**Binding text:**
   120‚Üí> Every tool MUST define: Purpose, User Problem, Inputs, Outputs, Validation Rules, Processing Steps, Success Criteria, Failure States, Empty States, Loading States, SEO Intent, Related Tools, Analytics Events. No tool exists without this contract.
   121‚Üí
   122‚Üí**Operational meaning:**
   123‚ÜíEvery tool's manifest (per `12_ToolManifestSpecification`) MUST include:
   124‚Üí
   125‚Üí| Contract Field | Manifest Field | Description |
   126‚Üí|---------------|----------------|-------------|
   127‚Üí| Purpose | `purpose` | One-sentence statement of what the tool does. |
   128‚Üí| User Problem | `userProblem` | The problem the user is trying to solve. |
   129‚Üí| Inputs | `inputSchema` (Zod) | Schema of accepted inputs. |
   130‚Üí| Outputs | `outputSchema` (Zod) | Schema of produced outputs. |
   131‚Üí| Validation Rules | `validationRules` | Specific validation constraints (size limits, format restrictions). |
   132‚Üí| Processing Steps | `stages.processing` | The Processing stage implementation. |
   133‚Üí| Success Criteria | `successCriteria` | What constitutes a successful execution. |
   134‚Üí| Failure States | `failureStates` | Enumerated failure modes with user-facing messages. |
   135‚Üí| Empty States | `emptyStates` | UI when no input has been provided yet. |
   136‚Üí| Loading States | `loadingStates` | UI during processing. |
   137‚Üí| SEO Intent | `seo.searchIntent` | Informational / Transactional / Navigational. |
   138‚Üí| Related Tools | `relatedTools` | Slugs of related tools for discoverability (PC-09). |
   139‚Üí| Analytics Events | `analytics.events` | List of analytics events the tool emits (PC-07). |
   140‚Üí
   141‚Üí**What it forbids:**
   142‚Üí- Tools with incomplete manifests.
   143‚Üí- Tools where behavior is undocumented (if it's not in the contract, it doesn't exist).
   144‚Üí- Tools with implicit inputs/outputs not in the schema.
   145‚Üí
   146‚Üí**Enforcement:**
   147‚Üí- Build-time manifest validation against Zod schema.
   148‚Üí- CI fails if manifest is incomplete.
   149‚Üí- PC-04 quality gate: Documentation review verifies contract completeness.
   150‚Üí
   151‚Üí**Interaction with other articles:**
   152‚Üí- PC-03 (Completion Standard): Contract is the foundation; completion standard verifies implementation matches contract.
   153‚Üí- PC-07 (Analytics): Analytics events are part of the contract.
   154‚Üí- PC-10 (Product Scalability): Manifest is the metadata source for codegen.

   158‚Üí### 5.3 PC-03 ‚Äî Tool Completion Standard
   159‚Üí
   160‚Üí**Binding text:**
   161‚Üí> A tool is only considered complete if it provides: Upload/Input, Validation, Processing, Preview, Download/Copy, Error Handling, Success Feedback, Accessibility, Mobile Support, SEO, Analytics, Documentation, Tests. All 13 items mandatory.
   162‚Üí
   163‚Üí**Operational meaning:**
   164‚ÜíThe 13 mandatory items:
   165‚Üí
   166‚Üí| # | Item | Requirement |
   167‚Üí|---|------|-------------|
   168‚Üí| 1 | Upload/Input | Tool accepts user input via form, file upload, or URL. |
   169‚Üí| 2 | Validation | Input validated with Zod; invalid input returns typed error. |
   170‚Üí| 3 | Processing | Tool Engine Processing stage implemented (browser or server per manifest). |
   171‚Üí| 4 | Preview | Result previewed before download; user can inspect. |
   172‚Üí| 5 | Download/Copy | Result downloadable (file) or copyable (text); per LOCK-07, no registration required for download. |
   173‚Üí| 6 | Error Handling | All error paths produce typed errors with user-facing messages (PC-08). |
   174‚Üí| 7 | Success Feedback | Successful execution shows clear success state (toast, highlight, etc.). |
   175‚Üí| 8 | Accessibility | WCAG 2.1 AA conformance; keyboard nav, screen reader, focus visibility, reduced motion (EC-06). |
   176‚Üí| 9 | Mobile Support | Tool usable on 360px viewport; touch targets ‚â•44x44px. |
   177‚Üí| 10 | SEO | Per `18_SEOSpecification`: metadata, structured data, FAQ, related tools, breadcrumb. |
   178‚Üí| 11 | Analytics | Emits all PC-07 required events. |
   179‚Üí| 12 | Documentation | Tool README documents purpose, inputs, outputs, edge cases. |
   180‚Üí| 13 | Tests | Unit tests for stages; E2E test for workflow; accessibility test. |
   181‚Üí
   182‚Üí**What it forbids:**
   183‚Üí- Tools missing any of the 13 items.
   184‚Üí- Tools with "TODO" or "coming soon" for any item.
   185‚Üí- Tools that work on desktop but not mobile.
   186‚Üí
   187‚Üí**Enforcement:**
   188‚Üí- PC-04 quality gates verify each item.
   189‚Üí- CI checks for tests, SEO metadata, analytics events.
   190‚Üí- Manual review for accessibility, mobile, UX.
   191‚Üí
   192‚Üí**Interaction with other articles:**
   193‚Üí- PC-04 (Quality Gates): Completion standard is the baseline; quality gates verify it.
   194‚Üí- EC-04 (Tool Template): Template scaffolds all 13 items.

   198‚Üí### 5.4 PC-04 ‚Äî Product Quality Gates
   199‚Üí
   200‚Üí**Binding text:**
   201‚Üí> A feature cannot reach Stable unless it passes: Functional review, Accessibility review, Performance review, SEO review, Security review, Documentation review, UX review. Quality gates are mandatory.
   202‚Üí
   203‚Üí**Operational meaning:**
   204‚ÜíSeven quality gates, each with objective criteria:
   205‚Üí
   206‚Üí| # | Gate | Criteria | Reviewer |
   207‚Üí|---|------|----------|----------|
   208‚Üí| 1 | Functional | All manifest-declared features work as specified; tests pass. | Engineer (peer) |
   209‚Üí| 2 | Accessibility | WCAG 2.1 AA; Lighthouse accessibility ‚â•95; keyboard nav tested; screen reader tested. | Engineer (trained) |
   210‚Üí| 3 | Performance | Lighthouse performance ‚â•90; bundle size within budget; TTFB <500ms. | Engineer |
   211‚Üí| 4 | SEO | All `18_SEOSpecification` requirements met; structured data valid; unique metadata. | SEO reviewer |
   212‚Üí| 5 | Security | Input validation; no secrets in client; RLS policies; no XSS vectors. | Security reviewer |
   213‚Üí| 6 | Documentation | Tool README complete; ADRs updated if architectural change; manifest accurate. | Tech writer or engineer |
   214‚Üí| 7 | UX | Layout follows PC-05; error states follow PC-08; success feedback present; mobile usable. | UX reviewer or designer |
   215‚Üí
   216‚Üí**Promotion workflow:**
   217‚Üí1. Tool is in `Testing` or `Beta` lifecycle state.
   218‚Üí2. Author requests promotion to `Stable` via PR.
   219‚Üí3. Each gate reviewer signs off (or rejects with feedback).
   220‚Üí4. All 7 gates pass ‚Üí promote to `Stable` (update manifest `lifecycle` field).
   221‚Üí5. Any gate fails ‚Üí address feedback; re-request.
   222‚Üí
   223‚Üí**What it forbids:**
   224‚Üí- Self-promotion to Stable without all 7 sign-offs.
   225‚Üí- Skipping gates "because it's a small change."
   226‚Üí
   227‚Üí**Enforcement:**
   228‚Üí- Admin panel tracks lifecycle transitions; requires sign-off records.
   229‚Üí- Audit trail (LOCK-11) records who signed off and when.
   230‚Üí- Code review checklist includes gate verification.
   231‚Üí
   232‚Üí**Interaction with other articles:**
   233‚Üí- PC-03 (Completion Standard): Completion is the baseline; gates verify completion plus additional quality.
   234‚Üí- LOCK-12 (Feature Lifecycle): Gates are the criteria for promotion to Stable.
   235‚Üí

   282‚Üí### 5.6 PC-06 ‚Äî Monetization Philosophy
   283‚Üí
   284‚Üí**Binding text:**
   285‚Üí> Revenue MUST never interrupt task completion. Advertising or premium prompts may appear only after value has been demonstrated. Core functionality remains free. Premium provides convenience, not necessity.
   286‚Üí
   287‚Üí**Operational meaning:**
   288‚Üí- Core tool workflow (Input ‚Üí Validation ‚Üí Processing ‚Üí Preview ‚Üí Download) is always free.
   289‚Üí- Ads appear only AFTER the result is shown (in the Result section or below).
   290‚Üí- Premium prompts appear only for value-adds: batch processing, cloud sync, AI features, higher file size limits.
   291‚Üí- No paywalls mid-workflow.
   292‚Üí- No "registration required to download" (per LOCK-07).
   293‚Üí
   294‚Üí**What it forbids:**
   295‚Üí- Ads before the result.
   296‚Üí- Paywalls on core functionality.
   297‚Üí- Forced registration to complete a task.
   298‚Üí- Premium features that are necessary for basic use (e.g., "premium required to download full resolution").
   299‚Üí
   300‚Üí**Enforcement:**
   301‚Üí- Code review: reviewer verifies no monetization in core workflow.
   302‚Üí- PC-04 UX gate: checks monetization placement.
   303‚Üí
   304‚Üí**Interaction with other articles:**
   305‚Üí- LOCK-07 (Guest-First): Reinforces no registration before value.
   306‚Üí- `01_BRD` ¬ß4.1: Monetization standards operationalize this.

   310‚Üí### 5.7 PC-07 ‚Äî Analytics Standard
   311‚Üí
   312‚Üí**Binding text:**
   313‚Üí> Every tool MUST emit consistent events. Minimum events: Tool Viewed, Tool Started, Validation Failed, Processing Started, Processing Completed, Download Attempted, Download Completed, Registration Prompt Viewed, Registration Completed, Tool Shared.
   314‚Üí
   315‚Üí**Operational meaning:**
   316‚ÜíMandatory analytics events (all tools emit these):
   317‚Üí
   318‚Üí| Event Name | Trigger | Payload |
   319‚Üí|-----------|---------|---------|
   320‚Üí| `tool_viewed` | User lands on tool page | `{ toolSlug, referrer, locale }` |
   321‚Üí| `tool_started` | User begins input (uploads file, types text) | `{ toolSlug, inputType }` |
   322‚Üí| `validation_failed` | Input validation fails | `{ toolSlug, field, errorKind }` |
   323‚Üí| `processing_started` | Processing stage begins | `{ toolSlug, inputSummary }` |
   324‚Üí| `processing_completed` | Processing stage completes successfully | `{ toolSlug, durationMs, outputSummary }` |
   325‚Üí| `download_attempted` | User clicks download/copy button | `{ toolSlug, outputFormat }` |
   326‚Üí| `download_completed` | Download/copy succeeds | `{ toolSlug, outputSize }` |
   327‚Üí| `registration_prompt_viewed` | Registration prompt shown | `{ toolSlug, trigger }` |
   328‚Üí| `registration_completed` | User completes registration | `{ toolSlug, method }` |
   329‚Üí| `tool_shared` | User shares tool (link copy, social) | `{ toolSlug, shareMethod }` |
   330‚Üí
   331‚Üí**What it forbids:**
   332‚Üí- Tools emitting custom event names instead of standard ones.
   333‚Üí- Tools omitting any of the 10 mandatory events.
   334‚Üí- Tools sending PII in event payloads (only userId if authenticated).
   335‚Üí
   336‚Üí**Enforcement:**
   337‚Üí- Tool Engine emits events automatically; tools don't write custom analytics code.
   338‚Üí- Manifest declares which events the tool emits (for tools with optional events like `tool_shared`).
   339‚Üí- CI verifies manifest's analytics configuration.
   340‚Üí
   341‚Üí**Interaction with other articles:**
   342‚Üí- PC-02 (Product Contract): Analytics events are part of the contract.
   343‚Üí- PC-10 (Product Scalability): Analytics config derived from manifest.
   344‚Üí

   346‚Üí
   347‚Üí### 5.8 PC-08 ‚Äî Error Experience
   348‚Üí
   349‚Üí**Binding text:**
   350‚Üí> Every error MUST: explain what happened, explain why, explain how to fix it. Never expose technical stack traces.
   351‚Üí
   352‚Üí**Operational meaning:**
   353‚ÜíEvery error has three components:
   354‚Üí
   355‚Üí1. **What happened** (plain language): "The image you uploaded is too large."
   356‚Üí2. **Why** (if known): "The maximum file size is 10MB; your file is 25MB."
   357‚Üí3. **How to fix it** (actionable): "Please upload a smaller image, or use the Image Compressor tool first."
   358‚Üí
   359‚Üí**Error types** (from `02_SAD` ¬ß6.3):
   360‚Üí- `validation` ‚Äî input validation failed.
   361‚Üí- `processing` ‚Äî tool logic failed (recoverable or not).
   362‚Üí- `quota_exceeded` ‚Äî user hit usage limit.
   363‚Üí- `auth_required` ‚Äî feature requires registration.
   364‚Üí- `server_unavailable` ‚Äî backend service down.
   365‚Üí
   366‚ÜíEach error type has a user-facing message template; tools fill in specifics.
   367‚Üí
   368‚Üí**What it forbids:**
   369‚Üí- Generic "Something went wrong" messages.
   370‚Üí- Raw stack traces or technical error codes visible to users.
   371‚Üí- Errors without recovery guidance.
   372‚Üí- Errors that blame the user ("You entered invalid input").
   373‚Üí
   374‚Üí**Enforcement:**
   375‚Üí- Tool Engine maps errors to user-facing messages.
   376‚Üí- Code review: reviewer verifies error messages meet the three-component standard.
   377‚Üí- PC-04 UX gate: checks error experience.
   378‚Üí
   379‚Üí**Interaction with other articles:**
   380‚Üí- EC-05 (Progressive Enhancement): Errors are graceful degradation points.
   381‚Üí
   382‚Üí---

   384‚Üí### 5.9 PC-09 ‚Äî Feature Discoverability
   385‚Üí
   386‚Üí**Binding text:**
   387‚Üí> Every tool MUST help users discover additional tools. Methods: Related Tools, Suggested Workflows, Category Navigation, Search, Recently Used, Popular Tools. The ecosystem should naturally expand user engagement.
   388‚Üí
   389‚Üí**Operational meaning:**
   390‚ÜíSix discoverability methods (all tools implement at least the first three):
   391‚Üí
   392‚Üí| Method | Requirement | Implementation |
   393‚Üí|--------|-------------|----------------|
   394‚Üí| Related Tools | Min 3 related tools listed in Result section. | Manifest `relatedTools` field; rendered by `RelatedTools` component. |
   395‚Üí| Suggested Workflows | If tool is part of a workflow (e.g., resize ‚Üí compress ‚Üí convert), suggest next steps. | Manifest `suggestedWorkflows` field (optional in Phase 1). |
   396‚Üí| Category Navigation | Tool page links to its category page. | Breadcrumb (LOCK-08) + sidebar. |
   397‚Üí| Search | Tool is searchable in site search. | Auto-indexed from manifest (PC-10). |
   398‚Üí| Recently Used | If authenticated, show recently used tools. | Identity Context history; UI in header. |
   399‚Üí| Popular Tools | Site-wide popular tools shown on category pages. | Derived from Analytics Context (PC-07). |
   400‚Üí
   401‚Üí**What it forbids:**
   402‚Üí- Tools with no related tools declared.
   403‚Üí- Tools that don't link to their category.
   404‚Üí- Dead-end pages (no path to other tools).
   405‚Üí
   406‚Üí**Enforcement:**
   407‚Üí- Manifest validation: `relatedTools` array has ‚â•3 entries.
   408‚Üí- Code review: verifies Related Tools section present.
   409‚Üí- PC-04 SEO gate: checks internal linking.
   410‚Üí
   411‚Üí**Interaction with other articles:**
   412‚Üí- LOCK-08 (SEO): Internal linking strengthens SEO.
   413‚Üí- PC-05 (UX Consistency): Related Tools section is part of canonical layout.
   414‚Üí

   417‚Üí### 5.10 PC-10 ‚Äî Product Scalability
   418‚Üí
   419‚Üí**Binding text:**
   420‚Üí> Every new tool should require minimal engineering effort. The Tool Manifest MUST contain enough metadata to automatically generate: Navigation, SEO, Sitemap, Categories, Search Index, Admin Inventory, Analytics Configuration. Metadata-first development is mandatory.
   421‚Üí
   422‚Üí**Operational meaning:**
   423‚ÜíAdding a tool = creating a folder with `manifest.ts` + stage files. The build-time codegen (`05_ProjectStructure` AD-04) automatically:
   424‚Üí
   425‚Üí| Generated Artifact | Source Manifest Field |
   426‚Üí|--------------------|-----------------------|
   427‚Üí| Navigation entries | `category`, `slug`, `title` |
   428‚Üí| SEO metadata | `seo` (title, description, OG, Twitter Card, JSON-LD) |
   429‚Üí| Sitemap entries | `slug`, `category`, `lastModified` |
   430‚Üí| Category taxonomies | `category` |
   431‚Üí| Search index | `title`, `description`, `keywords`, `faq` |
   432‚Üí| Admin inventory | All manifest fields |
   433‚Üí| Analytics configuration | `analytics.events` |
   434‚Üí
   435‚Üí**What it forbids:**
   436‚Üí- Manual registration of tools in navigation, sitemap, or admin.
   437‚Üí- Tools that require infrastructure changes outside their own folder.
   438‚Üí- Hardcoded tool lists anywhere in the codebase.
   439‚Üí
   440‚Üí**Enforcement:**
   441‚Üí- CI verifies generated files match manifests (`scripts/verify-registry.ts`).
   442‚Üí- Code review: reviewer verifies no manual registration.
   443‚Üí- PC-04 Documentation gate: checks manifest completeness.
   444‚Üí
   445‚Üí**Interaction with other articles:**
   446‚Üí- LOCK-05 (Plugin-Ready): Manifest is the plugin contract.
   447‚Üí- PC-02 (Product Contract): Manifest encodes the contract.
   448‚Üí- EC-04 (Tool Template): Template ensures manifest completeness.

   466‚Üí## 7. Best Practices
   467‚Üí
   468‚Üí### 7.1 When Designing a New Tool
   469‚Üí1. Define the ONE problem the tool solves (PC-01).
   470‚Üí2. Draft the product contract (PC-02) ‚Äî all 13 fields.
   471‚Üí3. Plan the 13 completion items (PC-03).
   472‚Üí4. Plan for all 7 quality gates (PC-04) before requesting promotion.
   473‚Üí5. Use the canonical layout (PC-05).
   474‚Üí6. Place monetization only after result (PC-06).
   475‚Üí7. Plan the 10 analytics events (PC-07).
   476‚Üí8. Draft error messages with what/why/how (PC-08).
   477‚Üí9. Identify 3+ related tools (PC-09).
   478‚Üí10. Ensure manifest is complete for codegen (PC-10).
   479‚Üí
   480‚Üí### 7.2 When Reviewing a Tool PR
   481‚Üí- Verify PC-01: single intent?
   482‚Üí- Verify PC-02: contract complete in manifest?
   483‚Üí- Verify PC-03: all 13 items present?
   484‚Üí- For promotion to Stable: verify all 7 gates passed (PC-04).
   485‚Üí- Verify PC-05: uses `ToolLayout`?
   486‚Üí- Verify PC-06: no monetization in core workflow?
   487‚Üí- Verify PC-07: emits all 10 events (or delegates to Tool Engine)?
   488‚Üí- Verify PC-08: error messages meet three-component standard?
   489‚Üí- Verify PC-09: ‚â•3 related tools declared?
   490‚Üí- Verify PC-10: manifest complete enough for codegen?
   491‚Üí
   492‚Üí### 7.3 When Requesting an Exception
   493‚Üí1. Write an ADR proposing the exception with rationale.
   494‚Üí2. Reference the PC article being excepted.
   495‚Üí3. Explain why the exception doesn't undermine the article's intent.
   496‚Üí4. Get Chief Architect approval.
   497‚Üí5. Document the exception in the tool's README.

   499‚Üí## 8. Examples
   500‚Üí
   501‚Üí### 8.1 Example: Image Resizer Tool
   502‚Üí
   503‚Üí**PC-01 (One Problem):** Resizes images to specified dimensions. Single intent.
   504‚Üí
   505‚Üí**PC-02 (Contract):**
   506‚Üí- Purpose: "Resize an image to specific width and height."
   507‚Üí- User Problem: "I need an image at a specific size for a profile picture / thumbnail / print."
   508‚Üí- Inputs: Image file (PNG, JPEG, WebP), target width, target height, optional maintain-aspect-ratio flag.
   509‚Üí- Outputs: Resized image file (same format as input).
   510‚Üí- Validation Rules: File ‚â§10MB; dimensions 1-10000px; supported formats only.
   511‚Üí- Processing Steps: Decode image ‚Üí resize via Canvas ‚Üí encode ‚Üí return blob.
   512‚Üí- Success Criteria: Output image has exact specified dimensions.
   513‚Üí- Failure States: `validation` (invalid file), `processing` (decode failure), `quota_exceeded` (file too large).
   514‚Üí- Empty States: "Upload an image to resize" with upload icon.
   515‚Üí- Loading States: Progress bar during resize.
   516‚Üí- SEO Intent: Transactional (user wants to DO something).
   517‚Üí- Related Tools: `image-compress`, `image-crop`, `image-format-convert`.
   518‚Üí- Analytics Events: All 10 standard events.
   519‚Üí
   520‚Üí**PC-03 (Completion):** All 13 items implemented.
   521‚Üí
   522‚Üí**PC-04 (Gates):** Passed all 7 gates; promoted to Stable.
   523‚Üí
   524‚Üí**PC-05 (Layout):** Uses `ToolLayout` with all 8 sections.
   525‚Üí
   526‚Üí**PC-06 (Monetization):** No ads before result. Ad appears after download. Premium upsell for batch resize (not blocking single resize).
   527‚Üí
   528‚Üí**PC-07 (Analytics):** Emits all 10 events via Tool Engine.
   529‚Üí
   530‚Üí**PC-08 (Errors):**
   531‚Üí- File too large: "Your image is too large (25MB). The maximum size is 10MB. Try the Image Compressor first to reduce its size."
   532‚Üí- Invalid format: "This file format isn't supported (GIF). Supported formats are PNG, JPEG, and WebP. Convert your image to one of these formats first."
   533‚Üí
   534‚Üí**PC-09 (Discoverability):** Related Tools section shows Image Compress, Image Crop, Image Format Convert.
   535‚Üí
   536‚Üí**PC-10 (Scalability):** Manifest complete; codegen generates nav, SEO, sitemap, search, admin automatically.
