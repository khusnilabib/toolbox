    11тЖТ    11тЖТ## 1. Purpose
    12тЖТ    12тЖТ
    13тЖТ    13тЖТThis Project Structure document defines the high-level layout of [PROJECT_NAME]'s codebase. It maps the bounded contexts from `03_DDD` and the architectural layers from `02_SAD` to physical directories. It defines the **Tool Registry Pattern** that enables plugin-ready architecture (LOCK-05), the **Module Boundaries** that enforce modularity (LOCK-04), and the **Stateless Tool Layout** that enables database-optional operation (LOCK-06).
    14тЖТ    14тЖТ
    15тЖТ    15тЖТThis document is the authoritative reference for "where does this code belong?" Every new file, every new module, every new tool must be placed according to this document. The structure is not a suggestion; it is enforced via ESLint boundary rules, code review, and CI checks.
    16тЖТ    16тЖТ
    17тЖТ    17тЖТThe structure scales from 30 tools (Phase 1) to 1,000+ tools (Phase 4) without reorganization. The unit of growth is the tool folder; adding a tool creates one new folder and modifies nothing outside the tools directory except auto-generated registry files. This linear scaling property is the single most important structural guarantee.

    40тЖТ    40тЖТ## 3. Architectural Decisions
    41тЖТ    41тЖТ
    42тЖТ    42тЖТ### AD-01 тАФ Feature-Based Architecture at the Top Level
    43тЖТ    43тЖТ
    44тЖТ    44тЖТ**Context.** Two common patterns exist: type-based (folders for `components/`, `hooks/`, `utils/`) and feature-based (folders for `tools/`, `auth/`, `admin/`). Type-based breaks down at scale because related files scatter across many folders; changing one feature requires touching many directories.
    45тЖТ    45тЖТ
    46тЖТ    46тЖТ**Decision.** Adopt feature-based architecture at the top level. Each bounded context (`03_DDD`) is a top-level directory. Within each context, files are organized by architectural layer (`02_SAD` AD-01: Presentation, Application, Domain, Infrastructure).
    47тЖТ    47тЖТ
    48тЖТ    48тЖТ```
    49тЖТ    49тЖТsrc/
    50тЖТ    50тЖТтФЬтФАтФА app/                    # Next.js App Router (routing layer)
    51тЖТ    51тЖТтФЬтФАтФА tools/                  # Tools Context (core domain)
    52тЖТ    52тЖТтФЬтФАтФА identity/               # Identity Context
    53тЖТ    53тЖТтФЬтФАтФА content/                # Content Context
    54тЖТ    54тЖТтФЬтФАтФА platform-ops/           # Platform Operations Context (Admin)
    55тЖТ    55тЖТтФЬтФАтФА billing/                # Billing Context (Phase 2+)
    56тЖТ    56тЖТтФЬтФАтФА analytics/              # Analytics Context
    57тЖТ    57тЖТтФЬтФАтФА shared/                 # Cross-cutting shared code
    58тЖТ    58тЖТтФФтФАтФА generated/              # Build-time generated files (registry, etc.)
    59тЖТ    59тЖТ```
    60тЖТ    60тЖТ
    61тЖТ    61тЖТ**Consequences.**
    62тЖТ    62тЖТ- тЬЕ Each context is self-contained; team can own a context end-to-end.
    63тЖТ    63тЖТ- тЬЕ Tool count scales linearly тАФ each tool is a folder under `tools/`.
    64тЖТ    64тЖТ- тЬЕ Boundary enforcement is structural (folder-per-context) + tooling (ESLint).
    65тЖТ    65тЖТ- тЪая╕П Some shared utilities needed across contexts; mitigated by `/shared` and `/packages`.
    66тЖТ    66тЖТ
    67тЖТ    67тЖТ**Implements:** LOCK-04 (Modular Architecture), EC-03 (Component Reuse First тАФ shared code lives in `/packages` and `/src/shared`).
    68тЖТ    68тЖТ
    69тЖТ    69тЖТ### AD-02 тАФ Layered Folders Within Each Context
    70тЖТ    70тЖТ
    71тЖТ    71тЖТ**Context.** `02_SAD` AD-01 mandates four layers (Presentation, Application, Domain, Infrastructure) with strict dependency rules. The folder structure must make these layers visible and enforceable.
    72тЖТ    72тЖТ
    73тЖТ    73тЖТ**Decision.** Within each bounded context directory, four subdirectories represent the layers:
    74тЖТ    74тЖТ
    75тЖТ    75тЖТ```
    76тЖТ    76тЖТsrc/identity/
    77тЖТ    77тЖТтФЬтФАтФА presentation/           # React components, pages (depends on Application)
    78тЖТ    78тЖТтФЬтФАтФА application/            # Use cases, hooks, server actions (depends on Domain)
    79тЖТ    79тЖТтФЬтФАтФА domain/                 # Entities, value objects, schemas (no external deps)
    80тЖТ    80тЖТтФФтФАтФА infrastructure/         # DB access, external APIs (depends on Domain)
    81тЖТ    81тЖТ```
    82тЖТ    82тЖТ
    83тЖТ    83тЖТDependency direction is enforced via ESLint `no-restricted-imports`:
    84тЖТ    84тЖТ- `presentation/` may import from `application/`, `domain/`, `shared/`.
    85тЖТ    85тЖТ- `application/` may import from `domain/`, `shared/`.
    86тЖТ    86тЖТ- `domain/` may import from `shared/` only.
    87тЖТ    87тЖТ- `infrastructure/` may import from `domain/`, `shared/`.
    88тЖТ    88тЖТ
    89тЖТ    89тЖТ**Consequences.**
    90тЖТ    90тЖТ- тЬЕ Layer boundaries are visible in any IDE file tree.
    91тЖТ    91тЖТ- тЬЕ ESLint catches violations at lint time, not review time.
    92тЖТ    92тЖТ- тЬЕ Domain logic is fully testable without React or DB setup.
    93тЖТ    93тЖТ- тЪая╕П Slightly more nesting than flat structure; justified by enforceability.
    94тЖТ    94тЖТ
    95тЖТ    95тЖТ**Implements:** LOCK-04, `02_SAD` AD-01.
    96тЖТ    96тЖТ
    97тЖТ    97тЖТ### AD-03 тАФ Tool Module Anatomy (One Folder Per Tool)
    98тЖТ    98тЖТ
    99тЖТ    99тЖТ**Context.** LOCK-04 mandates tools are independent modules. LOCK-05 mandates plugin-ready architecture. The structure must make adding a tool a self-contained operation: create a folder, define a manifest, implement stages.
   100тЖТ   100тЖТ
   101тЖТ   101тЖТ**Decision.** Each tool lives in `src/tools/[category]/[slug]/` and contains:
   102тЖТ   102тЖТ
   103тЖТ   103тЖТ```
   104тЖТ   104тЖТsrc/tools/image/image-resize/
   105тЖТ   105тЖТтФЬтФАтФА manifest.ts             # ToolManifest тАФ the aggregate root (LOCK-05)
   106тЖТ   106тЖТтФЬтФАтФА stages/
   107тЖТ   107тЖТтФВ   тФЬтФАтФА input.ts            # InputStage implementation
   108тЖТ   108тЖТтФВ   тФЬтФАтФА validation.ts       # ValidationStage (Zod schema)
   109тЖТ   109тЖТтФВ   тФЬтФАтФА processing.ts       # ProcessingStage (browser-side or server-side)
   110тЖТ   110тЖТтФВ   тФЬтФАтФА preview.ts          # PreviewStage (React component)
   111тЖТ   111тЖТтФВ   тФЬтФАтФА download.ts         # DownloadStage (packaging logic)
   112тЖТ   112тЖТтФВ   тФЬтФАтФА history.ts          # HistoryStage (optional; best-effort persistence)
   113тЖТ   113тЖТтФВ   тФФтФАтФА share.ts            # ShareStage (optional; link/QR generation)
   114тЖТ   114тЖТтФЬтФАтФА components/             # Tool-specific UI components (input forms, etc.)
   115тЖТ   115тЖТтФЬтФАтФА hooks/                  # Tool-specific hooks
   116тЖТ   116тЖТтФЬтФАтФА tests/
   117тЖТ   117тЖТтФВ   тФЬтФАтФА stages.test.ts      # Unit tests per stage
   118тЖТ   118тЖТтФВ   тФФтФАтФА e2e.test.ts         # E2E test for the tool workflow
   119тЖТ   119тЖТтФЬтФАтФА README.md               # Tool-specific docs (optional, for complex tools)
   120тЖТ   120тЖТтФФтФАтФА index.ts                # Re-exports manifest as default
   121тЖТ   121тЖТ```
   122тЖТ   122тЖТ
   123тЖТ   123тЖТ**Consequences.**
   124тЖТ   124тЖТ- тЬЕ Adding a tool = adding a folder; nothing else changes (registry auto-discovers).
   125тЖТ   125тЖТ- тЬЕ Tool is fully isolated; deleting the folder removes the tool completely.
   126тЖТ   126тЖТ- тЬЕ Tests colocated with implementation.
   127тЖТ   127тЖТ- тЬЕ Plugin-ready (LOCK-05): a third-party tool follows the same structure.
   128тЖТ   128тЖТ- тЪая╕П Many small folders; mitigated by clear naming and IDE support.
   129тЖТ   129тЖТ
   130тЖТ   130тЖТ**Implements:** LOCK-04, LOCK-05, LOCK-03 (Tool Engine stages map to files), EC-04 (Tool Template Standard тАФ every tool has identical structure).
   131тЖТ   131тЖТ
   132тЖТ   132тЖТ### AD-04 тАФ Tool Registry Pattern (Build-Time Codegen)
   133тЖТ   133тЖТ
   134тЖТ   134тЖТ**Context.** LOCK-05 mandates auto-discovery. Runtime registry lookups add latency and complicate Edge deployment. Build-time codegen produces static typed files that Edge can serve.
   135тЖТ   135тЖТ
   136тЖТ   136тЖТ**Decision.** A build script (`scripts/generate-registry.ts`) walks `src/tools/**/manifest.ts`, imports each manifest, validates it against the schema, and emits typed registry files:
   137тЖТ   137тЖТ
   138тЖТ   138тЖТ```
   139тЖТ   139тЖТsrc/generated/
   140тЖТ   140тЖТтФЬтФАтФА registry.ts             # Map<slug, ToolManifest> with helpers
   141тЖТ   141тЖТтФЬтФАтФА navigation.ts           # Category тЖТ Tool[] for nav menus
   142тЖТ   142тЖТтФЬтФАтФА sitemap.ts              # Sitemap entries for SEO
   143тЖТ   143тЖТтФЬтФАтФА seo-meta.ts             # Per-route SEO metadata (LOCK-08)
   144тЖТ   144тЖТтФЬтФАтФА admin-inventory.ts      # Tool list for admin panel
   145тЖТ   145тЖТтФФтФАтФА types.ts                # Generated TypeScript types
   146тЖТ   146тЖТ```
   147тЖТ   147тЖТ
   148тЖТ   148тЖТThe script runs:
   149тЖТ   149тЖТ- On every `pnpm dev` start (development).
   150тЖТ   150тЖТ- On every `pnpm build` (production).
   151тЖТ   151тЖТ- Via `pnpm gen:registry` (manual).
   152тЖТ   152тЖТ
   153тЖТ   153тЖТCI verifies that generated files are committed and match the manifests; mismatched generations fail the build.
   154тЖТ   154тЖТ
   155тЖТ   155тЖТ**Consequences.**
   156тЖТ   156тЖТ- тЬЕ Zero runtime cost for registry lookups.
   157тЖТ   157тЖТ- тЬЕ Type safety: malformed manifests fail the build.
   158тЖТ   158тЖТ- тЬЕ Auto-discovery: new tools appear in nav, sitemap, admin without manual wiring.
   159тЖТ   159тЖТ- тЬЕ Plugin-ready: third-party manifests follow the same schema; can be merged into registry.
   160тЖТ   160тЖТ- тЪая╕П Build time grows linearly with tool count; mitigated by parallel codegen at >500 tools.
   161тЖТ   161тЖТ
   162тЖТ   162тЖТ**Implements:** LOCK-05 (Plugin-Ready, auto-discovery), LOCK-08 (SEO metadata in registry).
   163тЖТ   163тЖТ
   164тЖТ   164тЖТ### AD-05 тАФ Shared Code Split Between `/packages` and `/src/shared`
   165тЖТ   165тЖТ
   166тЖТ   166тЖТ**Context.** Some code is shared across bounded contexts (UI primitives, utility functions, types). Where it lives affects reusability and dependency direction.
   167тЖТ   167тЖТ
   168тЖТ   168тЖТ**Decision.** Two locations for shared code, with distinct purposes:
   169тЖТ   169тЖТ
   170тЖТ   170тЖТ**`/packages/*`** тАФ published-style packages, fully self-contained, no dependencies on `src/`. Used for code that could theoretically be extracted to a separate npm package later. Examples:
   171тЖТ   171тЖТ- `/packages/tool-engine` тАФ the `ToolEngine<TInput, TOutput>` abstraction (LOCK-03).
   172тЖТ   172тЖТ- `/packages/ui` тАФ design system primitives (built on shadcn/ui, per `09_DesignSystem`).
   173тЖТ   173тЖТ- `/packages/utils` тАФ pure utility functions (formatting, math, etc.).
   174тЖТ   174тЖТ- `/packages/types` тАФ shared TypeScript types and Zod schemas.
   175тЖТ   175тЖТ
   176тЖТ   176тЖТ**`/src/shared/*`** тАФ application-specific shared code that depends on `src/` structure but is used across contexts. Examples:
   177тЖТ   177тЖТ- `/src/shared/lib` тАФ application utilities (auth client, supabase client).
   178тЖТ   178тЖТ- `/src/shared/config` тАФ env vars, feature flags client.
   179тЖТ   179тЖТ- `/src/shared/hooks` тАФ cross-context hooks (useUser, useLocale).
   180тЖТ   180тЖТ
   181тЖТ   181тЖТ**Consequences.**
   182тЖТ   182тЖТ- тЬЕ `/packages` code is highly reusable and could be open-sourced later.
   183тЖТ   183тЖТ- тЬЕ `/src/shared` keeps context-spanning app code organized.
   184тЖТ   184тЖТ- тЬЕ Clear ownership: `/packages` owned by platform team; `/src/shared` owned by whoever needs it.
   185тЖТ   185тЖТ- тЪая╕П Discipline required to put the right code in the right place; documented in `06_FolderStructure`.
   186тЖТ   186тЖТ
   187тЖТ   187тЖТ**Implements:** LOCK-04 (modularity), DRY principle.
   188тЖТ   188тЖТ
   189тЖТ   189тЖТ### AD-06 тАФ App Router Maps to Tool Registry
   190тЖТ   190тЖТ
   191тЖТ   191тЖТ**Context.** Next.js App Router uses file-based routing. With 1,000+ tools, having one route file per tool is unmanageable. But dynamic routes (`/tools/[category]/[slug]`) need to know which slugs are valid.
   192тЖТ   192тЖТ
   193тЖТ   193тЖТ**Decision.** Use Next.js dynamic routes that resolve against the generated registry:
   194тЖТ   194тЖТ
   195тЖТ   195тЖТ```
   196тЖТ   196тЖТsrc/app/
   197тЖТ   197тЖТтФЬтФАтФА (public)/
   198тЖТ   198тЖТтФВ   тФЬтФАтФА tools/[category]/[slug]/
   199тЖТ   199тЖТтФВ   тФВ   тФФтФАтФА page.tsx         # Single dynamic route for all tools
   200тЖТ   200тЖТтФВ   тФЬтФАтФА tools/[category]/
   201тЖТ   201тЖТтФВ   тФВ   тФФтФАтФА page.tsx         # Category landing page
   202тЖТ   202тЖТтФВ   тФФтФАтФА tools/
   203тЖТ   203тЖТтФВ       тФФтФАтФА page.tsx         # All-tools hub page
   204тЖТ   204тЖТтФЬтФАтФА (auth)/
   205тЖТ   205тЖТтФВ   тФЬтФАтФА login/page.tsx
   206тЖТ   206тЖТтФВ   тФЬтФАтФА register/page.tsx
   207тЖТ   207тЖТтФВ   тФФтФАтФА dashboard/page.tsx
   208тЖТ   208тЖТтФЬтФАтФА (admin)/
   209тЖТ   209тЖТтФВ   тФФтФАтФА admin/               # Per `18_AdminSpecification`
   210тЖТ   210тЖТтФФтФАтФА (marketing)/
   211тЖТ   211тЖТ    тФЬтФАтФА page.tsx             # Homepage
   212тЖТ   212тЖТ    тФЬтФАтФА about/page.tsx
   213тЖТ   213тЖТ    тФФтФАтФА blog/page.tsx        # Content Context articles
   214тЖТ   214тЖТ```
   215тЖТ   215тЖТ
   216тЖТ   216тЖТThe `[slug]/page.tsx` route:
   217тЖТ   217тЖТ1. Reads `params.slug` and `params.category`.
   218тЖТ   218тЖТ2. Looks up the manifest in the generated registry.
   219тЖТ   219тЖТ3. If not found тЖТ 404.
   220тЖТ   220тЖТ4. If found тЖТ renders the Tool Engine UI, lazy-loads the tool's Processing stage.
   221тЖТ   221тЖТ5. Generates SEO metadata from the manifest (LOCK-08).
   222тЖТ   222тЖТ
   223тЖТ   223тЖТ**Consequences.**
   224тЖТ   224тЖТ- тЬЕ One route file handles all tools; no route explosion.
   225тЖТ   225тЖТ- тЬЕ Sitemap, SEO metadata, navigation all derived from registry.
   226тЖТ   226тЖТ- тЬЕ Static generation possible for known slugs (via `generateStaticParams`).
   227тЖТ   227тЖТ- тЪая╕П Careful handling needed for dynamic params validation.
   228тЖТ   228тЖТ
   229тЖТ   229тЖТ**Implements:** LOCK-05 (registry-driven routing), LOCK-08 (SEO from manifest).
   230тЖТ   230тЖТ
   231тЖТ   231тЖТ### AD-07 тАФ Database-Optional Layout
   232тЖТ   232тЖТ
   233тЖТ   233тЖТ**Context.** LOCK-06 mandates that core tools function without the database. The structure must make this obvious: the Tools Context directory must have no DB imports.
   234тЖТ   234тЖТ
   235тЖТ   235тЖТ**Decision.** The `src/tools/` directory is structurally forbidden from importing anything from `infrastructure/` layers of any context. This is enforced via ESLint `no-restricted-imports`:
   236тЖТ   236тЖТ
   237тЖТ   237тЖТ```javascript
   238тЖТ   238тЖТ// .eslintrc тАФ tools context boundary
   239тЖТ   239тЖТ{
   240тЖТ   240тЖТ  files: ['src/tools/**/*'],
   241тЖТ   241тЖТ  rules: {
   242тЖТ   242тЖТ    'no-restricted-imports': ['error', {
   243тЖТ   243тЖТ      patterns: [
   244тЖТ   244тЖТ        '@/identity/infrastructure/*',
   245тЖТ   245тЖТ        '@/content/infrastructure/*',
   246тЖТ   246тЖТ        '@/platform-ops/infrastructure/*',
   247тЖТ   247тЖТ        '@/billing/infrastructure/*',
   248тЖТ   248тЖТ        '@/analytics/infrastructure/*',
   249тЖТ   249тЖТ        // Tools context has its OWN infrastructure only for server-side tools
   250тЖТ   250тЖТ        // but it cannot depend on OTHER contexts' infrastructure
   251тЖТ   251тЖТ      ]
   252тЖТ   252тЖТ    }]
   253тЖТ   253тЖТ  }
   254тЖТ   254тЖТ}
   255тЖТ   255тЖТ```
   256тЖТ   256тЖТ
   257тЖТ   257тЖТTools that need to persist data (e.g., history) do so via published interfaces (server actions in Identity Context), not direct DB calls.
   258тЖТ   258тЖТ
   259тЖТ   259тЖТ**Consequences.**
   260тЖТ   260тЖТ- тЬЕ Structural enforcement of LOCK-06; cannot be violated by accident.
   261тЖТ   261тЖТ- тЬЕ Tools are testable without DB setup.
   262тЖТ   262тЖТ- тЬЕ If DB is down, tools still work (server actions fail gracefully).
   263тЖТ   263тЖТ- тЪая╕П Slightly more indirection for persistence; justified by LOCK-06.
   264тЖТ   264тЖТ
   265тЖТ   265тЖТ**Implements:** LOCK-06 (Database Optional), EC-05 (Progressive Enhancement тАФ structural enforcement of graceful degradation).
   266тЖТ   266тЖТ
   267тЖТ   267тЖТ## 4. Design Principles
   268тЖТ   268тЖТ
   269тЖТ   269тЖТ### P1 тАФ One Tool, One Folder
   270тЖТ   270тЖТEvery tool is one folder. Adding a tool = adding a folder. Removing a tool = deleting a folder. No side effects on other tools.
   271тЖТ   271тЖТ
   272тЖТ   272тЖТ### P2 тАФ Layers Are Visible
   273тЖТ   273тЖТThe four layers (Presentation, Application, Domain, Infrastructure) are visible as subdirectories within each context. This makes architectural decisions inspectable.
   274тЖТ   274тЖТ
   275тЖТ   275тЖТ### P3 тАФ Generated Files Are Separate
   276тЖТ   276тЖТBuild-time generated files live in `/src/generated/` and are clearly marked. They are committed to the repo (for type safety across the team) but never manually edited.
   277тЖТ   277тЖТ
   278тЖТ   278тЖТ### P4 тАФ Shared Code Has a Home
   279тЖТ   279тЖТCross-context code lives in `/packages` (if reusable) or `/src/shared` (if app-specific). Code is never duplicated across contexts (LOCK-09).
   280тЖТ   280тЖТ
   281тЖТ   281тЖТ### P5 тАФ Boundaries Are Enforced
   282тЖТ   282тЖТFolder structure alone is insufficient; ESLint boundary rules enforce dependency direction. Code review verifies boundary compliance.
   283тЖТ   283тЖТ
   284тЖТ   284тЖТ### P6 тАФ Growth Is Linear
   285тЖТ   285тЖТAdding tools, articles, admin features, or users scales linearly: one new folder per item, no reorganization of existing folders.

   517тЖТ   517тЖТ## 8. Module Boundaries
   518тЖТ   518тЖТ
   519тЖТ   519тЖТ### 8.1 Boundary Enforcement
   520тЖТ   520тЖТ
   521тЖТ   521тЖТBoundaries are enforced at three levels:
   522тЖТ   522тЖТ
   523тЖТ   523тЖТ1. **Folder structure** тАФ visible in IDE; humans can see boundaries.
   524тЖТ   524тЖТ2. **ESLint `no-restricted-imports`** тАФ automated; fails lint on violation.
   525тЖТ   525тЖТ3. **Code review** тАФ final check; reviewers verify boundary compliance.
   526тЖТ   526тЖТ
   527тЖТ   527тЖТ### 8.2 Boundary Rules
   528тЖТ   528тЖТ
   529тЖТ   529тЖТ| Source тЖТ Target | Allowed? | Reason |
   530тЖТ   530тЖТ|-----------------|----------|--------|
   531тЖТ   531тЖТ| Tool тЖТ Tool (same tool) | тЬЕ | Same module |
   532тЖТ   532тЖТ| Tool тЖТ Tool (different tool) | тЭМ | LOCK-04 isolation |
   533тЖТ   533тЖТ| Tool тЖТ Tool `_shared/` | тЬЕ | Tools-context-internal shared |
   534тЖТ   534тЖТ| Tool тЖТ `@packages/*` | тЬЕ | Reusable packages |
   535тЖТ   535тЖТ| Tool тЖТ `@/shared/*` | тЬЕ | App-specific shared |
   536тЖТ   536тЖТ| Tool тЖТ `@/identity/infrastructure/*` | тЭМ | LOCK-06 |
   537тЖТ   537тЖТ| Tool тЖТ `@/content/infrastructure/*` | тЭМ | LOCK-06 |
   538тЖТ   538тЖТ| Tool тЖТ Server Action (Identity) | тЬЕ | Published interface |
   539тЖТ   539тЖТ| `presentation/` тЖТ `infrastructure/` | тЭМ | Layer violation |
   540тЖТ   540тЖТ| `domain/` тЖТ `react` | тЭМ | Layer violation |
   541тЖТ   541тЖТ| Context A тЖТ Context B (internal) | тЭМ | Use published API or events |
   542тЖТ   542тЖТ
   543тЖТ   543тЖТ### 8.3 Server Actions as Published Interfaces
   544тЖТ   544тЖТ
   545тЖТ   545тЖТServer actions are the sanctioned way for tools (and other contexts) to invoke cross-context functionality:
   546тЖТ   546тЖТ
   547тЖТ   547тЖТ```typescript
   548тЖТ   548тЖТ// src/identity/application/actions/history.ts
   549тЖТ   549тЖТ'use server';
   550тЖТ   550тЖТexport async function saveHistoryEntry(entry: HistoryEntryInput) {
   551тЖТ   551тЖТ  // Identity Context owns this; tools call it
   552тЖТ   552тЖТ}
   553тЖТ   553тЖТ

   588тЖТ   588тЖТ## 10. Standards
   589тЖТ   589тЖТ
   590тЖТ   590тЖТ### 10.1 Directory Naming Standards
   591тЖТ   591тЖТ- All directories: `kebab-case`.
   592тЖТ   592тЖТ- Tool slugs: `kebab-case`, URL-safe, unique within category.
   593тЖТ   593тЖТ- Context directories: match bounded context name (`identity`, `content`, etc.).
   594тЖТ   594тЖТ
   595тЖТ   595тЖТ### 10.2 File Placement Standards
   596тЖТ   596тЖТ- A file belongs to exactly one layer within one context.
   597тЖТ   597тЖТ- Shared utilities go in `/packages` (if reusable) or `/src/shared` (if app-specific).
   598тЖТ   598тЖТ- Tests are colocated with the code they test (`tests/` subdirectory).
   599тЖТ   599тЖТ
   600тЖТ   600тЖТ### 10.3 Generated File Standards
   601тЖТ   601тЖТ- All generated files start with `// AUTO-GENERATED. Do not edit manually.`
   602тЖТ   602тЖТ- Generated files live in `/src/generated/` only.
   603тЖТ   603тЖТ- Manual edits to generated files are reverted by CI.
   604тЖТ   604тЖТ
   605тЖТ   605тЖТ### 10.4 Tool Folder Standards
   606тЖТ   606тЖТ- Every tool folder MUST contain `manifest.ts` and `index.ts`.
   607тЖТ   607тЖТ- Every tool folder MUST contain `stages/` with at least 5 stage files (Input, Validation, Processing, Preview, Download).
   608тЖТ   608тЖТ- Every tool folder MUST contain `tests/` with at least one stage test.
   609тЖТ   609тЖТ
   610тЖТ   610тЖТ## 11. Best Practices
   611тЖТ   611тЖТ
   612тЖТ   612тЖТ### 11.1 When Adding a New Tool
   613тЖТ   613тЖТ1. Create folder: `src/tools/[category]/[slug]/`.
   614тЖТ   614тЖТ2. Copy the tool template (`scripts/tool-template/`) into the new folder.
   615тЖТ   615тЖТ3. Edit `manifest.ts` with tool metadata.
   616тЖТ   616тЖТ4. Implement each stage file.
   617тЖТ   617тЖТ5. Run `pnpm gen:registry`.
   618тЖТ   618тЖТ6. Verify the tool appears in navigation.
   619тЖТ   619тЖТ7. Run `pnpm test` and `pnpm lint`.
   620тЖТ   620тЖТ8. PR includes the new folder + updated generated files.
   621тЖТ   621тЖТ
   622тЖТ   622тЖТ### 11.2 When Refactoring a Tool
   623тЖТ   623тЖТ- Keep changes within the tool folder where possible.
   624тЖТ   624тЖТ- If changes affect `_shared/` or `@packages/*`, get Chief Architect approval (LOCK-09).
   625тЖТ   625тЖТ- Update tests in the same PR.
   626тЖТ   626тЖТ
   627тЖТ   627тЖТ### 11.3 When Adding a New Bounded Context
   628тЖТ   628тЖТ1. Get Chief Architect approval (requires DDD amendment per `03_DDD`).
   629тЖТ   629тЖТ2. Create `src/[context-name]/` with four layer subdirectories.
   630тЖТ   630тЖТ3. Create `drizzle/[context-name]/` for migrations.
   631тЖТ   631тЖТ4. Add context to ESLint boundary rules.
   632тЖТ   632тЖТ5. Update this document via PR.
   633тЖТ   633тЖТ
   634тЖТ   634тЖТ### 11.4 When Moving Code Between Layers
   635тЖТ   635тЖТ- Moving code DOWN (Presentation тЖТ Application) is usually safe.
   636тЖТ   636тЖТ- Moving code UP (Domain тЖТ Application) requires justification; Domain should be stable.
   637тЖТ   637тЖТ- Moving code between contexts requires DDD amendment.
   638тЖТ   638тЖТ
   639тЖТ   639тЖТ## 12. Future Scalability
   640тЖТ   640тЖТ
   641тЖТ   641тЖТ### 12.1 Scaling to 1,000+ Tools
   642тЖТ   642тЖТ- Folder structure scales linearly; no reorganization needed.
   643тЖТ   643тЖТ- Build time scales linearly; mitigations at >500 tools:
   644тЖТ   644тЖТ  - Parallel manifest import during codegen.
   645тЖТ   645тЖТ  - Cache codegen output; invalidate only on manifest change.
   646тЖТ   646тЖТ  - Consider moving to a single index file per category.
   647тЖТ   647тЖТ
   648тЖТ   648тЖТ### 12.2 Scaling to Multiple Teams
   649тЖТ   649тЖТ- Each team owns one or more bounded contexts.
   650тЖТ   650тЖТ- Cross-team changes via published interfaces (server actions, events).
   651тЖТ   651тЖТ- `/packages` owned by platform team; changes require their review.
   652тЖТ   652тЖТ
   653тЖТ   653тЖТ### 12.3 Preparing for Plugin Marketplace (Phase 4)
   654тЖТ   654тЖТ- Tool folder structure is the plugin contract.
   655тЖТ   655тЖТ- Phase 4 adds:
   656тЖТ   656тЖТ  - `/plugins/` directory for third-party tools (separate from `/src/tools/`).
   657тЖТ   657тЖТ  - Plugin manifest signing (verifies publisher).
   658тЖТ   658тЖТ  - Plugin sandboxing (limits APIs available to third-party code).
   659тЖТ   659тЖТ  - Plugin versioning (manifest declares supported platform version).
   660тЖТ   660тЖТ- No changes to first-party tool structure; plugins follow the same anatomy.
   661тЖТ   661тЖТ
   662тЖТ   662тЖТ### 12.4 Preparing for Multi-Tenancy (Phase 3+)
   663тЖТ   663тЖТ- Each context's domain layer gains `tenantId` on relevant aggregates.
   664тЖТ   664тЖТ- Infrastructure layer enforces tenant isolation in queries.
   665тЖТ   665тЖТ- No structural changes to top-level layout.
   666тЖТ   666тЖТ
   667тЖТ   667тЖТ### 12.5 Preparing for Mobile App (Phase 3+)
   668тЖТ   668тЖТ- `/packages` code (types, schemas, utilities) is portable to React Native.
   669тЖТ   669тЖТ- `/src/tools/` Processing stages are portable (pure TS).
   670тЖТ   670тЖТ- UI layers rebuilt natively; share types via `/packages/types`.
   671тЖТ   671тЖТ
