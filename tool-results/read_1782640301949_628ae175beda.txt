    10→## 1. Purpose
    11→
    12→This Project Charter formally authorizes the existence of **[PROJECT_NAME]** — a **browser-first productivity ecosystem that enables users to solve everyday digital tasks without installing software and without requiring an account.** The platform consolidates hundreds to thousands of utility tools under a single, scalable, privacy-respecting surface. The charter defines the project's mission, vision, scope, guiding principles, twelve **Architectural Locks** (§3), twelve **Engineering Constitution** articles (§4), ten **Product Constitution** articles (§5), ten **Data & Growth Architecture** articles (§6), ten **Platform Operations Constitution** articles (§7), and governance model. It serves as the foundational reference from which all subsequent architecture, business, engineering, product, growth, and operational decisions derive.
    13→
    14→Every downstream document in the `/docs` repository must remain consistent with this charter, the twelve Architectural Locks in §3, the twelve Engineering Constitution articles in §4, the ten Product Constitution articles in §5, the ten Data & Growth Architecture articles in §6, and the ten Platform Operations Constitution articles in §7. Priority order: Architectural Locks > Engineering Constitution > Product Constitution > Data & Growth Architecture > Platform Operations Constitution > rest of charter > all other documents. If any document conflicts, the higher-priority layer wins until formally amended through the revision process defined in §15. This rule exists because architectural drift is the leading cause of platform decay; without a single authoritative source, teams gradually make inconsistent micro-decisions that compound into structural incoherence.
    15→
    16→**Product decisions are architectural decisions.** The Product Constitution binds every tool, present and future, with the same authority as the Architectural Locks and Engineering Constitution. A tool that violates PC-01 through PC-10 cannot be promoted to Stable (PC-04).
    17→
    18→**Data & growth decisions are architectural decisions.** The Data & Growth Architecture binds every analytics event, SEO element, search index, feature flag, audit log, API version, and growth metric with the same authority. A feature that violates DGA-01 through DGA-10 cannot ship to production.
    19→
    20→**Operational decisions are architectural decisions.** The Platform Operations Constitution binds every deployment, monitoring configuration, backup procedure, security control, incident response, and cost consideration with the same authority. A deployment that violates POC-01 through POC-10 cannot proceed to production.
    21→
    22→The charter is intentionally short on implementation detail. It defines *what* and *why*, never *how*. The *how* lives in `02_SAD`, `05_ProjectStructure`, `13_DatabaseDesign`, and related technical documents.
    23→
    24→## 2. Platform Identity
    25→
    26→[PROJECT_NAME] is **NOT** simply an online tools website. The official positioning is:
    27→
    28→> **A browser-first productivity ecosystem that enables users to solve everyday digital tasks without installing software and without requiring an account.**
    29→
    30→The phrase **"Browser-first Productivity Ecosystem"** is part of the project's permanent identity. Four brand values anchor every product, design, and engineering decision:
    31→
    32→1. **Privacy** — User data stays on the user's device whenever technically possible.
    33→2. **Speed** — Sub-second first interaction; no unnecessary network round-trips for browser-capable tasks.
    34→3. **Simplicity** — Every tool has one obvious primary action; no clutter, no tours, no forced onboarding.
    35→4. **Accessibility** — WCAG 2.1 AA from day 1; usable on a 360px viewport and a screen reader.
    36→
    37→These values are not aspirational. They are evaluation criteria for every PR, every design mock, and every tool added to the backlog.

    41→## 3. Architectural Locks (Permanent Rules)
    42→
    43→The following twelve decisions are **locked** as permanent project rules. They have higher priority than any future document unless explicitly revised through a charter amendment (§10). Every downstream document MUST be consistent with these locks. The lock identifier (e.g., `LOCK-01`) is referenced from every doc that implements or depends on it.
    44→
    45→### LOCK-01 — Platform Identity
    46→
    47→[PROJECT_NAME] is officially positioned as a **browser-first productivity ecosystem**. The full positioning statement is in §2 above. This lock prevents any future repositioning (e.g., "SaaS suite", "online tools directory") without explicit charter amendment.
    48→
    49→### LOCK-02 — Browser-First Philosophy
    50→
    51→Whenever technically possible, processing MUST happen entirely inside the user's browser. This is a platform-level architectural commitment, not a cost-saving measure.
    52→
    53→**Browser-first by default:**
    54→- Image conversion, resize, compression, passport photo, background removal
    55→- PDF merge/split (client-side where feasible via pdf-lib / WASM)
    56→- Text formatting, JSON formatting, code formatters
    57→- UUID generation, password generation, QR generation
    58→- All calculators, unit converters, color converters
    59→
    60→**Server-side only when technically unavoidable:**
    61→- AI inference (summarization, translation, captioning)
    62→- OCR on large documents
    63→- File processing beyond browser memory capacity
    64→- Premium cloud features (batch, scheduled, sync)
    65→
    66→Benefits: better privacy, faster UX, lower infrastructure cost, better scalability, reduced server dependency. See `02_SAD` §Browser-First Decision Matrix for the authoritative classification.
    67→
    68→### LOCK-03 — Tool Engine Philosophy
    69→
    70→Every tool MUST follow one standardized lifecycle:
    71→
    72→```
    73→Input → Validation → Processing → Preview → Download → History (optional) → Share (optional)
    74→```
    75→
    76→Tools MUST reuse this lifecycle instead of implementing custom workflows whenever possible. This becomes the standard **Tool Engine** architecture. The lifecycle is implemented as a typed pipeline (`ToolEngine<TInput, TOutput>`) detailed in `02_SAD` §Tool Engine and `11_ACD` §Tool Engine Component.
    77→
    78→### LOCK-04 — Modular Architecture
    79→
    80→Every tool is an independent module. Modules MUST be:
    81→
    82→- **Independent** — operates without other modules being present or healthy
    83→- **Reusable** — composable into new tools or workflows
    84→- **Replaceable** — swappable without affecting siblings
    85→- **Maintainable** — fixable in isolation; tests run in isolation
    86→- **Discoverable** — registered for navigation, search, and SEO
    87→
    88→A module MUST NOT depend directly on another module's internal implementation. Shared functionality lives only in shared libraries/components under `/packages` and `/src/shared`. See `05_ProjectStructure` §Module Boundaries.
    89→
    90→### LOCK-05 — Plugin-Ready Architecture
    91→
    92→The platform MUST support future plugin/module registration. Each tool MUST eventually expose metadata through a manifest/registry system.
    93→
    94→**Long-term objective:** Adding a new tool should require minimal modification outside the tool's own module. Architecture MUST anticipate auto-discovery and registry-driven navigation. The registry pattern is defined in `05_ProjectStructure` §Tool Registry and the manifest schema in `10_FBRD` §Tool Manifest.
    95→
    96→### LOCK-06 — Database Optional Philosophy
    97→
    98→The database is NOT required for the platform to operate. Core tools MUST continue functioning even if database services are temporarily unavailable.
    99→
   100→**Database responsibilities (with graceful degradation):**
   101→- Authentication, user profile, history, favorites, cloud sync, analytics, admin, content, settings
   102→
   103→**Core tool processing MUST NOT depend on database availability.** Architecture enforces this through strict separation between the **Tool Engine** (stateless, browser-side) and **Platform Services** (stateful, server-side). See `02_SAD` §Layered Architecture and §Graceful Degradation.
   104→
   105→### LOCK-07 — Guest-First User Experience
   106→
   107→Guest users MAY: browse tools, upload files, configure settings, process files, preview results.
   108→
   109→Registration is requested ONLY when users attempt to: download results, save history, favorite tools, synchronize devices, access premium capabilities.
   110→
   111→The onboarding flow MUST maximize trust and minimize friction. **No mandatory registration before demonstrating value.** See `16_UserFlow` and `01_BRD` §4.2.
   112→
   113→### LOCK-08 — SEO Constitution
   114→
   115→SEO is a first-class architectural concern, not a post-launch optimization. Every tool page MUST support:
   116→
   117→- Unique URL
   118→- Unique metadata (title, description, keywords)
   119→- Canonical URL
   120→- Open Graph tags
   121→- Twitter Card tags
   122→- Structured Data (JSON-LD: SoftwareApplication + FAQPage + BreadcrumbList)
   123→- Breadcrumb navigation
   124→- FAQ section
   125→- Related Tools section
   126→- Internal Linking (bidirectional where sensible)
   127→- Search Intent Mapping (informational / transactional / navigational)
   128→
   129→**No duplicate metadata or duplicate content.** SEO requirements influence architecture from the beginning — the URL structure, the routing layer, and the tool manifest schema all encode SEO requirements. See `15_SEOSpecification`.
   130→
   131→### LOCK-09 — AI Development Constitution
   132→
   133→This project is AI-assisted. The AI assistant (and any human using AI tooling) MUST NOT:
   134→
   135→- Duplicate components or business logic
   136→- Change architecture
   137→- Change folder conventions
   138→- Install new dependencies
   139→- Introduce breaking patterns
   140→
   141→...without explicit human approval recorded in the PR description.
   142→
   143→The AI MUST always prefer **extending existing systems** over creating parallel implementations. **Consistency has higher priority than speed.** See `22_AI_Guideline` and `19_DevelopmentGuideline` §AI-Assisted Development.
   144→
   145→### LOCK-10 — Design Philosophy
   146→
   147→Visual language follows **developer-first minimalism** inspired by modern engineering products (Vercel-like).
   148→
   149→Characteristics:
   150→- Monochrome palette (black / white / grays + single accent)
   151→- High contrast (WCAG AAA where feasible)
   152→- Spacious layout (generous padding, breathing room)
   153→- Premium feel (precise typography, deliberate spacing)
   154→- Fast interfaces (no decorative animations, ≤200ms interaction feedback)
   155→- Minimal animation (only state-change animations that aid comprehension)
   156→- Accessible (WCAG 2.1 AA minimum)
   157→- Mobile-first (360px baseline, scales up)
   158→- Dark mode ready (token-driven, no hardcoded colors)
   159→- Light mode ready (default)
   160→
   161→**Avoid decorative UI that does not improve usability.** Every visual element must justify its existence with a usability purpose. See `09_DesignSystem` and `12_UDS`.
   162→
   163→### LOCK-11 — Admin Philosophy
   164→
   165→The Admin Panel is NOT merely a CMS. It is the **operational control center** of the platform.
   166→
   167→Long-term modules include:
   168→- Dashboard, Users, Roles, Permissions
   169→- Categories, Tools, Media
   170→- SEO, Articles, Analytics
   171→- Advertisements
   172→- Feature Flags, Experiments
   173→- System Health, Logs, Audit Trail
   174→- Settings
   175→
   176→Admin architecture MUST anticipate enterprise-scale growth. RBAC enforced from Phase 1 (`17_RBAC`); audit trail mandatory from Phase 1. See `18_AdminSpecification`.
   177→
   178→### LOCK-12 — Feature Lifecycle
   179→
   180→Every feature MUST have a maturity status. The canonical lifecycle:
   181→
   182→```
   183→Concept → Planned → Design → Development → Testing → Beta → Stable → Deprecated → Archived
   184→```
   185→
   186→This lifecycle MUST eventually be represented inside the Admin Panel as a first-class status field on every tool, feature, and module. Each tool's manifest (`11_FBRD` §Tool Manifest) MUST declare its current lifecycle status. Status transitions are audited.

   190→## 4. Engineering Constitution (Mandatory Engineering Rules)
   191→
   192→The Architectural Locks in §3 define **what the platform is**. The Engineering Constitution defines **how engineering is done** — the mandatory discipline that every contributor (human or AI) must follow. These twelve articles supersede implementation preferences. Violations block PR merge.
   193→
   194→Where an Engineering Constitution article implements or extends an Architectural Lock, the lock ID is cited. Where an article is operationalized by a specific document, that document is referenced.
   195→
   196→### EC-01 — Documentation First
   197→
   198→Documentation is the single source of truth. No implementation may exist without corresponding documentation. Every Pull Request must satisfy:
   199→- Documentation updated to reflect the change.
   200→- Architecture remains compliant with all LOCKs and ECs.
   201→- ADR (`06_ArchitectureDecisionRecords`) updated if an architectural decision changes.
   202→
   203→*Implements:* LOCK-09 (AI Discipline). *Operationalized by:* `20_DevelopmentGuideline`, `06_ArchitectureDecisionRecords`.
   204→
   205→### EC-02 — One Source of Truth
   206→
   207→Every business rule must exist in exactly one place. Avoid duplicated constants, duplicated validation logic, duplicated schemas, duplicated utilities, or duplicated business workflows. If duplication is discovered, it must be refactored.
   208→
   209→*Implements:* LOCK-04 (Modular), LOCK-09. *Operationalized by:* `08_CodingStandards` §DRY Enforcement, `12_ACD` §Shared Component Inventory.
   210→
   211→### EC-03 — Component Reuse First
   212→
   213→Before creating a new component, hook, utility, service, or helper:
   214→1. Search existing implementation.
   215→2. Extend existing implementation if appropriate.
   216→3. Create new only when justified.
   217→
   218→The platform grows by composition rather than duplication.
   219→
   220→*Implements:* LOCK-04, LOCK-09. *Operationalized by:* `12_ACD` §Component Catalog, `08_CodingStandards` §Reuse Checklist.
   221→
   222→### EC-04 — Tool Template Standard
   223→
   224→Every tool MUST follow an identical internal structure. Each tool exposes:
   225→- metadata, manifest, validation, processing, preview, download, SEO, tests, documentation.
   226→
   227→No tool may define an entirely custom lifecycle without architectural approval.
   228→
   229→*Implements:* LOCK-03 (Tool Engine), LOCK-05 (Plugin-Ready). *Operationalized by:* `05_ProjectStructure` §Tool Module Anatomy, `11_FBRD` §Tool Manifest, `07_FolderStructure` §Tool Folder Template.
   230→
   231→### EC-05 — Progressive Enhancement
   232→
   233→The platform must remain usable when:
   234→- JavaScript partially fails.
   235→- APIs are temporarily unavailable.
   236→- Database is offline.
   237→- Authentication service is unavailable.
   238→
   239→Whenever possible, gracefully degrade rather than fail.
   240→
   241→*Implements:* LOCK-06 (Database Optional), LOCK-07 (Guest-First). *Operationalized by:* `02_SAD` §Graceful Degradation, `17_UserFlow` §Degraded Journeys.
   242→
   243→### EC-06 — Accessibility First
   244→
   245→Every UI component must satisfy accessibility requirements. Minimum expectations:
   246→- Keyboard navigation, focus visibility, screen reader support, semantic HTML.
   247→- WCAG AA contrast.
   248→- Reduced motion compatibility.
   249→
   250→Accessibility is not optional.
   251→
   252→*Implements:* LOCK-10 (Design Philosophy). *Operationalized by:* `10_DesignSystem` §Accessibility, `13_UDS` §Accessibility Standards, `21_TestingStrategy` §Accessibility Testing.
   253→
   254→### EC-07 — Performance Budget
   255→
   256→Every feature must respect predefined performance budgets. Targets include:
   257→- Fast initial render, minimal JavaScript, lazy loading where appropriate, tree shaking, code splitting, image optimization.
   258→
   259→Performance regressions are treated as bugs.
   260→
   261→*Implements:* LOCK-02 (Browser-First, performance angle). *Operationalized by:* `02_SAD` §Performance Standards, `08_CodingStandards` §Performance Budget, `21_TestingStrategy` §Performance Testing, `22_DeploymentGuide` §Performance Monitoring.
   262→
   263→### EC-08 — Security by Default
   264→
   265→Default assumptions:
   266→- Validate every input. Sanitize every output. Principle of least privilege.
   267→- Secure headers. Strict TypeScript. Secrets never committed.
   268→- Dependency review required.
   269→
   270→Security is proactive, not reactive.
   271→
   272→*Implements:* Supports all LOCKs. *Operationalized by:* `08_CodingStandards` §Security Rules, `15_APIConvention` §Security, `14_DatabaseDesign` §Row-Level Security, `18_RBAC`, `22_DeploymentGuide` §Security Headers.
   273→
   274→### EC-09 — Testing Philosophy
   275→
   276→Testing pyramid: Unit Tests → Integration Tests → End-to-End Tests. Every reusable engine must be testable independently. Tool Engine, Registry, Validators, and Shared Utilities are designed for testing from the beginning.
   277→
   278→*Implements:* LOCK-04 (Maintainable). *Operationalized by:* `21_TestingStrategy`, `08_CodingStandards` §Testability Requirements.
   279→
   280→### EC-10 — Design System Governance
   281→
   282→No page may introduce ad hoc UI patterns. Every visual element must originate from the Design System. Spacing, typography, colors, radius, shadows, icons, and motion remain consistent across the ecosystem.
   283→
   284→*Implements:* LOCK-10 (Design Philosophy). *Operationalized by:* `10_DesignSystem`, `12_ACD` §Design System Components, `08_CodingStandards` §Design System Compliance.
   285→
   286→### EC-11 — AI Collaboration Rules
   287→
   288→AI is a permanent engineering partner. AI must:
   289→- Preserve consistency. Explain architectural trade-offs. Reference existing documentation. Avoid unnecessary abstractions. Minimize technical debt.
   290→
   291→When uncertainty exists: **ask before assuming.**
   292→
   293→*Implements:* LOCK-09 (AI Development Constitution). *Operationalized by:* `23_AI_Guideline`, `24_ZAI_Context`, `20_DevelopmentGuideline` §AI-Assisted Development.
   294→
   295→### EC-12 — Enterprise Readiness
   296→
   297→Although the project begins on free infrastructure (Vercel Free, Supabase Free, GitHub), the architecture must remain capable of scaling to enterprise deployment without major redesign. No implementation should assume free-tier limitations as permanent.
   298→
   299→*Implements:* Supports LOCK-04, `00_Project_Charter` §10 Future Expansion. *Operationalized by:* `04_TechStack` §Upgrade Paths, `02_SAD` §Future Scalability, `24_DeploymentGuide` §Enterprise Migration.

   303→## 5. Product Constitution (Mandatory Product Rules)
   304→
   305→The Architectural Locks in §3 define **what the platform is**. The Engineering Constitution in §4 defines **how engineering is done**. The Product Constitution defines **how every tool behaves as a product** — the binding rules that ensure 1,000+ tools feel like one cohesive ecosystem rather than a patchwork of independent utilities.
   306→
   307→**Product decisions are architectural decisions.** A tool that violates the Product Constitution cannot be promoted to Stable status (PC-04). These ten articles are binding on every tool, present and future, and supersede product preferences. The full text is expanded in `11_ProductConstitution`; this section establishes them as charter-level commitments.
   308→
   309→### PC-01 — Every Tool Solves One Problem
   310→
   311→Each tool MUST solve exactly one clearly defined user problem. Avoid feature creep. One page. One intent. One outcome.
   312→
   313→*Implements:* LOCK-01 (Platform Identity — ecosystem of single-purpose tools). *Operationalized by:* `11_ProductConstitution` §PC-01, `13_FBRD` §Tool Product Contract.
   314→
   315→### PC-02 — Every Tool Has a Product Contract
   316→
   317→Every tool MUST define: Purpose, User Problem, Inputs, Outputs, Validation Rules, Processing Steps, Success Criteria, Failure States, Empty States, Loading States, SEO Intent, Related Tools, Analytics Events. No tool exists without this contract.
   318→
   319→*Implements:* LOCK-03 (Tool Engine), LOCK-05 (Plugin-Ready — manifest encodes contract). *Operationalized by:* `11_ProductConstitution` §PC-02, `12_ToolManifestSpecification`, `13_FBRD` §Tool Product Contract.
   320→
   321→### PC-03 — Tool Completion Standard
   322→
   323→A tool is only considered complete if it provides: Upload/Input, Validation, Processing, Preview, Download/Copy, Error Handling, Success Feedback, Accessibility, Mobile Support, SEO, Analytics, Documentation, Tests. All 13 items mandatory.
   324→
   325→*Implements:* LOCK-03, EC-04, EC-06, EC-09. *Operationalized by:* `11_ProductConstitution` §PC-03, `22_DevelopmentGuideline` §Definition of Done, `23_TestingStrategy`.
   326→
   327→### PC-04 — Product Quality Gates
   328→
   329→A feature cannot reach Stable unless it passes: Functional review, Accessibility review, Performance review, SEO review, Security review, Documentation review, UX review. Quality gates are mandatory.
   330→
   331→*Implements:* LOCK-12 (Feature Lifecycle — Stable requires all gates). *Operationalized by:* `11_ProductConstitution` §PC-04, `22_DevelopmentGuideline` §Quality Gates, `21_AdminSpecification` §Feature Lifecycle.
   332→
   333→### PC-05 — UX Consistency
   334→
   335→Every tool page follows the same layout: Hero → Tool → Result → FAQ → Related Tools → Documentation → Feedback → Footer. Users should never relearn navigation between tools.
   336→
   337→*Implements:* LOCK-10 (Design Philosophy), EC-10 (Design System Governance). *Operationalized by:* `11_ProductConstitution` §PC-05, `15_UDS` §Tool Page Layout, `14_ACD` §Tool Page Components.
   338→
   339→### PC-06 — Monetization Philosophy
   340→
   341→Revenue MUST never interrupt task completion. Advertising or premium prompts may appear only after value has been demonstrated. Core functionality remains free. Premium provides convenience, not necessity.
   342→
   343→*Implements:* LOCK-07 (Guest-First UX), `01_BRD` §4.1 (Monetization Standards). *Operationalized by:* `11_ProductConstitution` §PC-06, `19_UserFlow` §Monetization Touchpoints.
   344→
   345→### PC-07 — Analytics Standard
   346→
   347→Every tool MUST emit consistent events. Minimum events: Tool Viewed, Tool Started, Validation Failed, Processing Started, Processing Completed, Download Attempted, Download Completed, Registration Prompt Viewed, Registration Completed, Tool Shared.
   348→
   349→*Implements:* LOCK-04 (Maintainable — consistent analytics). *Operationalized by:* `11_ProductConstitution` §PC-07, `12_ToolManifestSpecification` §Analytics Configuration, `21_AdminSpecification` §Analytics Module.
   350→
   351→### PC-08 — Error Experience
   352→
   353→Every error MUST: explain what happened, explain why, explain how to fix it. Never expose technical stack traces.
   354→
   355→*Implements:* EC-05 (Progressive Enhancement — graceful error UX). *Operationalized by:* `11_ProductConstitution` §PC-08, `15_UDS` §Error States, `14_ACD` §Error Components.
   356→
   357→### PC-09 — Feature Discoverability
   358→
   359→Every tool MUST help users discover additional tools. Methods: Related Tools, Suggested Workflows, Category Navigation, Search, Recently Used, Popular Tools. The ecosystem should naturally expand user engagement.
   360→
   361→*Implements:* LOCK-08 (SEO — internal linking). *Operationalized by:* `11_ProductConstitution` §PC-09, `12_ToolManifestSpecification` §Related Tools, `18_SEOSpecification` §Internal Linking.
   362→
   363→### PC-10 — Product Scalability
   364→
   365→Every new tool should require minimal engineering effort. The Tool Manifest MUST contain enough metadata to automatically generate: Navigation, SEO, Sitemap, Categories, Search Index, Admin Inventory, Analytics Configuration. Metadata-first development is mandatory.
   366→
   367→*Implements:* LOCK-05 (Plugin-Ready Architecture), EC-04 (Tool Template Standard). *Operationalized by:* `11_ProductConstitution` §PC-10, `12_ToolManifestSpecification`, `05_ProjectStructure` §Tool Registry Pattern.

   371→## 6. Data & Growth Architecture (Mandatory Growth Rules)
   372→
   373→The Architectural Locks in §3 define **what the platform is**. The Engineering Constitution in §4 defines **how engineering is done**. The Product Constitution in §5 defines **how every tool behaves as a product**. The Data & Growth Architecture defines **how the platform grows sustainably** — the binding rules that ensure 1,000+ tools generate compounding analytics, SEO authority, search discoverability, and operational insight rather than fragmented data silos.
   374→
   375→**Data & growth decisions are architectural decisions.** A feature that violates the DGA cannot ship to production. These ten articles are binding on every system that touches data persistence, analytics, SEO, search, content, feature flags, audit logs, API contracts, or growth metrics. The full text is expanded in `16_EventSchemaSpecification`, `17_AnalyticsArchitecture`, `18_SearchArchitecture`, and other operationalizing documents; this section establishes them as charter-level commitments.
   376→
   377→### DGA-01 — Database as a Product Service
   378→
   379→The database is not merely persistence. It is a product capability. Only information that creates long-term user value should be persisted. Temporary processing results should remain browser-local whenever possible (LOCK-02).
   380→
   381→*Implements:* LOCK-02 (Browser-First), LOCK-06 (Database Optional). *Operationalized by:* `19_DatabaseDesign`, `02_SAD` AD-05.
   382→
   383→### DGA-02 — Event-Driven Analytics
   384→
   385→Every important action MUST produce a standardized analytics event. Analytics MUST be vendor-neutral. The event model supports Google Analytics, PostHog, Plausible, and self-hosted analytics without changing business logic. Analytics providers are adapters; the event schema is canonical.
   386→
   387→*Implements:* PC-07 (Analytics Standard), LOCK-04 (Modular — adapter pattern). *Operationalized by:* `16_EventSchemaSpecification`, `17_AnalyticsArchitecture`.
   388→
   389→### DGA-03 — SEO Metadata as Structured Data
   390→
   391→SEO MUST originate from structured metadata. The Tool Manifest is the canonical source for: Title, Description, Keywords, Canonical URL, Open Graph, Twitter Card, JSON-LD, FAQ, Breadcrumb, Related Tools, Search Intent. No SEO values are hardcoded inside pages.
   392→
   393→*Implements:* LOCK-08 (SEO Constitution), PC-10 (Product Scalability — manifest generates SEO). *Operationalized by:* `12_ToolManifestSpecification` §SEO, `21_SEOSpecification`.
   394→
   395→### DGA-04 — Search Architecture
   396→
   397→Search MUST eventually support: Instant Search, Category Search, Related Tools, Popular Tools, Recently Used, Synonyms, Fuzzy Matching, Tags. Search indexes are generated from the Tool Manifest.
   398→
   399→*Implements:* PC-09 (Feature Discoverability), LOCK-05 (Plugin-Ready — manifest-driven index). *Operationalized by:* `18_SearchArchitecture`.
   400→
   401→### DGA-05 — Content Architecture
   402→
   403→Documentation, Help, Blog, Tutorials, and Tool Pages are separate content types. Future CMS integration MUST NOT affect the Tool Engine. Content remains isolated from tool execution.
   404→
   405→*Implements:* LOCK-04 (Modular — content context is separate), `03_DDD` AD-01 (Content Context). *Operationalized by:* `19_DatabaseDesign` §Content Context, `24_AdminSpecification` §Content Modules.
   406→
   407→### DGA-06 — Feature Flags
   408→
   409→Every major feature MUST be deployable independently. Support: Beta Features, A/B Experiments, Regional Rollouts, Gradual Rollouts, Internal Testing. Feature Flags are a first-class platform capability.
   410→
   411→*Implements:* LOCK-11 (Admin Philosophy — feature flags in admin), LOCK-12 (Feature Lifecycle — flags gate lifecycle). *Operationalized by:* `24_AdminSpecification` §Feature Flags, `03_DDD` §Platform Ops Context.
   412→
   413→### DGA-07 — Auditability
   414→
   415→Every administrative action MUST be auditable. Examples: Tool Published, Tool Deprecated, SEO Updated, User Role Changed, Settings Modified. Audit logs are immutable.
   416→
   417→*Implements:* LOCK-11 (Admin Philosophy — audit trail), EC-08 (Security by Default). *Operationalized by:* `23_RBAC`, `24_AdminSpecification` §Audit Trail, `19_DatabaseDesign` §Audit Entries.
   418→
   419→### DGA-08 — API Evolution
   420→
   421→Public APIs MUST be versioned. Deprecation MUST follow a documented policy. Backward compatibility is preferred. Breaking changes require ADR entries.
   422→
   423→*Implements:* EC-12 (Enterprise Readiness — stable API contracts), LOCK-05 (Plugin-Ready — stable contracts for plugins). *Operationalized by:* `20_APIConvention`.
   424→

   439→## 7. Platform Operations Constitution (Mandatory Operations Rules)
   440→
   441→The Architectural Locks in §3 define **what the platform is**. The Engineering Constitution in §4 defines **how engineering is done**. The Product Constitution in §5 defines **how every tool behaves as a product**. The Data & Growth Architecture in §6 defines **how the platform grows sustainably**. The Platform Operations Constitution defines **how the platform is deployed, monitored, maintained, secured, and operated** — the binding rules that ensure the platform remains reliable, observable, and operable at scale.
   442→
   443→**Operational decisions are architectural decisions.** A deployment that violates POC-01 through POC-10 cannot proceed to production. These ten articles are binding on every deployment, monitoring configuration, backup procedure, security control, incident response, and cost consideration. The full text is expanded in `24_PlatformOperationsConstitution`, `25_DeploymentArchitecture`, `26_ObservabilitySpecification`, `27_BackupAndRecovery`, and `28_ReleaseManagement`; this section establishes them as charter-level commitments.
   444→
   445→### POC-01 — Reliability First
   446→
   447→The platform MUST be designed to continue operating despite partial failures. Examples: Database unavailable, Analytics unavailable, AI provider unavailable, Storage unavailable, Third-party API unavailable. Graceful degradation is mandatory.
   448→
   449→*Implements:* LOCK-06 (Database Optional), EC-05 (Progressive Enhancement). *Operationalized by:* `25_DeploymentArchitecture`, `26_ObservabilitySpecification`, `02_SAD` §Graceful Degradation.
   450→
   451→### POC-02 — Observability by Default
   452→
   453→Every important subsystem MUST expose: Logs, Metrics, Health checks, Error reports, Performance measurements. Monitoring MUST be built into the architecture rather than added later.
   454→
   455→*Implements:* DGA-07 (Auditability — logging), EC-07 (Performance Budget — measurement). *Operationalized by:* `26_ObservabilitySpecification`.
   456→
   457→### POC-03 — Release Strategy
   458→
   459→Adopt a predictable release workflow. Recommended environments: Local → Development → Preview → Production. Every production deployment MUST be reproducible.
   460→
   461→*Implements:* EC-12 (Enterprise Readiness), DGA-06 (Feature Flags — independent deployment). *Operationalized by:* `28_ReleaseManagement`, `25_DeploymentArchitecture`.
   462→
   463→### POC-04 — Rollback Strategy
   464→
   465→Every deployment MUST support rollback. Rollback procedures MUST be documented. Database migrations MUST follow forward-compatible patterns whenever possible.
   466→
   467→*Implements:* EC-12, DGA-08 (API Evolution — backward compatibility). *Operationalized by:* `28_ReleaseManagement` §Rollback, `19_DatabaseDesign` §Migrations.
   468→
   469→### POC-05 — Backup & Recovery
   470→
   471→Critical data requires documented backup and recovery procedures. Recovery objectives MUST define: Recovery Point Objective (RPO), Recovery Time Objective (RTO).
   472→
   473→*Implements:* EC-08 (Security by Default), DGA-01 (Database as Product Service). *Operationalized by:* `27_BackupAndRecovery`.
   474→
   475→### POC-06 — Operational Security
   476→
   477→Operational access MUST follow least privilege. Examples: Environment variables, Service accounts, Deployment tokens, Database credentials, Admin access. Every sensitive operation MUST be auditable.
   478→
   479→*Implements:* EC-08, DGA-07 (Auditability), LOCK-11 (Admin Philosophy). *Operationalized by:* `23_RBAC`, `25_DeploymentArchitecture` §Secrets Management.
   480→
   481→### POC-07 — Incident Management
   482→
   483→Operational incidents MUST follow a standard lifecycle: Detected → Acknowledged → Investigating → Mitigated → Resolved → Postmortem. Every significant incident MUST produce documentation and, where appropriate, a new ADR.
   484→
   485→*Implements:* DGA-07 (Auditability), EC-01 (Documentation First — postmortems). *Operationalized by:* `28_ReleaseManagement` §Incident Management.
   486→
   487→### POC-08 — Monitoring Standards
   488→
   489→Define minimum monitoring requirements: Uptime, Response Time, Error Rate, Build Status, Deployment Status, API Health, Database Health. These metrics MUST become dashboard-ready.
   490→
   491→*Implements:* POC-02 (Observability by Default). *Operationalized by:* `26_ObservabilitySpecification` §Monitoring Standards.
   492→
   493→### POC-09 — Cost Awareness
   494→
   495→Infrastructure MUST be monitored for cost efficiency. Architecture decisions MUST consider: Compute usage, Storage usage, Database usage, Bandwidth, Build minutes. Optimize for sustainable growth.
   496→
   497→*Implements:* EC-12 (Enterprise Readiness — sustainable cost scaling). *Operationalized by:* `26_ObservabilitySpecification` §Cost Monitoring.
   498→
   499→### POC-10 — Future Enterprise Operations
   500→
   501→Although the project starts on free infrastructure, the operational model MUST support future migration to enterprise environments without architectural redesign.
   502→
   503→*Implements:* EC-12, DGA-10 (Future Marketplace Readiness — enterprise operations). *Operationalized by:* `25_DeploymentArchitecture` §Enterprise Migration, `27_BackupAndRecovery` §Enterprise DR.

   645→## 11. Best Practices
   646→
   647→### 11.1 Decision-Making
   648→- **Always prefer long-term scalability over short-term convenience.** A 2-hour shortcut that costs 20 hours of refactoring later is a net loss.
   649→- **Never introduce a new dependency without justification.** Every dependency is a future maintenance liability. Justify in the PR description and link to the docs.
   650→- **Never change architecture without explaining the impact.** Architectural changes require a written impact analysis in the PR.
   651→
   652→### 11.2 Documentation
   653→- **Docs are the source of truth.** If a behavior is not in the docs, it does not exist.
   654→- **Update docs in the same PR as the code.** Code PRs without doc updates are blocked.
   655→- **Cross-references must be live.** Broken links are P1 bugs.
   656→
   657→### 11.3 Tool Development
   658→- **Isolate every tool.** A bug in one tool must never affect another.
   659→- **Reuse the workflow pattern.** New tools should compose existing workflows, not invent new ones.
   660→- **Make tools discoverable.** Every tool links to related tools; internal linking is mandatory.
   661→
   662→### 11.4 Performance
   663→- **Lazy-load every tool.** Tool code is code-split; only the tool being used loads.
   664→- **Cache aggressively.** Static assets immutable, HTML revalidated, tool outputs cached in IndexedDB when sensible.
   665→- **Measure before optimizing.** No premature optimization without a profile.
