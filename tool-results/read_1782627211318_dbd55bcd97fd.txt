    11→    11→## 1. Purpose
    12→    12→
    13→    13→This Tool Manifest Specification defines the **canonical schema** that every tool in [PROJECT_NAME] must implement. The ToolManifest is the single source of truth for a tool's identity, behavior, contract, and metadata. It is the aggregate root of the Tools Context (`03_DDD`), the contract for the Tool Registry (`05_ProjectStructure` AD-04), and the foundation for navigation, SEO, sitemap, search, admin inventory, and analytics configuration (PC-10).
    14→    14→
    15→    15→The manifest is the most important schema in the platform. Every system that needs to know about a tool — routing, navigation, SEO, admin, analytics, search, future plugin marketplace — derives its data from the manifest. If the manifest is wrong, every downstream system is wrong. If the manifest is incomplete, downstream systems produce incomplete output. This document specifies the schema in enough detail that a tool author can write a correct manifest without consulting any other reference.
    16→    16→
    17→    17→The manifest is also the **plugin contract** for the future Phase 4 marketplace. Third-party plugins will produce manifests following the same schema (with additional signing/sandboxing fields). Designing the schema comprehensively now ensures the plugin marketplace requires no schema changes later.

    39→    39→## 3. Architectural Decisions
    40→    40→
    41→    41→### AD-01 — Manifest is the Aggregate Root
    42→    42→
    43→    43→**Context.** `03_DDD` AD-04 designates ToolManifest as the aggregate root of the Tools Context. This means the manifest is the single source of truth for a tool's identity and behavior. All other tool-related data (executions, history entries) reference the manifest but don't redefine it.
    44→    44→
    45→    45→**Decision.** The ToolManifest is the aggregate root. It is a self-contained, serializable description of a tool. Any code producing a valid ToolManifest can be registered as a tool — first-party or third-party.
    46→    46→
    47→    47→**Implements:** LOCK-05 (Plugin-Ready), `03_DDD` AD-04, PC-10 (Product Scalability).
    48→    48→
    49→    49→### AD-02 — Manifest is Build-Time Consumed
    50→    50→
    51→    51→**Context.** Runtime manifest parsing adds latency and complicates Edge deployment. Build-time codegen produces static typed files that Edge can serve with zero runtime cost.
    52→    52→
    53→    53→**Decision.** The Tool Manifest is consumed at build time by `scripts/generate-registry.ts` (`05_ProjectStructure` AD-04). The script imports each `manifest.ts`, validates it against the Zod schema, and emits generated files (`registry.ts`, `navigation.ts`, `sitemap.ts`, `seo-meta.ts`, `admin-inventory.ts`, `types.ts`).
    54→    54→
    55→    55→**Implements:** LOCK-05, PC-10, EC-07 (Performance Budget — no runtime cost).
    56→    56→
    57→    57→### AD-03 — Manifest is Zod-Validated
    58→    58→
    59→    59→**Context.** A malformed manifest produces broken downstream artifacts. Without validation, errors surface at runtime, not build time.
    60→    60→
    61→    61→**Decision.** Every manifest is validated against a Zod schema at build time. Validation failures block the build. The Zod schema is the single source of truth; the TypeScript type is inferred from it.
    62→    62→
    63→    63→**Implements:** EC-08 (Security by Default — input validation), EC-02 (One Source of Truth).
    64→    64→
    65→    65→### AD-04 — Manifest is Versioned
    66→    66→
    67→    67→**Context.** As the platform evolves, the manifest schema will need to change (new fields, renamed fields, removed fields). Without versioning, schema changes break existing tools silently.
    68→    68→
    69→    69→**Decision.** Every manifest declares `manifestVersion` (e.g., `1.0.0`). Schema changes follow semver:
    70→    70→- **Patch (1.0.0 → 1.0.1):** bug fixes, no schema change.
    71→    71→- **Minor (1.0.0 → 1.1.0):** additive fields; existing manifests still valid.
    72→    72→- **Major (1.0.0 → 2.0.0):** breaking changes; existing manifests must be migrated.
    73→    73→
    74→    74→The codegen script supports multiple manifest versions and migrates older manifests to the current internal representation.
    75→    75→
    76→    76→**Implements:** EC-12 (Enterprise Readiness — stable contract evolution), LOCK-05 (Plugin-Ready — plugin compatibility).
    77→    77→
    78→    78→### AD-05 — Manifest Encodes the Product Contract
    79→    79→
    80→    80→**Context.** PC-02 mandates every tool has a product contract (purpose, inputs, outputs, validation, etc.). The contract must live somewhere; the manifest is the natural location.
    81→    81→
    82→    82→**Decision.** The ToolManifest schema includes all PC-02 product contract fields. Tools without complete contracts fail manifest validation.
    83→    83→
    84→    84→**Implements:** PC-02 (Product Contract), PC-10 (Product Scalability — contract drives codegen).
    85→    85→
    86→    86→### AD-06 — Manifest Encodes Analytics Configuration
    87→    87→
    88→    88→**Context.** PC-07 mandates every tool emits consistent analytics events. The manifest must declare which events the tool emits (beyond the standard 10 that the Tool Engine emits automatically).
    89→    89→
    90→    90→**Decision.** The manifest's `analytics` field declares the tool's analytics configuration: standard events (always emitted by Tool Engine) plus any tool-specific custom events.
    91→    91→
    92→    92→**Implements:** PC-07 (Analytics Standard), PC-10 (Product Scalability — analytics config derived from manifest).
    93→    93→
    94→    94→### AD-07 — Manifest Encodes SEO Metadata
    95→    95→
    96→    96→**Context.** LOCK-08 mandates every tool page has complete SEO metadata. The manifest must encode this metadata for codegen to consume.
    97→    97→
    98→    98→**Decision.** The manifest's `seo` field encodes: title, description, keywords, canonical URL, Open Graph, Twitter Card, JSON-LD structured data, FAQ, breadcrumb, search intent.
    99→    99→
   100→   100→**Implements:** LOCK-08 (SEO Constitution), PC-09 (Feature Discoverability — related tools for internal linking), PC-10 (Product Scalability — SEO derived from manifest).
   101→   101→
   102→   102→### AD-08 — Manifest Encodes Related Tools
   103→   103→
   104→   104→**Context.** PC-09 mandates every tool helps users discover additional tools. Related tools must be declared somewhere; the manifest is the natural location.
   105→   105→
   106→   106→**Decision.** The manifest's `relatedTools` field is an array of tool slugs. The codegen generates bidirectional links (if tool A lists tool B as related, tool B's page shows tool A in "Related" — unless tool B explicitly excludes A).
   107→   107→
   108→   108→**Implements:** PC-09 (Feature Discoverability), LOCK-08 (SEO — internal linking).
   109→   109→
   110→   110→## 4. Design Principles
   111→   111→
   112→   112→### P1 — Manifest is the Single Source of Truth
   113→   113→Anything not in the manifest is not part of the platform's understanding of the tool. No out-of-band configuration, no hardcoded tool lists, no manual registration.
   114→   114→
   115→   115→### P2 — Manifest is Serializable
   116→   116→The manifest can be serialized to JSON (for plugin marketplace distribution, admin API responses, etc.). No functions in the manifest — only data. Stage implementations are referenced by import, not embedded.
   117→   117→
   118→   118→### P3 — Manifest is Comprehensive
   119→   119→Every downstream system's needs are anticipated. If a new system needs manifest data, extend the schema via ADR rather than creating a parallel data source.
   120→   120→
   121→   121→### P4 — Manifest is Validated
   122→   122→Zod schema validation at build time catches errors early. No manifest reaches production with missing required fields or invalid values.
   123→   123→
   124→   124→### P5 — Manifest is Versioned
   125→   125→Schema evolution is governed by semver. Tools can be migrated gradually. Breaking changes require migration tooling.
   126→   126→
   127→   127→### P6 — Manifest is Plugin-Ready
   128→   128→The schema is public and stable enough that third-party plugins can produce valid manifests. Phase 4 plugin marketplace extends the schema with signing/sandboxing fields, not replaces it.

   134→   134→```typescript
   135→   135→// packages/tool-engine/src/types.ts
   136→   136→
   137→   137→import type { ZodSchema } from 'zod';
   138→   138→import type { ComponentType } from 'react';
   139→   139→
   140→   140→/**
   141→   141→ * The canonical ToolManifest schema.
   142→   142→ * Every tool MUST produce a manifest conforming to this type.
   143→   143→ *
   144→   144→ * @see /docs/12_ToolManifestSpecification.md
   145→   145→ */
   146→   146→export interface ToolManifest {
   147→   147→  // ─── Identity ───────────────────────────────────────────────
   148→   148→  manifestVersion: string;            // semver, e.g., "1.0.0"
   149→   149→  slug: string;                       // URL-safe, unique within category
   150→   150→  category: ToolCategory;             // 'image' | 'pdf' | 'developer' | ...
   151→   151→  title: string;                      // "Image Resizer"
   152→   152→  description: string;                // One-sentence description for SEO
   153→   153→  lifecycle: FeatureLifecycle;        // LOCK-12: 'concept' | 'planned' | ... | 'archived'
   154→   154→  version: string;                    // Tool's own version, semver
   155→   155→
   156→   156→  // ─── Product Contract (PC-02) ──────────────────────────────
   157→   157→  purpose: string;                    // One-sentence: what the tool does
   158→   158→  userProblem: string;                // The problem the user is solving
   159→   159→  inputSchema: ZodSchema;             // Zod schema for input validation
   160→   160→  outputSchema: ZodSchema;            // Zod schema for output
   161→   161→  validationRules: ValidationRule[];  // Specific constraints (size, format, etc.)
   162→   162→  successCriteria: string;            // What constitutes success
   163→   163→  failureStates: FailureState[];      // Enumerated failure modes
   164→   164→  emptyStates: EmptyState[];          // UI states when no input
   165→   165→  loadingStates: LoadingState[];      // UI states during processing
   166→   166→
   167→   167→  // ─── Execution (LOCK-02, LOCK-03) ──────────────────────────
   168→   168→  execution: 'browser' | 'server';    // LOCK-02 classification
   169→   169→  stages: ToolStages;                 // Stage implementations (imported, not serialized)
   170→   170→
   171→   171→  // ─── SEO (LOCK-08) ─────────────────────────────────────────
   172→   172→  seo: ToolSEOConfig;
   173→   173→
   174→   174→  // ─── Discoverability (PC-09) ───────────────────────────────
   175→   175→  relatedTools: string[];             // Slugs of related tools (min 3)
   176→   176→  suggestedWorkflows?: SuggestedWorkflow[];  // Optional, Phase 1+
   177→   177→
   178→   178→  // ─── Analytics (PC-07) ─────────────────────────────────────
   179→   179→  analytics: ToolAnalyticsConfig;
   180→   180→
   181→   181→  // ─── Limits & Constraints ──────────────────────────────────
   182→   182→  limits: ToolLimits;
   183→   183→
   184→   184→  // ─── i18n (Phase 2+; English-only in Phase 1) ──────────────
   185→   185→  i18n?: ToolI18nConfig;
   186→   186→
   187→   187→  // ─── Plugin Extensions (Phase 4) ───────────────────────────
   188→   188→  plugin?: PluginExtension;           // Only for third-party plugins
   189→   189→}

   196→   196→  | 'developer'
   197→   197→  | 'text'
   198→   198→  | 'converters'
   199→   199→  | 'seo'

   216→   216→  | 'informational'                    // User wants to learn
   217→   217→  | 'transactional'                    // User wants to do something

   219→   219→

   226→   226→

   231→   231→    what: string;                      // What happened (plain language)

   235→   235→}

   264→   264→  description: string;                 // Meta description (≤160 chars)
   265→   265→  keywords: string[];                  // Target keywords
   266→   266→  canonicalUrl: string;                // Canonical URL
   267→   267→  openGraph: {
   268→   268→    title: string;
   269→   269→    description: string;
   270→   270→    image: string;                     // URL to OG image
   271→   271→    type: 'website';
   272→   272→  };
   273→   273→  twitterCard: {
   274→   274→    card: 'summary_large_image';
   275→   275→    title: string;
   276→   276→    description: string;
   277→   277→    image: string;
   278→   278→  };
   279→   279→  structuredData: {
   280→   280→    '@type': 'SoftwareApplication';
   281→   281→    name: string;
   282→   282→    applicationCategory: string;

   288→   288→      ratingCount: number;
   289→   289→    };
   290→   290→  };

   349→   349→### 5.2 Zod Validation Schema
   350→   350→
   351→   351→```typescript
   352→   352→// packages/tool-engine/src/manifest-schema.ts
   353→   353→
   354→   354→import { z } from 'zod';
   355→   355→
   356→   356→export const toolManifestSchema = z.object({
   357→   357→  // Identity
   358→   358→  manifestVersion: z.string().regex(/^\d+\.\d+\.\d+$/),
   359→   359→  slug: z.string().regex(/^[a-z0-9-]+$/).min(3).max(50),
   360→   360→  category: z.enum([
   361→   361→    'image', 'pdf', 'developer', 'text',
   362→   362→    'converters', 'seo', 'calculators', 'utility', 'ai',
   363→   363→  ]),
   364→   364→  title: z.string().min(3).max(60),
   365→   365→  description: z.string().min(50).max(160),
   366→   366→  lifecycle: z.enum([
   367→   367→    'concept', 'planned', 'design', 'development',
   368→   368→    'testing', 'beta', 'stable', 'deprecated', 'archived',
   369→   369→  ]),
   370→   370→  version: z.string().regex(/^\d+\.\d+\.\d+$/),
   371→   371→
   372→   372→  // Product Contract
   373→   373→  purpose: z.string().min(10).max(120),
   374→   374→  userProblem: z.string().min(20).max(300),
   375→   375→  inputSchema: z.any(),  // ZodSchema instance, validated at runtime
   376→   376→  outputSchema: z.any(),
   377→   377→  validationRules: z.array(z.object({
   378→   378→    field: z.string(),
   379→   379→    type: z.enum(['required', 'maxSize', 'minSize', 'format', 'custom']),
   380→   380→    value: z.union([z.string(), z.number()]).optional(),
   381→   381→    message: z.string(),
   382→   382→  })).min(1),
   383→   383→  successCriteria: z.string().min(10).max(200),
   384→   384→  failureStates: z.array(z.object({
   385→   385→    kind: z.enum(['validation', 'processing', 'quota_exceeded', 'auth_required', 'server_unavailable']),
   386→   386→    cause: z.string(),
   387→   387→    userMessage: z.object({
   388→   388→      what: z.string(),
   389→   389→      why: z.string().optional(),
   390→   390→      howToFix: z.string(),
   391→   391→    }),
   392→   392→  })).min(1),
   393→   393→  emptyStates: z.array(z.object({
   394→   394→    scenario: z.string(),
   395→   395→    title: z.string(),
   396→   396→    description: z.string(),
   397→   397→    cta: z.object({
   398→   398→      label: z.string(),
   399→   399→      action: z.string(),
   400→   400→    }).optional(),
   401→   401→  })).min(1),
   402→   402→  loadingStates: z.array(z.object({
   403→   403→    scenario: z.string(),
   404→   404→    title: z.string(),
   405→   405→    description: z.string(),
   406→   406→    estimatedDuration: z.number().optional(),
   407→   407→  })).min(1),
   408→   408→
   409→   409→  // Execution
   410→   410→  execution: z.enum(['browser', 'server']),
   411→   411→  stages: z.object({
   412→   412→    input: z.any(),
   413→   413→    validation: z.any(),
   414→   414→    processing: z.any(),
   415→   415→    preview: z.any(),
   416→   416→    download: z.any(),
   417→   417→    history: z.any().optional(),
   418→   418→    share: z.any().optional(),
   419→   419→  }),
   420→   420→
   421→   421→  // SEO
   422→   422→  seo: z.object({
   423→   423→    searchIntent: z.enum(['informational', 'transactional', 'navigational']),
   424→   424→    title: z.string().min(10).max(60),
   425→   425→    description: z.string().min(50).max(160),
   426→   426→    keywords: z.array(z.string()).min(3).max(10),
   427→   427→    canonicalUrl: z.string().url(),
   428→   428→    openGraph: z.object({
   429→   429→      title: z.string(),
   430→   430→      description: z.string(),
   431→   431→      image: z.string().url(),
   432→   432→      type: z.literal('website'),
   433→   433→    }),
   434→   434→    twitterCard: z.object({
   435→   435→      card: z.literal('summary_large_image'),
   436→   436→      title: z.string(),
   437→   437→      description: z.string(),
   438→   438→      image: z.string().url(),
   439→   439→    }),

   801→   801→### 8.1 `src/generated/registry.ts`
   802→   802→
   803→   803→```typescript
   804→   804→// AUTO-GENERATED. Do not edit manually.
   805→   805→import type { ToolManifest } from '@packages/tool-engine';
   806→   806→import imageResize from '@/tools/image/image-resize/manifest';
   807→   807→import imageCompress from '@/tools/image/image-compress/manifest';
   808→   808→// ... all tool manifests
   809→   809→
   810→   810→export const allManifests: ToolManifest[] = [imageResize, imageCompress, /* ... */];
   811→   811→
   812→   812→export const registry = {
   813→   813→  bySlug: (slug: string) => allManifests.find(m => m.slug === slug),
   814→   814→  byCategory: (category: string) => allManifests.filter(m => m.category === category),
   815→   815→  relatedTo: (slug: string) => {
   816→   816→    const tool = allManifests.find(m => m.slug === slug);
   817→   817→    if (!tool) return [];
   818→   818→    return tool.relatedTools
   819→   819→      .map(s => allManifests.find(m => m.slug === s))
   820→   820→      .filter(Boolean);
   821→   821→  },
   822→   822→  all: () => allManifests,
   823→   823→};
   824→   824→```

   943→   943→## 9. Standards
   944→   944→
   945→   945→### 9.1 Manifest Authoring Standards
   946→   946→- Every manifest follows the schema in §5.
   947→   947→- Every manifest is validated against the Zod schema at build time.
   948→   948→- Every manifest is committed to the repo (not generated at runtime).
   949→   949→- Every manifest declares `manifestVersion` for compatibility tracking.
   950→   950→
   951→   951→### 9.2 Manifest Validation Standards
   952→   952→- Build fails on any validation error.
   953→   953→- CI runs `scripts/verify-registry.ts` to verify generated files match manifests.
   954→   954→- Pre-commit hook runs manifest validation.
   955→   955→
   956→   956→### 9.3 Manifest Versioning Standards
   957→   957→- `manifestVersion` follows semver.
   958→   958→- Schema changes require corresponding `manifestVersion` bump.
   959→   959→- Migration tooling provided for major version bumps.
   960→   960→
   961→   961→### 9.4 Manifest Immutability Standards
   962→   962→- Once a manifest is published (tool in `stable` lifecycle), its `slug` MUST NOT change.
   963→   963→- `category` changes require migration (URL redirects).
   964→   964→- Breaking schema changes require migration path for existing tools.
   965→   965→
   966→   966→## 10. Best Practices
   967→   967→
   968→   968→### 10.1 When Writing a Manifest
   969→   969→1. Start from the tool template (`scripts/tool-template/`).
   970→   970→2. Fill identity fields first (slug, category, title).
   971→   971→3. Define the product contract (purpose, userProblem, schemas).
   972→   972→4. Implement stages; reference them in `stages`.
   973→   973→5. Write SEO metadata (consult `18_SEOSpecification`).
   974→   974→6. Identify 3+ related tools.
   975→   975→7. Configure analytics (standard events are automatic; add custom if needed).
   976→   976→8. Set limits conservatively (can be relaxed later).
   977→   977→9. Validate locally: `pnpm gen:registry`.
   978→   978→10. Verify generated files are correct.
   979→   979→
   980→   980→### 10.2 When Modifying a Manifest
   981→   981→- Bump `version` field for any change.
   982→   982→- Bump `manifestVersion` only for schema changes.
   983→   983→- Update `lifecycle` per LOCK-12.
   984→   984→- Update `seo.lastModified` implicitly via git.
   985→   985→
   986→   986→### 10.3 When Reviewing a Manifest PR
   987→   987→- Verify all required fields present.
   988→   988→- Verify Zod schemas are valid.
   989→   989→- Verify SEO metadata meets `18_SEOSpecification`.
   990→   990→- Verify `relatedTools` has ≥3 entries.
   991→   991→- Verify `failureStates` messages meet PC-08 (what/why/how).
   992→   992→- Verify `limits` are reasonable.
   993→   993→- Verify generated files are updated.
   994→   994→
