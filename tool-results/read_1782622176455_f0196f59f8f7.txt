    11тЖТ## 1. Purpose
    12тЖТ
    13тЖТThis Domain-Driven Design document defines the conceptual model of [PROJECT_NAME]. It identifies the **bounded contexts** that partition the system, the **ubiquitous language** that ensures engineers and domain experts mean the same thing by the same word, the **aggregates** that enforce consistency boundaries, and the **context map** that defines how contexts relate to one another.
    14тЖТ
    15тЖТDDD is the conceptual counterpart to `02_SAD`. Where the SAD defines physical layers and runtime topology, DDD defines semantic boundaries тАФ what belongs together conceptually, what must stay separate, and why. Without DDD, modular architecture (LOCK-04) degrades into "folders that happen to be separate" with no enforcement of conceptual boundaries. Engineers start reaching across contexts for convenience, and within 18 months the codebase is a tangled web of cross-references that resists change.
    16тЖТ
    17тЖТThis document is the authoritative source for: what counts as a Tool, what counts as a User, what counts as Content, what counts as an Admin action, and how these concepts interact without violating each other's invariants. Every code module, every database table, every API endpoint must belong to exactly one bounded context, and that context's rules apply.

    38тЖТ## 3. Architectural Decisions
    39тЖТ
    40тЖТ### AD-01 тАФ Six Bounded Contexts
    41тЖТ
    42тЖТ**Context.** The platform serves distinct concerns: executing tools, managing users, publishing content, operating the platform (admin), processing payments, and observing behavior. Mixing these concerns in a single domain model leads to ambiguity (e.g., does "User" mean the same thing in auth, in analytics, and in billing?) and to tangled schemas (a User table with 80 columns covering auth, profile, billing, analytics).
    43тЖТ
    44тЖТ**Decision.** Partition the system into six bounded contexts:
    45тЖТ
    46тЖТ1. **Tools Context** тАФ the core domain. Tool definitions, manifests, execution lifecycle, tool categories, tool registry.
    47тЖТ2. **Identity Context** тАФ users, authentication, sessions, profiles, RBAC role assignment.
    48тЖТ3. **Content Context** тАФ articles, guides, SEO landing pages, media library, editorial workflow.
    49тЖТ4. **Platform Operations Context** (Admin) тАФ admin actions, feature flags, experiments, audit trail, system health.
    50тЖТ5. **Billing Context** тАФ subscriptions, plans, invoices, usage limits, payment provider integration.
    51тЖТ6. **Analytics Context** тАФ tool usage events, page views, funnels, aggregated metrics.
    52тЖТ
    53тЖТ**Consequences.**
    54тЖТ- тЬЕ Each context has its own ubiquitous language; "User" in Identity means auth identity, while in Billing it means "customer", while in Analytics it means "actor".
    55тЖТ- тЬЕ Each context owns its own schema; no shared mega-table.
    56тЖТ- тЬЕ Teams can specialize by context in Phase 3+.
    57тЖТ- тЪая╕П Cross-context features require explicit integration (domain events, published interfaces).
    58тЖТ
    59тЖТ**Implements:** LOCK-04 (Modular Architecture at the conceptual level), EC-02 (One Source of Truth тАФ each concept lives in exactly one context).
    60тЖТ
    61тЖТ### AD-02 тАФ Tools Context is the Core Domain
    62тЖТ
    63тЖТ**Context.** The platform's identity (LOCK-01) is "browser-first productivity ecosystem." The Tools Context is what makes this true тАФ without tools, there is no ecosystem. Other contexts exist to support tools (Identity: who can use them; Content: SEO that funnels to them; Billing: premium tool features; Analytics: how they're used; Platform Ops: operating them).
    64тЖТ
    65тЖТ**Decision.** Tools Context is designated the **core domain**. It receives the most architectural attention, the strictest invariants, and the most investment in design. All other contexts are **supporting** or **generic** subdomains.
    66тЖТ
    67тЖТ| Context | Domain Type | Investment Level |
    68тЖТ|---------|------------|------------------|
    69тЖТ| Tools | Core | High тАФ competitive advantage lives here |
    70тЖТ| Identity | Generic | Low тАФ use Supabase Auth; minimal customization |
    71тЖТ| Content | Supporting | Medium тАФ SEO differentiator but well-understood patterns |
    72тЖТ| Platform Ops | Supporting | Medium тАФ admin is operationally critical |
    73тЖТ| Billing | Generic | Low тАФ use Stripe-standard patterns |
    74тЖТ| Analytics | Generic | Low тАФ use standard event-tracking patterns |
    75тЖТ
    76тЖТ**Consequences.**
    77тЖТ- тЬЕ Architectural effort concentrated where it matters.
    78тЖТ- тЬЕ Generic contexts can use off-the-shelf solutions (Supabase Auth, Stripe, Posthog).
    79тЖТ- тЪая╕П Core domain must be hand-built; no shortcuts.
    80тЖТ
    81тЖТ**Implements:** LOCK-01 (Platform Identity), LOCK-03 (Tool Engine).
    82тЖТ
    83тЖТ### AD-03 тАФ Database-Optional Requires Stateless Tools Context
    84тЖТ
    85тЖТ**Context.** LOCK-06 mandates that core tools function without the database. The Tools Context must therefore be inherently stateless тАФ its aggregates must not require persistence to function.
    86тЖТ
    87тЖТ**Decision.** The Tools Context's primary aggregate, **Tool**, is a stateless concept. A Tool is defined by its manifest (code) and its inputs (provided by user). Execution produces outputs that are ephemeral unless explicitly persisted by the Identity Context (history) or Billing Context (cloud storage).
    88тЖТ
    89тЖТ```
    90тЖТTools Context (stateless)
    91тЖТтФЬтФАтФА Tool aggregate (manifest + lifecycle stages)
    92тЖТтФЬтФАтФА ToolCategory (taxonomy)
    93тЖТтФЬтФАтФА ToolExecution (ephemeral; not persisted)
    94тЖТтФФтФАтФА ToolRegistry (build-time; not runtime DB)
    95тЖТ
    96тЖТIdentity Context (stateful, DB-backed)
    97тЖТтФФтФАтФА HistoryEntry (references Tool by slug, persists outputs)
    98тЖТ```
    99тЖТ
   100тЖТ**Consequences.**
   101тЖТ- тЬЕ Tools Context has zero DB dependency; LOCK-06 satisfied structurally.
   102тЖТ- тЬЕ Tools can be developed and tested without DB setup.
   103тЖТ- тЪая╕П History/favorites belong to Identity Context, not Tools тАФ keeps boundaries clean.
   104тЖТ
   105тЖТ**Implements:** LOCK-06 (Database Optional).
   106тЖТ
   107тЖТ### AD-04 тАФ Plugin-Ready via Manifest Aggregate
   108тЖТ
   109тЖТ**Context.** LOCK-05 mandates plugin-ready architecture. The Tools Context must expose a stable, versioned contract that third-party plugins can implement.
   110тЖТ
   111тЖТ**Decision.** The **ToolManifest** is the aggregate root of the Tools Context. It is a self-contained, serializable description of a tool. Any code that produces a valid ToolManifest can be registered as a tool тАФ first-party or third-party.
   112тЖТ
   113тЖТ```
   114тЖТToolManifest (aggregate root)
   115тЖТтФЬтФАтФА Identity: slug, category, lifecycle status
   116тЖТтФЬтФАтФА Description: title, description, FAQ, related tools
   117тЖТтФЬтФАтФА Execution: 'browser' | 'server', stage implementations
   118тЖТтФЬтФАтФА Schemas: input ZodSchema, output ZodSchema
   119тЖТтФЬтФАтФА SEO: per-tool SEO config (LOCK-08)
   120тЖТтФФтФАтФА Versioning: manifest_version, tool_version
   121тЖТ```
   122тЖТ
   123тЖТThe manifest schema is public (documented in `10_FBRD`) and versioned. Breaking changes require a manifest_version bump and a migration path for existing tools.
   124тЖТ
   125тЖТ**Consequences.**
   126тЖТ- тЬЕ Phase 4 plugin marketplace requires no architectural change тАФ just a signing/sandboxing layer on top.
   127тЖТ- тЬЕ First-party and third-party tools are structurally identical.
   128тЖТ- тЪая╕П Manifest schema evolution requires discipline; breaking changes are expensive.
   129тЖТ
   130тЖТ**Implements:** LOCK-05 (Plugin-Ready), LOCK-12 (Feature Lifecycle тАФ status lives in manifest).
   131тЖТ
   132тЖТ### AD-05 тАФ Context Integration via Domain Events
   133тЖТ
   134тЖТ**Context.** Bounded contexts must communicate (e.g., when a user completes a tool, Analytics needs to know; when a user upgrades, Billing needs to notify Identity). Direct context-to-context calls create coupling.
   135тЖТ
   136тЖТ**Decision.** Contexts integrate via **domain events**. A context publishes an event when something notable happens; other contexts subscribe. Events are typed, versioned, and asynchronous.
   137тЖТ
   138тЖТ```
   139тЖТTools Context publishes:
   140тЖТ- ToolExecutionStarted { toolSlug, userId?, inputSummary, timestamp }
   141тЖТ- ToolExecutionCompleted { toolSlug, userId?, outputSummary, durationMs }
   142тЖТ- ToolExecutionFailed { toolSlug, userId?, errorKind, timestamp }
   143тЖТ
   144тЖТIdentity Context publishes:
   145тЖТ- UserRegistered { userId, email, source }
   146тЖТ- UserLoggedIn { userId, method }
   147тЖТ- HistoryEntrySaved { userId, toolSlug, entryId }
   148тЖТ
   149тЖТBilling Context publishes:
   150тЖТ- SubscriptionStarted { userId, planId, startDate }
   151тЖТ- SubscriptionCancelled { userId, planId, endDate }
   152тЖТ- UsageLimitExceeded { userId, limitType, currentValue }
   153тЖТ
   154тЖТContent Context publishes:
   155тЖТ- ArticlePublished { articleId, slug, category }
   156тЖТ- ArticleUpdated { articleId, slug }
   157тЖТ
   158тЖТPlatform Ops Context publishes:
   159тЖТ- FeatureFlagToggled { flagKey, newValue, changedBy }
   160тЖТ- ToolStatusChanged { toolSlug, oldStatus, newStatus, changedBy }
   161тЖТ```
   162тЖТ
   163тЖТ**Consequences.**
   164тЖТ- тЬЕ Contexts remain decoupled; subscribers can be added/removed without publishers changing.
   165тЖТ- тЬЕ Async by default; no context blocks another.
   166тЖТ- тЪая╕П Eventual consistency between contexts; UI must account for this.
   167тЖТ- тЪая╕П Event schema evolution requires versioning strategy.
   168тЖТ
   169тЖТ**Implements:** LOCK-04 (modularity at integration level).

   171тЖТ## 4. Design Principles
   172тЖТ
   173тЖТ### P1 тАФ One Concept, One Context
   174тЖТEvery domain concept belongs to exactly one bounded context. "User" exists in Identity; Billing has "Customer" (a role played by a User); Analytics has "Actor" (an anonymous or identified event source). This prevents the "god table" anti-pattern. *Implements EC-02 (One Source of Truth).*
   175тЖТ
   176тЖТ### P2 тАФ Contexts Don't Share Tables
   177тЖТEach context owns its own database tables (or Supabase schema). Cross-context data access is via published API or domain event, never via direct table join. This enforces context boundaries at the data layer.
   178тЖТ
   179тЖТ### P3 тАФ Aggregates Enforce Invariants
   180тЖТEvery aggregate has invariants that the aggregate root enforces on every mutation. External code cannot bypass these invariants by directly modifying fields. This keeps the domain model honest.
   181тЖТ
   182тЖТ### P4 тАФ Ubiquitous Language is Binding
   183тЖТTerms defined in the glossary (┬з6) have one meaning in their context. Code MUST use these terms; PRs that introduce synonyms are rejected. This prevents the "synonym drift" that makes large codebases incomprehensible.
   184тЖТ
   185тЖТ### P5 тАФ Tools Context is Stateless
   186тЖТThe Tools Context MUST NOT depend on persistence to function. This is structural enforcement of LOCK-06, not a runtime check.
   187тЖТ
   188тЖТ### P6 тАФ Manifest is the Contract
   189тЖТThe ToolManifest is the only contract between a tool and the platform. Anything not in the manifest is not part of the platform's understanding of the tool.
   190тЖТ
   191тЖТ## 5. Bounded Contexts (Detailed)
   192тЖТ
   193тЖТ### 5.1 Tools Context (Core Domain)
   194тЖТ
   195тЖТ**Responsibility.** Define, register, and execute tools.
   196тЖТ
   197тЖТ**Aggregates:**
   198тЖТ- **Tool** (aggregate root) тАФ defined by ToolManifest; stateless.
   199тЖТ- **ToolCategory** тАФ taxonomy node; categories can have subcategories.
   200тЖТ- **ToolExecution** тАФ ephemeral record of one run; not persisted by Tools Context (Identity may persist a reference).
   201тЖТ
   202тЖТ**Invariants:**
   203тЖТ- A Tool MUST have a unique slug within its category.
   204тЖТ- A Tool MUST declare exactly one execution mode ('browser' | 'server').
   205тЖТ- A Tool MUST implement all required Tool Engine stages (LOCK-03).
   206тЖТ- A Tool's lifecycle status transitions follow LOCK-12 (Concept тЖТ Planned тЖТ ... тЖТ Archived).
   207тЖТ
   208тЖТ**Owns:** `tools` (registry, not runtime table), `tool_categories`.
   209тЖТ
   210тЖТ### 5.2 Identity Context
   211тЖТ
   212тЖТ**Responsibility.** Authenticate users, manage profiles, assign roles, persist user-owned data (history, favorites).
   213тЖТ
   214тЖТ**Aggregates:**
   215тЖТ- **User** (aggregate root) тАФ auth identity, profile, role assignments.
   216тЖТ- **Session** тАФ active auth session; scoped to device.
   217тЖТ- **HistoryEntry** тАФ record of a tool execution saved by a user; references Tool by slug (not FK to Tools Context).
   218тЖТ- **Favorite** тАФ bookmarked tool by user.
   219тЖТ
   220тЖТ**Invariants:**
   221тЖТ- A User has exactly one role at a time (per `17_RBAC`).
   222тЖТ- HistoryEntries are immutable once created; deleted via soft-delete.
   223тЖТ- Favorites are unique per (userId, toolSlug).
   224тЖТ
   225тЖТ**Owns:** `users`, `sessions`, `history_entries`, `favorites`.
   226тЖТ
   227тЖТ**Note:** Identity Context gracefully degrades per LOCK-06 тАФ if its DB is unavailable, Tools Context still functions; users are treated as guests.
   228тЖТ
   229тЖТ### 5.3 Content Context
   230тЖТ
   231тЖТ**Responsibility.** Author, edit, publish, and serve articles, guides, and SEO landing pages.
   232тЖТ
   233тЖТ**Aggregates:**
   234тЖТ- **Article** (aggregate root) тАФ title, body, slug, category, SEO metadata, publish status.
   235тЖТ- **MediaAsset** тАФ image, video, or document used in articles or tool pages.
   236тЖТ- **TaxonomyTerm** тАФ tag or category applied to articles for navigation/SEO.
   237тЖТ
   238тЖТ**Invariants:**
   239тЖТ- Article slugs are unique globally.
   240тЖТ- Articles cannot be published without SEO metadata (LOCK-08).
   241тЖТ- MediaAssets are referenced by URL; deletion is soft with orphan check.
   242тЖТ
   243тЖТ**Owns:** `articles`, `media_assets`, `taxonomy_terms`, `article_taxonomy`.
   244тЖТ
   245тЖТ### 5.4 Platform Operations Context (Admin)
   246тЖТ
   247тЖТ**Responsibility.** Operate the platform: feature flags, experiments, audit trail, system health, ad management.
   248тЖТ
   249тЖТ**Aggregates:**
   250тЖТ- **FeatureFlag** (aggregate root) тАФ key, enabled state, targeting rules, rollout percentage.
   251тЖТ- **Experiment** тАФ A/B test definition with variants and traffic allocation.
   252тЖТ- **AuditEntry** тАФ immutable record of an admin action.
   253тЖТ- **SystemHealthCheck** тАФ latest status of each infrastructure component.
   254тЖТ- **Advertisement** тАФ ad placement definition with targeting.
   255тЖТ
   256тЖТ**Invariants:**
   257тЖТ- AuditEntries are append-only; never edited or deleted.
   258тЖТ- FeatureFlag changes are audited.
   259тЖТ- Experiments cannot overlap on the same audience segment without explicit conflict resolution.
   260тЖТ
   261тЖТ**Owns:** `feature_flags`, `experiments`, `audit_entries`, `system_health_checks`, `advertisements`.
   262тЖТ
   263тЖТ### 5.5 Billing Context
   264тЖТ
   265тЖТ**Responsibility.** Manage subscriptions, plans, invoices, usage tracking against limits.
   266тЖТ
   267тЖТ**Aggregates:**
   268тЖТ- **Subscription** (aggregate root) тАФ userId, planId, status, current period, renewal date.
   269тЖТ- **Plan** тАФ defined tier (free, premium, enterprise) with limits and price.
   270тЖТ- **Invoice** тАФ billing record from payment provider.
   271тЖТ- **UsageRecord** тАФ monthly counter per user per limit type (tool_uses, ai_calls, storage_bytes).
   272тЖТ
   273тЖТ**Invariants:**
   274тЖТ- A User has at most one active Subscription.
   275тЖТ- UsageRecords reset monthly; rollover not supported in Phase 1.
   276тЖТ- Plan changes are pro-rated per Stripe-standard logic.
   277тЖТ
   278тЖТ**Owns:** `subscriptions`, `plans`, `invoices`, `usage_records`.
   279тЖТ
   280тЖТ**Note:** Billing Context is Phase 2+. Phase 1 has a single implicit "free" plan; no billing tables needed.
   281тЖТ
   282тЖТ### 5.6 Analytics Context
   283тЖТ
   284тЖТ**Responsibility.** Capture, aggregate, and report on user behavior and tool performance.
   285тЖТ
   286тЖТ**Aggregates:**
   287тЖТ- **Event** (aggregate root) тАФ raw event record (page_view, tool_started, tool_completed, etc.).
   288тЖТ- **DailyAggregate** тАФ pre-aggregated metric per (date, toolSlug, eventType).
   289тЖТ- **Funnel** тАФ defined sequence of events for conversion tracking.
   290тЖТ
   291тЖТ**Invariants:**
   292тЖТ- Events are immutable once ingested.
   293тЖТ- Aggregates are computed nightly from raw events; raw events retained for 90 days, aggregates indefinitely.
   294тЖТ- Personally identifiable information in events is minimized (user ID only, no email/IP).
   295тЖТ
   296тЖТ**Owns:** `events`, `daily_aggregates`, `funnels`.
   297тЖТ
   298тЖТ**Note:** Analytics Context is the consumer of domain events from all other contexts. It does not publish events; it only subscribes.

   300тЖТ## 6. Ubiquitous Language Glossary
   301тЖТ
   302тЖТTerms below have ONE meaning within their context. When a term is used in a different context, it is a different concept and SHOULD have a different name. Where the same word is unavoidable (e.g., "user"), the context qualifier makes the meaning unambiguous.
   303тЖТ
   304тЖТ### Tools Context
   305тЖТ
   306тЖТ| Term | Definition |
   307тЖТ|------|------------|
   308тЖТ| **Tool** | A discrete utility that follows the Tool Engine lifecycle (LOCK-03). Identified by slug within a category. Stateless. |
   309тЖТ| **ToolManifest** | The aggregate root describing a Tool. Self-contained, serializable. Defined in `10_FBRD`. |
   310тЖТ| **ToolCategory** | A taxonomy node grouping related Tools (e.g., "image", "pdf"). |
   311тЖТ| **ToolExecution** | One run of a Tool from Input to Download. Ephemeral; not persisted by Tools Context. |
   312тЖТ| **ToolRegistry** | The build-time-generated map of all ToolManifests. Not a runtime DB table. |
   313тЖТ| **Stage** | One step in the Tool Engine lifecycle (Input, Validation, Processing, Preview, Download, History, Share). |
   314тЖТ| **Browser Tool** | A Tool whose Processing stage runs in the user's browser (LOCK-02). |
   315тЖТ| **Server Tool** | A Tool whose Processing stage runs on the server. Requires explicit user consent. |
   316тЖТ
   317тЖТ### Identity Context
   318тЖТ
   319тЖТ| Term | Definition |
   320тЖТ|------|------------|
   321тЖТ| **User** | An authenticated identity. Has exactly one role. May be guest (null User). |
   322тЖТ| **Session** | An active authentication token scoped to a device. Expires or is revoked. |
   323тЖТ| **Role** | A label (guest, user, premium, editor, admin, super_admin) granting permissions per `17_RBAC`. |
   324тЖТ| **HistoryEntry** | A record of a ToolExecution saved by a User. References Tool by slug. |
   325тЖТ| **Favorite** | A User's bookmark of a Tool. |
   326тЖТ| **Guest** | An unauthenticated visitor. Treated as null User; tools still function (LOCK-06, LOCK-07). |
   327тЖТ
   328тЖТ### Content Context
   329тЖТ
   330тЖТ| Term | Definition |
   331тЖТ|------|------------|
   332тЖТ| **Article** | An editorial piece (guide, list, comparison) published for SEO and user education. |
   333тЖТ| **MediaAsset** | An image, video, or document used in Articles or Tool pages. Stored in Supabase Storage. |
   334тЖТ| **TaxonomyTerm** | A tag or category applied to Articles for navigation and SEO. |
   335тЖТ| **PublishStatus** | One of: draft, in_review, scheduled, published, archived. |
   336тЖТ
   337тЖТ### Platform Operations Context
   338тЖТ
   339тЖТ| Term | Definition |
   340тЖТ|------|------------|
   341тЖТ| **FeatureFlag** | A toggle controlling availability of a feature for a segment of users. |
   342тЖТ| **Experiment** | An A/B test with variants and traffic allocation. |
   343тЖТ| **AuditEntry** | An immutable record of an admin action (who, what, when, before, after). |
   344тЖТ| **SystemHealthCheck** | Latest status of an infrastructure component (up, degraded, down). |
   345тЖТ| **Advertisement** | A defined ad placement with targeting rules and active/inactive state. |
   346тЖТ| **FeatureLifecycle** | The status of a Tool or feature: Concept тЖТ Planned тЖТ Design тЖТ Development тЖТ Testing тЖТ Beta тЖТ Stable тЖТ Deprecated тЖТ Archived (LOCK-12). |
   347тЖТ
   348тЖТ### Billing Context
   349тЖТ
   350тЖТ| Term | Definition |
   351тЖТ|------|------------|
   352тЖТ| **Customer** | A User in their billing capacity. Same person, different aggregate. |
   353тЖТ| **Subscription** | A User's active plan with start/end dates and renewal terms. |
   354тЖТ| **Plan** | A defined tier (free, premium, enterprise) with limits and price. |
   355тЖТ| **Invoice** | A billing record from the payment provider (Stripe). |
   356тЖТ| **UsageRecord** | A monthly counter tracking a User's consumption against Plan limits. |
   357тЖТ
   358тЖТ### Analytics Context
   359тЖТ
   360тЖТ| Term | Definition |
   361тЖТ|------|------------|
   362тЖТ| **Actor** | The source of an event. May be a User (if authenticated) or an anonymous visitor. |
   363тЖТ| **Event** | A raw record of user behavior (page_view, tool_started, tool_completed, etc.). |
   364тЖТ| **DailyAggregate** | A pre-computed metric per (date, toolSlug, eventType). |
   365тЖТ| **Funnel** | A defined sequence of Events for conversion analysis. |
   366тЖТ| **Session** (Analytics) | A browsing session, distinct from Identity Context's Session. Identified by cookie, not auth token. |

   368тЖТ## 7. Context Map
   369тЖТ
   370тЖТThe context map defines how contexts relate. Relationships follow standard DDD patterns (Konvers, Vernon).
   371тЖТ
   372тЖТ```
   373тЖТ                      тФМтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР

   408тЖТ### Anti-Corruption Layers
   409тЖТ
   410тЖТWhere a context integrates with an external system, an Anti-Corruption Layer (ACL) translates external concepts to internal ones:
   411тЖТ
   412тЖТ- **Identity ACL:** Wraps Supabase Auth; translates Supabase user records to Identity Context's User aggregate.
   413тЖТ- **Billing ACL:** Wraps Stripe; translates Stripe customer/subscription to Billing Context aggregates.
   414тЖТ- **Analytics ACL:** Wraps external analytics provider (if used in Phase 2+); translates provider events to Analytics Context Events.
   415тЖТ
   416тЖТACLs prevent external schema changes from corrupting internal domain models.

   418тЖТ## 8. Aggregate Consistency Rules
   419тЖТ
   420тЖТ### 8.1 Transaction Boundaries
   421тЖТ- A single transaction modifies exactly ONE aggregate.
   422тЖТ- Cross-aggregate consistency is achieved via domain events, eventually.
   423тЖТ- No transaction spans multiple bounded contexts.
   424тЖТ
   425тЖТ### 8.2 Aggregate Size
   426тЖТ- Aggregates are kept small. A Tool is its manifest. A User is its profile + role. A HistoryEntry is a single record.
   427тЖТ- Large nested structures (e.g., User with all their HistoryEntries) are modeled as separate aggregates with references, not as one big aggregate.
   428тЖТ
   429тЖТ### 8.3 Reference Patterns
   430тЖТ- Aggregates reference other aggregates by ID (slug, userId, articleId), never by direct object reference.
   431тЖТ- Cross-context references use the foreign context's public identifier (e.g., HistoryEntry references `toolSlug`, not a Tools Context internal ID).
   432тЖТ
   433тЖТ## 9. Standards
   434тЖТ
   435тЖТ### 9.1 Context Ownership Standards
   436тЖТ- Every code module MUST belong to exactly one bounded context (declared in module header comment).
   437тЖТ- Every database table MUST belong to exactly one context (declared in migration header).
   438тЖТ- Cross-context imports are flagged in code review and require explicit justification.
   439тЖТ
   440тЖТ### 9.2 Domain Event Standards
   441тЖТ- Every domain event MUST be typed (TypeScript interface) and versioned (`event_version: 1`).
   442тЖТ- Events MUST be idempotent (subscribers can safely receive duplicates).
   443тЖТ- Events MUST NOT contain PII beyond userId (no emails, no file contents).
   444тЖТ- Event payloads MUST be small (<10KB); large data referenced by ID.
   445тЖТ
   446тЖТ### 9.3 Schema Ownership Standards
   447тЖТ- Each context owns a Postgres schema (e.g., `tools`, `identity`, `content`, `ops`, `billing`, `analytics`).
   448тЖТ- No cross-schema foreign keys. References are by string ID.
   449тЖТ- Migrations are scoped per context; no migration touches another context's schema.
   450тЖТ
   451тЖТ## 10. Best Practices
   452тЖТ
   453тЖТ### 10.1 When Adding a New Concept
   454тЖТ1. Identify which bounded context it belongs to. If it doesn't fit any, you may need a new context (rare; requires Chief Architect approval).
   455тЖТ2. Add the term to the ubiquitous language glossary (┬з6).
   456тЖТ3. Define the aggregate it belongs to and the invariants it must enforce.
   457тЖТ4. Document any domain events it publishes or subscribes to.
   458тЖТ
   459тЖТ### 10.2 When Two Contexts Need to Share Data
   460тЖТ- **Default:** publish a domain event from the owner; subscriber reacts.
   461тЖТ- **Synchronous read needed:** owner exposes a published read API; subscriber calls it. Never read owner's tables directly.
   462тЖТ- **Synchronous write needed:** this is a smell. Re-architect so writes are local to the owning context, triggered by event.
   463тЖТ
   464тЖТ### 10.3 When Renaming a Domain Concept
   465тЖТ- Update the glossary first.
   466тЖТ- Update all code that uses the term.
   467тЖТ- Reject PRs that introduce synonyms for the renamed concept.
   468тЖТ- Add a deprecation note in the old term's location pointing to the new term.
   469тЖТ
   470тЖТ### 10.4 When Modeling a New Tool
   471тЖТ- The tool's domain concept is always a **Tool** (an aggregate in Tools Context).
   472тЖТ- The tool's execution logic belongs in the Tool's Processing stage, not in a separate domain.
   473тЖТ- The tool does NOT own user data; history of tool use belongs to Identity Context.
   474тЖТ
   475тЖТ## 11. Future Scalability
   476тЖТ
   477тЖТ### 11.1 Adding New Bounded Contexts
   478тЖТPhase 3+ may introduce:
   479тЖТ- **Collaboration Context** тАФ shared workspaces, team folders.
   480тЖТ- **Marketplace Context** тАФ third-party plugin catalog, reviews, transactions.
   481тЖТ- **API Context** тАФ public API keys, rate limits, usage billing for external API consumers.
   482тЖТ
   483тЖТAdding a context requires: (a) Chief Architect approval, (b) DDD amendment, (c) schema namespace allocation, (d) context map update.
   484тЖТ
   485тЖТ### 11.2 Context Splitting
   486тЖТIf a context grows too large (e.g., Identity grows to include team management, organization management), split it. Split criteria: >15 aggregates, >20 tables, or team ownership boundary.
   487тЖТ
   488тЖТ### 11.3 Event Schema Evolution
   489тЖТ- Events are versioned (`event_version` field).
   490тЖТ- Subscribers handle all versions they care about; unknown versions are logged and skipped (not crashed).
   491тЖТ- Breaking changes (field removal, semantic change) require a new event name (e.g., `ToolExecutionStarted` тЖТ `ToolExecutionStartedV2`).
   492тЖТ
   493тЖТ### 11.4 Multi-Tenancy (Phase 3+)
   494тЖТ- Each context's schema gains a `tenant_id` column.
   495тЖТ- Aggregate roots enforce tenant isolation in queries.
   496тЖТ- Platform Ops Context manages tenant administration.
