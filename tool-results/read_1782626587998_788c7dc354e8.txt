    11→    11→## 1. Purpose
    12→    12→
    13→    13→This Folder Structure document defines the granular file and folder conventions for [PROJECT_NAME]. Where `05_ProjectStructure` defines the high-level layout (top-level directories, bounded contexts, layer boundaries, registry pattern), this document specifies the rules for files within those directories: naming, organization, grouping, and what belongs where.
    14→    14→
    15→    15→The reason this document exists separately from `05_ProjectStructure` is that folder structure failures happen at the file level, not the directory level. An engineer who creates `imageResize.ts` instead of `image-resize.ts`, or who puts a React component in `domain/`, or who creates `utils2.ts` because `utils.ts` got too big — these are the failures that compound into an unmaintainable codebase. Without explicit rules, the file-level structure drifts in dozens of small ways that no single reviewer catches.
    16→    16→
    17→    17→This document enforces LOCK-04 (modularity), LOCK-05 (plugin-ready architecture via consistent tool structure), LOCK-06 (database optional via structural separation), EC-03 (component reuse first), EC-04 (tool template standard), and EC-01 (documentation first — every folder has a reason recorded here).

    40→    40→## 3. Architectural Decisions
    41→    41→
    42→    42→### AD-01 — File Naming: Three Conventions Based on File Type
    43→    43→
    44→    44→**Context.** Mixing naming conventions (some files kebab-case, some PascalCase, some camelCase) creates visual inconsistency and forces engineers to remember per-file rules. A clear convention based on file type reduces cognitive load.
    45→    45→
    46→    46→**Decision.** Three naming conventions, each tied to file type:
    47→    47→
    48→    48→| File Type | Convention | Example |
    49→    49→|-----------|-----------|---------|
    50→    50→| Non-component files (utils, hooks, schemas, configs, tests) | `kebab-case.ts` | `image-utils.ts`, `use-tool-history.ts` |
    51→    51→| React component files (single component per file) | `PascalCase.tsx` | `ToolCard.tsx`, `ImageUploader.tsx` |
    52→    52→| Test files | `[name].test.ts` or `[name].test.tsx` | `image-utils.test.ts`, `ToolCard.test.tsx` |
    53→    53→| Type definition files | `kebab-case.types.ts` or `[name].types.ts` | `tool.types.ts` |
    54→    54→| Index files (barrels) | `index.ts` or `index.tsx` | always `index` |
    55→    55→| Documentation | `kebab-case.md` | `architecture-decisions.md` |
    56→    56→| Configuration | `kebab-case.config.ts` | `tailwind.config.ts`, `drizzle.config.ts` |
    57→    57→
    58→    58→**Rationale:**
    59→    59→- `kebab-case` for non-components is the JavaScript ecosystem standard (Next.js, Vercel conventions).
    60→    60→- `PascalCase` for components matches the component name (`ToolCard` is in `ToolCard.tsx`), making imports visually obvious.
    61→    61→- Test suffix `.test.ts` colocates tests with their subjects, easily greppable.
    62→    62→
    63→    63→**Implements:** EC-04 (Tool Template Standard — consistent naming).
    64→    64→
    65→    65→### AD-02 — One Component Per File
    66→    66→
    67→    67→**Context.** Files with multiple components become hard to navigate, hard to test, and create import ambiguity.
    68→    68→
    69→    69→**Decision.** Each `.tsx` file in `presentation/` or `components/` contains exactly ONE exported React component. The file name matches the component name (PascalCase). Small "private" sub-components used only by the main component may live in the same file, but only if not exported.
    70→    70→
    71→    71→**Consequences.**
    72→    72→- ✅ File name = component name = import path; no ambiguity.
    73→    73→- ✅ Tests target one component per file.
    74→    74→- ⚠️ More files; mitigated by clear folder organization.
    75→    75→
    76→    76→**Implements:** EC-09 (Testing — one component per file makes testing straightforward).
    77→    77→
    78→    78→### AD-03 — Tool Folder Template (EC-04)
    79→    79→
    80→    80→**Context.** EC-04 mandates every tool follows an identical internal structure. The exact structure must be documented so tool authors have a template to copy.
    81→    81→
    82→    82→**Decision.** Every tool folder follows this exact structure:
    83→    83→
    84→    84→```
    85→    85→src/tools/[category]/[tool-slug]/
    86→    86→├── manifest.ts              # ToolManifest — the aggregate root
    87→    87→├── index.ts                 # Re-exports manifest as default
    88→    88→├── stages/
    89→    89→│   ├── input.ts             # InputStage implementation
    90→    90→│   ├── validation.ts        # ValidationStage (Zod schema + validator)
    91→    91→│   ├── processing.ts        # ProcessingStage (browser-side or server-side)
    92→    92→│   ├── preview.tsx          # PreviewStage (React component)
    93→    93→│   ├── download.ts          # DownloadStage (packaging logic)
    94→    94→│   ├── history.ts           # HistoryStage (optional)
    95→    95→│   └── share.ts             # ShareStage (optional)
    96→    96→├── components/              # Tool-specific React components
    97→    97→│   ├── InputForm.tsx        # Configuration form (if any)
    98→    98→│   ├── ProcessingProgress.tsx  # Progress UI
    99→    99→│   └── ResultDisplay.tsx    # Result preview UI
   100→   100→├── hooks/                   # Tool-specific hooks
   101→   101→│   └── use-processing.ts    # e.g., useProcessing()
   102→   102→├── lib/                     # Tool-specific pure utilities
   103→   103→│   └── image-ops.ts         # e.g., image manipulation functions
   104→   104→├── tests/
   105→   105→│   ├── stages.test.ts       # Unit tests per stage
   106→   106→│   ├── input-form.test.tsx  # Component tests
   107→   107→│   └── e2e.test.ts          # Playwright E2E test
   108→   108→└── README.md                # Tool-specific docs (optional for simple tools)
   109→   109→```
   110→   110→
   111→   111→**Rules:**
   112→   112→- `manifest.ts` and `index.ts` are mandatory.
   113→   113→- `stages/` is mandatory with at least 5 files (input, validation, processing, preview, download).
   114→   114→- `stages/history.ts` and `stages/share.ts` are optional.
   115→   115→- `components/`, `hooks/`, `lib/` are optional; created only when needed.
   116→   116→- `tests/` is mandatory with at least `stages.test.ts`.
   117→   117→- `README.md` is mandatory for tools with non-obvious behavior; optional for trivial tools.
   118→   118→
   119→   119→**Implements:** EC-04 (Tool Template Standard), LOCK-05 (Plugin-Ready — third-party tools follow same structure).
   120→   120→
   121→   121→### AD-04 — Colocated Tests
   122→   122→
   123→   123→**Context.** Separating tests into a top-level `tests/` directory separates tests from the code they test, making it harder to keep them in sync.
   124→   124→
   125→   125→**Decision.** Tests are colocated with the code they test:
   126→   126→- Unit tests: `tests/` subdirectory within the module.
   127→   127→- Component tests: `tests/` subdirectory within the component's folder.
   128→   128→- E2E tests: project-level `tests/e2e/` directory (because they span modules).
   129→   129→- Integration tests: project-level `tests/integration/` directory.
   130→   130→
   131→   131→**Consequences.**
   132→   132→- ✅ Tests are easy to find next to the code.
   133→   133→- ✅ Deleting a module deletes its tests.
   134→   134→- ⚠️ E2E tests span modules; centralized for clarity.
   135→   135→
   136→   136→**Implements:** EC-09 (Testing Philosophy).
   137→   137→
   138→   138→### AD-05 — Layer Subdirectory Pattern Within Contexts
   139→   139→
   140→   140→**Context.** `05_ProjectStructure` AD-02 mandates four layer subdirectories per context. Within each layer, files need further organization.
   141→   141→
   142→   142→**Decision.** Standard subdirectory pattern within each layer:
   143→   143→
   144→   144→```
   145→   145→src/[context]/
   146→   146→├── presentation/
   147→   147→│   ├── components/          # Context-specific React components
   148→   148→│   ├── layouts/             # Layout components
   149→   149→│   └── pages/               # Page-level components (if not in app/)
   150→   150→├── application/
   151→   151→│   ├── actions/             # Server actions
   152→   152→│   ├── hooks/               # Application hooks
   153→   153→│   ├── services/            # Application services
   154→   154→│   └── use-cases/           # Use case orchestration
   155→   155→├── domain/
   156→   156→│   ├── entities/            # Domain entities
   157→   157→│   ├── value-objects/       # Value objects
   158→   158→│   ├── aggregates/          # Aggregate roots
   159→   159→│   ├── events/              # Domain events
   160→   160→│   └── schemas/             # Zod schemas for domain types
   161→   161→└── infrastructure/
   162→   162→    ├── repositories/        # Drizzle-based repositories
   163→   163→    ├── clients/             # External API clients
   164→   164→    └── storage/             # File storage adapters
   165→   165→```
   166→   166→
   167→   167→Subdirectories are created only when needed; an empty context may start with empty layer directories.
   168→   168→
   169→   169→**Implements:** LOCK-04 (Modular), `02_SAD` AD-01 (Layered Architecture).
   170→   170→
   171→   171→### AD-06 — Shared Code Split Rule
   172→   172→
   173→   173→**Context.** `05_ProjectStructure` AD-05 defines `/packages` and `/src/shared` as two locations for shared code. The decision rule for which goes where must be explicit.
   174→   174→
   175→   175→**Decision.** Decision rule:
   176→   176→
   177→   177→| Code Characteristic | Location |
   178→   178→|---------------------|----------|
   179→   179→| Has zero dependencies on `src/` structure | `/packages` |
   180→   180→| Could theoretically be published to npm | `/packages` |
   181→   181→| Used by multiple bounded contexts | `/packages` or `/src/shared` (see below) |
   182→   182→| Depends on `src/` structure (e.g., imports from another context's published API) | `/src/shared` |
   183→   183→| App-specific configuration (env vars, feature flags client) | `/src/shared` |
   184→   184→| Cross-context hooks (useUser, useLocale) | `/src/shared` |
   185→   185→| Pure utilities (formatting, math) | `/packages/utils` |
   186→   186→| Type definitions shared across contexts | `/packages/types` |
   187→   187→| Zod schemas shared across contexts | `/packages/types` |
   188→   188→| UI primitives (Button, Input, Card) | `/packages/ui` |
   189→   189→| Tool Engine abstraction | `/packages/tool-engine` |
   190→   190→
   191→   191→When uncertain, default to `/src/shared` — promoting to `/packages` is easy; demoting is harder.
   192→   192→
   193→   193→**Implements:** EC-02 (One Source of Truth), EC-03 (Component Reuse First).
   194→   194→
   195→   195→### AD-07 — Generated Files Convention
   196→   196→
   197→   197→**Context.** `05_ProjectStructure` AD-04 generates files via build-time codegen. Generated files must be clearly marked and never manually edited.
   198→   198→
   199→   199→**Decision.** All generated files:
   200→   200→- Live in `src/generated/` only.
   201→   201→- Start with the header comment `// AUTO-GENERATED. Do not edit manually.`
   202→   202→- Are committed to the repo (for type safety across the team).
   203→   203→- Are verified by CI (`scripts/verify-registry.ts` fails the build if generated files are out of sync with manifests).
   204→   204→
   205→   205→**Consequences.**
   206→   206→- ✅ Type safety without running codegen before lint.
   207→   207→- ✅ Diff visibility in PRs.
   208→   208→- ⚠️ Discipline required to never manually edit.
   209→   209→
   210→   210→**Implements:** LOCK-05 (Plugin-Ready via codegen), EC-01 (Documentation — generated files are documented as such).
   211→   211→
   212→   212→### AD-08 — Documentation File Placement
   213→   213→
   214→   214→**Context.** EC-01 mandates documentation first. Documentation files must be discoverable.
   215→   215→
   216→   216→**Decision.** Documentation placement rules:
   217→   217→
   218→   218→| Document Type | Location |
   219→   219→|--------------|----------|
   220→   220→| Project-level docs (architecture, conventions) | `/docs/*.md` |
   221→   221→| Tool-specific docs | `src/tools/[category]/[slug]/README.md` |
   222→   222→| Package docs | `/packages/[name]/README.md` |
   223→   223→| ADRs | `/docs/06_ArchitectureDecisionRecords.md` (consolidated) |
   224→   224→| API docs | `/docs/17_APIConvention.md` + inline JSDoc |
   225→   225→| README | Project root `README.md` (links to `/docs`) |
   226→   226→
   227→   227→**Implements:** EC-01 (Documentation First).

   229→   229→## 4. Design Principles
   230→   230→
   231→   231→### P1 — Convention Over Configuration
   232→   232→Follow the conventions in this document; don't invent new patterns. If a convention doesn't fit your case, propose an ADR rather than deviating silently.
   233→   233→
   234→   234→### P2 — Colocation
   235→   235→Code, tests, and docs for a module live together. Don't separate them by type.
   236→   236→
   237→   237→### P3 — Discoverability
   238→   238→An engineer browsing the file tree should be able to find what they need without grepping. Folder and file names must be self-explanatory.
   239→   239→
   240→   240→### P4 — Minimal Subdirectories
   241→   241→Don't create subdirectories until you have 5+ files in a folder. Premature subdirectories create empty navigation.
   242→   242→
   243→   243→### P5 — Naming Reflects Purpose
   244→   244→File and folder names reflect what they contain, not who created them or when. Avoid names like `utils-v2.ts` or `johns-stuff/`.
   245→   245→
   246→   246→### P6 — Generated Files Are Sacred
   247→   247→Never manually edit generated files. If a generated file is wrong, fix the source (manifest, codegen script) and regenerate.

   276→   276→
   277→   277→### 6.1 `/src/app/` — Next.js App Router
   278→   278→
   279→   279→```
   280→   280→src/app/
   281→   281→├── (public)/                # Public route group (no auth required)
   282→   282→│   ├── tools/[category]/[slug]/page.tsx   # Single dynamic route for all tools
   283→   283→│   ├── tools/[category]/page.tsx          # Category landing
   284→   284→│   └── tools/page.tsx                     # All-tools hub
   285→   285→├── (auth)/                  # Auth-required route group
   286→   286→│   ├── login/page.tsx
   287→   287→│   ├── register/page.tsx
   288→   288→│   ├── dashboard/page.tsx
   289→   289→│   └── settings/page.tsx
   290→   290→├── (admin)/                 # Admin route group (RBAC required)
   291→   291→│   ├── admin/layout.tsx     # Admin layout with sidebar
   292→   292→│   └── admin/page.tsx       # Admin dashboard
   293→   293→├── (marketing)/             # Marketing pages
   294→   294→│   ├── page.tsx             # Homepage
   295→   295→│   ├── about/page.tsx
   296→   296→│   ├── pricing/page.tsx
   297→   297→│   └── blog/page.tsx        # Content Context articles
   298→   298→├── api/                     # API routes (server-side tools, webhooks)
   299→   299→│   ├── tools/[slug]/route.ts   # Server-side tool endpoints
   300→   300→│   └── webhooks/               # Webhook handlers
   301→   301→├── layout.tsx               # Root layout (HTML, body, providers)
   302→   302→├── globals.css              # Global styles (Tailwind imports, CSS custom properties)
   303→   303→├── not-found.tsx            # 404 page
   304→   304→└── error.tsx                # Global error boundary
   305→   305→```
   306→   306→
   307→   307→**Rules:**
   308→   308→- Route groups `(name)` don't affect URL; used for organization and shared layouts.
   309→   309→- One `page.tsx` per route; no business logic in page files (delegate to context's application layer).
   310→   310→- `layout.tsx` files contain shared layout, navigation, and providers.

   563→   563→## 8. Standards
   564→   564→
   565→   565→### 8.1 File Naming Standards
   566→   566→- Non-component TypeScript files: `kebab-case.ts`
   567→   567→- Component files: `PascalCase.tsx`
   568→   568→- Test files: `[name].test.ts` or `[name].test.tsx`
   569→   569→- Type files: `[name].types.ts`
   570→   570→- Documentation: `kebab-case.md`
   571→   571→- Config: `kebab-case.config.ts`
   572→   572→
   573→   573→### 8.2 Folder Naming Standards
   574→   574→- All folders: `kebab-case`
   575→   575→- Special prefixes: `_shared/` (sorts first, indicates internal shared)
   576→   576→- Route groups in `app/`: `(name)` per Next.js convention
   577→   577→
   578→   578→### 8.3 File Size Standards (per `00_Project_Charter` §7.1)
   579→   579→- Soft limit: 300 lines per file
   580→   580→- Hard limit: 500 lines per file
   581→   581→- Files approaching 300 lines should be refactored into smaller modules
   582→   582→- Files exceeding 500 lines block PR merge
   583→   583→
   584→   584→### 8.4 Folder Cardinality Standards
   585→   585→- Don't create a subdirectory until parent has 5+ files
   586→   586→- Don't create more than 2 levels of nesting within a context's layer
   587→   587→- Single-file folders are acceptable (e.g., `stages/input.ts`)
   588→   588→
   589→   589→### 8.5 Index File Standards
   590→   590→- `index.ts` files are barrel files that re-export public APIs
   591→   591→- They do NOT contain logic
   592→   592→- They prevent deep import paths (`@/tools/image/image-resize` vs `@/tools/image/image-resize/manifest`)
   593→   593→
   594→   594→### 8.6 Test File Standards
   595→   595→- Every test file is named `[subject].test.ts` or `[subject].test.tsx`
   596→   596→- Test files live in `tests/` subdirectory of their subject's folder
   597→   597→- E2E tests live in project-level `tests/e2e/`
   598→   598→- Test fixtures live in `tests/fixtures/`
   599→   599→
   600→   600→## 9. Best Practices
   601→   601→
   602→   602→### 9.1 When Creating a New Tool
   603→   603→1. Run `pnpm new-tool [category] [slug]` to scaffold from template.
   604→   604→2. Edit `manifest.ts` with tool metadata.
   605→   605→3. Implement each stage in `stages/`.
   606→   606→4. Add components, hooks, lib as needed.
   607→   607→5. Write tests in `tests/`.
   608→   608→6. Run `pnpm gen:registry` to update generated files.
   609→   609→7. Verify with `pnpm test` and `pnpm lint`.
   610→   610→
   611→   611→### 9.2 When Adding a Component
   612→   612→1. Check `@packages/ui` for existing component (EC-03).
   613→   613→2. Check `src/shared/components/` for app-specific shared component.
   614→   614→3. If tool-specific, place in tool's `components/` folder.
   615→   615→4. Name file `PascalCase.tsx` matching component name.
   616→   616→5. One component per file (AD-02).
   617→   617→
   618→   618→### 9.3 When Adding a Utility Function
   619→   619→1. Check `@packages/utils` for existing utility (EC-03).
   620→   620→2. Check context's `domain/` or `application/` for context-specific utility.
   621→   621→3. If tool-specific, place in tool's `lib/` folder.
   622→   622→4. Name file `kebab-case.ts` describing the utility group (e.g., `image-ops.ts`, not `helpers.ts`).
   623→   623→
   624→   624→### 9.4 When Adding a Test
   625→   625→1. Place next to the code being tested (AD-04).
   626→   626→2. Name `[subject].test.ts`.
   627→   627→3. E2E tests go in `tests/e2e/` with descriptive name `[scenario].spec.ts`.
   628→   628→4. Use fixtures from `tests/fixtures/` for sample files.
   629→   629→
   630→   630→### 9.5 When Refactoring
   631→   631→1. Preserve file naming conventions.
   632→   632→2. Don't create temporary folders like `old/`, `deprecated/`, `v2/`.
   633→   633→3. If deleting a file, delete its tests too (colocated).
   634→   634→4. Update imports across the codebase; rely on TypeScript to catch misses.
   635→   635→
   636→   636→### 9.6 When Adding Documentation
   637→   637→1. Project-level docs go in `/docs/` with numbered filename.
   638→   638→2. Tool-specific docs go in tool's `README.md`.
   639→   639→3. Package docs go in package's `README.md`.
   640→   640→4. Inline code comments use JSDoc for public APIs.
   641→   641→
   642→   642→## 10. Future Scalability
   643→   643→
   644→   644→### 10.1 Scaling to 1,000+ Tools
   645→   645→- Tool folder template scales linearly.
   646→   646→- At >500 tools, consider adding a `tools/_index.ts` for faster IDE navigation.
   647→   647→- Category folders may need sub-categories (e.g., `image/resize/`, `image/convert/`); add via ADR if needed.
   648→   648→
   649→   649→### 10.2 Scaling Packages
   650→   650→- `/packages` can grow to 10+ packages without issue.
   651→   651→- Each package remains self-contained; no cross-package imports except via published APIs.
   652→   652→
   653→   653→### 10.3 Plugin Directory (Phase 4)
   654→   654→- Add `/plugins/` directory for third-party tools.
   655→   655→- Plugin folder structure mirrors tool folder structure (AD-03).
   656→   656→- Plugins are signed and sandboxed per future ADR.
   657→   657→
   658→   658→### 10.4 Multi-Tenancy (Phase 3+)
   659→   659→- Each context's `infrastructure/` layer gains `tenancy/` subdirectory for tenant-aware repositories.
   660→   660→- No structural changes to top-level layout.
