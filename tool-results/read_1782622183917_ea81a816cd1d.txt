    11‚Üí    11‚Üí## 1. Purpose
    12‚Üí    12‚Üí
    13‚Üí    13‚ÜíThis Domain-Driven Design document defines the conceptual model of [PROJECT_NAME]. It identifies the **bounded contexts** that partition the system, the **ubiquitous language** that ensures engineers and domain experts mean the same thing by the same word, the **aggregates** that enforce consistency boundaries, and the **context map** that defines how contexts relate to one another.
    14‚Üí    14‚Üí
    15‚Üí    15‚ÜíDDD is the conceptual counterpart to `02_SAD`. Where the SAD defines physical layers and runtime topology, DDD defines semantic boundaries ‚Äî what belongs together conceptually, what must stay separate, and why. Without DDD, modular architecture (LOCK-04) degrades into "folders that happen to be separate" with no enforcement of conceptual boundaries. Engineers start reaching across contexts for convenience, and within 18 months the codebase is a tangled web of cross-references that resists change.
    16‚Üí    16‚Üí
    17‚Üí    17‚ÜíThis document is the authoritative source for: what counts as a Tool, what counts as a User, what counts as Content, what counts as an Admin action, and how these concepts interact without violating each other's invariants. Every code module, every database table, every API endpoint must belong to exactly one bounded context, and that context's rules apply.

    38‚Üí    38‚Üí## 3. Architectural Decisions
    39‚Üí    39‚Üí
    40‚Üí    40‚Üí### AD-01 ‚Äî Six Bounded Contexts
    41‚Üí    41‚Üí
    42‚Üí    42‚Üí**Context.** The platform serves distinct concerns: executing tools, managing users, publishing content, operating the platform (admin), processing payments, and observing behavior. Mixing these concerns in a single domain model leads to ambiguity (e.g., does "User" mean the same thing in auth, in analytics, and in billing?) and to tangled schemas (a User table with 80 columns covering auth, profile, billing, analytics).
    43‚Üí    43‚Üí
    44‚Üí    44‚Üí**Decision.** Partition the system into six bounded contexts:
    45‚Üí    45‚Üí
    46‚Üí    46‚Üí1. **Tools Context** ‚Äî the core domain. Tool definitions, manifests, execution lifecycle, tool categories, tool registry.
    47‚Üí    47‚Üí2. **Identity Context** ‚Äî users, authentication, sessions, profiles, RBAC role assignment.
    48‚Üí    48‚Üí3. **Content Context** ‚Äî articles, guides, SEO landing pages, media library, editorial workflow.
    49‚Üí    49‚Üí4. **Platform Operations Context** (Admin) ‚Äî admin actions, feature flags, experiments, audit trail, system health.
    50‚Üí    50‚Üí5. **Billing Context** ‚Äî subscriptions, plans, invoices, usage limits, payment provider integration.
    51‚Üí    51‚Üí6. **Analytics Context** ‚Äî tool usage events, page views, funnels, aggregated metrics.
    52‚Üí    52‚Üí
    53‚Üí    53‚Üí**Consequences.**
    54‚Üí    54‚Üí- ‚úÖ Each context has its own ubiquitous language; "User" in Identity means auth identity, while in Billing it means "customer", while in Analytics it means "actor".
    55‚Üí    55‚Üí- ‚úÖ Each context owns its own schema; no shared mega-table.
    56‚Üí    56‚Üí- ‚úÖ Teams can specialize by context in Phase 3+.
    57‚Üí    57‚Üí- ‚öÝÔ∏è Cross-context features require explicit integration (domain events, published interfaces).
    58‚Üí    58‚Üí
    59‚Üí    59‚Üí**Implements:** LOCK-04 (Modular Architecture at the conceptual level), EC-02 (One Source of Truth ‚Äî each concept lives in exactly one context).
    60‚Üí    60‚Üí
    61‚Üí    61‚Üí### AD-02 ‚Äî Tools Context is the Core Domain
    62‚Üí    62‚Üí
    63‚Üí    63‚Üí**Context.** The platform's identity (LOCK-01) is "browser-first productivity ecosystem." The Tools Context is what makes this true ‚Äî without tools, there is no ecosystem. Other contexts exist to support tools (Identity: who can use them; Content: SEO that funnels to them; Billing: premium tool features; Analytics: how they're used; Platform Ops: operating them).
    64‚Üí    64‚Üí
    65‚Üí    65‚Üí**Decision.** Tools Context is designated the **core domain**. It receives the most architectural attention, the strictest invariants, and the most investment in design. All other contexts are **supporting** or **generic** subdomains.
    66‚Üí    66‚Üí
    67‚Üí    67‚Üí| Context | Domain Type | Investment Level |
    68‚Üí    68‚Üí|---------|------------|------------------|
    69‚Üí    69‚Üí| Tools | Core | High ‚Äî competitive advantage lives here |
    70‚Üí    70‚Üí| Identity | Generic | Low ‚Äî use Supabase Auth; minimal customization |
    71‚Üí    71‚Üí| Content | Supporting | Medium ‚Äî SEO differentiator but well-understood patterns |
    72‚Üí    72‚Üí| Platform Ops | Supporting | Medium ‚Äî admin is operationally critical |
    73‚Üí    73‚Üí| Billing | Generic | Low ‚Äî use Stripe-standard patterns |
    74‚Üí    74‚Üí| Analytics | Generic | Low ‚Äî use standard event-tracking patterns |
    75‚Üí    75‚Üí
    76‚Üí    76‚Üí**Consequences.**
    77‚Üí    77‚Üí- ‚úÖ Architectural effort concentrated where it matters.
    78‚Üí    78‚Üí- ‚úÖ Generic contexts can use off-the-shelf solutions (Supabase Auth, Stripe, Posthog).
    79‚Üí    79‚Üí- ‚öÝÔ∏è Core domain must be hand-built; no shortcuts.
    80‚Üí    80‚Üí
    81‚Üí    81‚Üí**Implements:** LOCK-01 (Platform Identity), LOCK-03 (Tool Engine).
    82‚Üí    82‚Üí
    83‚Üí    83‚Üí### AD-03 ‚Äî Database-Optional Requires Stateless Tools Context
    84‚Üí    84‚Üí
    85‚Üí    85‚Üí**Context.** LOCK-06 mandates that core tools function without the database. The Tools Context must therefore be inherently stateless ‚Äî its aggregates must not require persistence to function.
    86‚Üí    86‚Üí
    87‚Üí    87‚Üí**Decision.** The Tools Context's primary aggregate, **Tool**, is a stateless concept. A Tool is defined by its manifest (code) and its inputs (provided by user). Execution produces outputs that are ephemeral unless explicitly persisted by the Identity Context (history) or Billing Context (cloud storage).
    88‚Üí    88‚Üí
    89‚Üí    89‚Üí```
    90‚Üí    90‚ÜíTools Context (stateless)
    91‚Üí    91‚Üí‚îú‚îÄ‚îÄ Tool aggregate (manifest + lifecycle stages)
    92‚Üí    92‚Üí‚îú‚îÄ‚îÄ ToolCategory (taxonomy)
    93‚Üí    93‚Üí‚îú‚îÄ‚îÄ ToolExecution (ephemeral; not persisted)
    94‚Üí    94‚Üí‚îî‚îÄ‚îÄ ToolRegistry (build-time; not runtime DB)
    95‚Üí    95‚Üí
    96‚Üí    96‚ÜíIdentity Context (stateful, DB-backed)
    97‚Üí    97‚Üí‚îî‚îÄ‚îÄ HistoryEntry (references Tool by slug, persists outputs)
    98‚Üí    98‚Üí```
    99‚Üí    99‚Üí
   100‚Üí   100‚Üí**Consequences.**
   101‚Üí   101‚Üí- ‚úÖ Tools Context has zero DB dependency; LOCK-06 satisfied structurally.
   102‚Üí   102‚Üí- ‚úÖ Tools can be developed and tested without DB setup.
   103‚Üí   103‚Üí- ‚öÝÔ∏è History/favorites belong to Identity Context, not Tools ‚Äî keeps boundaries clean.
   104‚Üí   104‚Üí
   105‚Üí   105‚Üí**Implements:** LOCK-06 (Database Optional).
   106‚Üí   106‚Üí
   107‚Üí   107‚Üí### AD-04 ‚Äî Plugin-Ready via Manifest Aggregate
   108‚Üí   108‚Üí
   109‚Üí   109‚Üí**Context.** LOCK-05 mandates plugin-ready architecture. The Tools Context must expose a stable, versioned contract that third-party plugins can implement.
   110‚Üí   110‚Üí
   111‚Üí   111‚Üí**Decision.** The **ToolManifest** is the aggregate root of the Tools Context. It is a self-contained, serializable description of a tool. Any code that produces a valid ToolManifest can be registered as a tool ‚Äî first-party or third-party.
   112‚Üí   112‚Üí
   113‚Üí   113‚Üí```
   114‚Üí   114‚ÜíToolManifest (aggregate root)
   115‚Üí   115‚Üí‚îú‚îÄ‚îÄ Identity: slug, category, lifecycle status
   116‚Üí   116‚Üí‚îú‚îÄ‚îÄ Description: title, description, FAQ, related tools
   117‚Üí   117‚Üí‚îú‚îÄ‚îÄ Execution: 'browser' | 'server', stage implementations
   118‚Üí   118‚Üí‚îú‚îÄ‚îÄ Schemas: input ZodSchema, output ZodSchema
   119‚Üí   119‚Üí‚îú‚îÄ‚îÄ SEO: per-tool SEO config (LOCK-08)
   120‚Üí   120‚Üí‚îî‚îÄ‚îÄ Versioning: manifest_version, tool_version
   121‚Üí   121‚Üí```
   122‚Üí   122‚Üí
   123‚Üí   123‚ÜíThe manifest schema is public (documented in `10_FBRD`) and versioned. Breaking changes require a manifest_version bump and a migration path for existing tools.
   124‚Üí   124‚Üí
   125‚Üí   125‚Üí**Consequences.**
   126‚Üí   126‚Üí- ‚úÖ Phase 4 plugin marketplace requires no architectural change ‚Äî just a signing/sandboxing layer on top.
   127‚Üí   127‚Üí- ‚úÖ First-party and third-party tools are structurally identical.
   128‚Üí   128‚Üí- ‚öÝÔ∏è Manifest schema evolution requires discipline; breaking changes are expensive.
   129‚Üí   129‚Üí
   130‚Üí   130‚Üí**Implements:** LOCK-05 (Plugin-Ready), LOCK-12 (Feature Lifecycle ‚Äî status lives in manifest).
   131‚Üí   131‚Üí
   132‚Üí   132‚Üí### AD-05 ‚Äî Context Integration via Domain Events
   133‚Üí   133‚Üí
   134‚Üí   134‚Üí**Context.** Bounded contexts must communicate (e.g., when a user completes a tool, Analytics needs to know; when a user upgrades, Billing needs to notify Identity). Direct context-to-context calls create coupling.
   135‚Üí   135‚Üí
   136‚Üí   136‚Üí**Decision.** Contexts integrate via **domain events**. A context publishes an event when something notable happens; other contexts subscribe. Events are typed, versioned, and asynchronous.
   137‚Üí   137‚Üí
   138‚Üí   138‚Üí```
   139‚Üí   139‚ÜíTools Context publishes:
   140‚Üí   140‚Üí- ToolExecutionStarted { toolSlug, userId?, inputSummary, timestamp }
   141‚Üí   141‚Üí- ToolExecutionCompleted { toolSlug, userId?, outputSummary, durationMs }
   142‚Üí   142‚Üí- ToolExecutionFailed { toolSlug, userId?, errorKind, timestamp }
   143‚Üí   143‚Üí
   144‚Üí   144‚ÜíIdentity Context publishes:
   145‚Üí   145‚Üí- UserRegistered { userId, email, source }
   146‚Üí   146‚Üí- UserLoggedIn { userId, method }
   147‚Üí   147‚Üí- HistoryEntrySaved { userId, toolSlug, entryId }
   148‚Üí   148‚Üí
   149‚Üí   149‚ÜíBilling Context publishes:
   150‚Üí   150‚Üí- SubscriptionStarted { userId, planId, startDate }
   151‚Üí   151‚Üí- SubscriptionCancelled { userId, planId, endDate }
   152‚Üí   152‚Üí- UsageLimitExceeded { userId, limitType, currentValue }
   153‚Üí   153‚Üí
   154‚Üí   154‚ÜíContent Context publishes:
   155‚Üí   155‚Üí- ArticlePublished { articleId, slug, category }
   156‚Üí   156‚Üí- ArticleUpdated { articleId, slug }
   157‚Üí   157‚Üí
   158‚Üí   158‚ÜíPlatform Ops Context publishes:
   159‚Üí   159‚Üí- FeatureFlagToggled { flagKey, newValue, changedBy }
   160‚Üí   160‚Üí- ToolStatusChanged { toolSlug, oldStatus, newStatus, changedBy }
   161‚Üí   161‚Üí```
   162‚Üí   162‚Üí
   163‚Üí   163‚Üí**Consequences.**
   164‚Üí   164‚Üí- ‚úÖ Contexts remain decoupled; subscribers can be added/removed without publishers changing.
   165‚Üí   165‚Üí- ‚úÖ Async by default; no context blocks another.
   166‚Üí   166‚Üí- ‚öÝÔ∏è Eventual consistency between contexts; UI must account for this.
   167‚Üí   167‚Üí- ‚öÝÔ∏è Event schema evolution requires versioning strategy.
   168‚Üí   168‚Üí
   169‚Üí   169‚Üí**Implements:** LOCK-04 (modularity at integration level).

   171‚Üí   171‚Üí## 4. Design Principles
   172‚Üí   172‚Üí
   173‚Üí   173‚Üí### P1 ‚Äî One Concept, One Context
   174‚Üí   174‚ÜíEvery domain concept belongs to exactly one bounded context. "User" exists in Identity; Billing has "Customer" (a role played by a User); Analytics has "Actor" (an anonymous or identified event source). This prevents the "god table" anti-pattern. *Implements EC-02 (One Source of Truth).*
   175‚Üí   175‚Üí
   176‚Üí   176‚Üí### P2 ‚Äî Contexts Don't Share Tables
   177‚Üí   177‚ÜíEach context owns its own database tables (or Supabase schema). Cross-context data access is via published API or domain event, never via direct table join. This enforces context boundaries at the data layer.
   178‚Üí   178‚Üí
   179‚Üí   179‚Üí### P3 ‚Äî Aggregates Enforce Invariants
   180‚Üí   180‚ÜíEvery aggregate has invariants that the aggregate root enforces on every mutation. External code cannot bypass these invariants by directly modifying fields. This keeps the domain model honest.
   181‚Üí   181‚Üí
   182‚Üí   182‚Üí### P4 ‚Äî Ubiquitous Language is Binding
   183‚Üí   183‚ÜíTerms defined in the glossary (¬ß6) have one meaning in their context. Code MUST use these terms; PRs that introduce synonyms are rejected. This prevents the "synonym drift" that makes large codebases incomprehensible.
   184‚Üí   184‚Üí
   185‚Üí   185‚Üí### P5 ‚Äî Tools Context is Stateless
   186‚Üí   186‚ÜíThe Tools Context MUST NOT depend on persistence to function. This is structural enforcement of LOCK-06, not a runtime check.
   187‚Üí   187‚Üí
   188‚Üí   188‚Üí### P6 ‚Äî Manifest is the Contract
   189‚Üí   189‚ÜíThe ToolManifest is the only contract between a tool and the platform. Anything not in the manifest is not part of the platform's understanding of the tool.
   190‚Üí   190‚Üí
   191‚Üí   191‚Üí## 5. Bounded Contexts (Detailed)
   192‚Üí   192‚Üí
   193‚Üí   193‚Üí### 5.1 Tools Context (Core Domain)
   194‚Üí   194‚Üí
   195‚Üí   195‚Üí**Responsibility.** Define, register, and execute tools.
   196‚Üí   196‚Üí
   197‚Üí   197‚Üí**Aggregates:**
   198‚Üí   198‚Üí- **Tool** (aggregate root) ‚Äî defined by ToolManifest; stateless.
   199‚Üí   199‚Üí- **ToolCategory** ‚Äî taxonomy node; categories can have subcategories.
   200‚Üí   200‚Üí- **ToolExecution** ‚Äî ephemeral record of one run; not persisted by Tools Context (Identity may persist a reference).
   201‚Üí   201‚Üí
   202‚Üí   202‚Üí**Invariants:**
   203‚Üí   203‚Üí- A Tool MUST have a unique slug within its category.
   204‚Üí   204‚Üí- A Tool MUST declare exactly one execution mode ('browser' | 'server').
   205‚Üí   205‚Üí- A Tool MUST implement all required Tool Engine stages (LOCK-03).
   206‚Üí   206‚Üí- A Tool's lifecycle status transitions follow LOCK-12 (Concept ‚Üí Planned ‚Üí ... ‚Üí Archived).
   207‚Üí   207‚Üí
   208‚Üí   208‚Üí**Owns:** `tools` (registry, not runtime table), `tool_categories`.
   209‚Üí   209‚Üí
   210‚Üí   210‚Üí### 5.2 Identity Context
   211‚Üí   211‚Üí
   212‚Üí   212‚Üí**Responsibility.** Authenticate users, manage profiles, assign roles, persist user-owned data (history, favorites).
   213‚Üí   213‚Üí
   214‚Üí   214‚Üí**Aggregates:**
   215‚Üí   215‚Üí- **User** (aggregate root) ‚Äî auth identity, profile, role assignments.
   216‚Üí   216‚Üí- **Session** ‚Äî active auth session; scoped to device.
   217‚Üí   217‚Üí- **HistoryEntry** ‚Äî record of a tool execution saved by a user; references Tool by slug (not FK to Tools Context).
   218‚Üí   218‚Üí- **Favorite** ‚Äî bookmarked tool by user.
   219‚Üí   219‚Üí
   220‚Üí   220‚Üí**Invariants:**
   221‚Üí   221‚Üí- A User has exactly one role at a time (per `17_RBAC`).
   222‚Üí   222‚Üí- HistoryEntries are immutable once created; deleted via soft-delete.
   223‚Üí   223‚Üí- Favorites are unique per (userId, toolSlug).
   224‚Üí   224‚Üí
   225‚Üí   225‚Üí**Owns:** `users`, `sessions`, `history_entries`, `favorites`.
   226‚Üí   226‚Üí
   227‚Üí   227‚Üí**Note:** Identity Context gracefully degrades per LOCK-06 ‚Äî if its DB is unavailable, Tools Context still functions; users are treated as guests.
   228‚Üí   228‚Üí
   229‚Üí   229‚Üí### 5.3 Content Context
   230‚Üí   230‚Üí
   231‚Üí   231‚Üí**Responsibility.** Author, edit, publish, and serve articles, guides, and SEO landing pages.
   232‚Üí   232‚Üí
   233‚Üí   233‚Üí**Aggregates:**
   234‚Üí   234‚Üí- **Article** (aggregate root) ‚Äî title, body, slug, category, SEO metadata, publish status.
   235‚Üí   235‚Üí- **MediaAsset** ‚Äî image, video, or document used in articles or tool pages.
   236‚Üí   236‚Üí- **TaxonomyTerm** ‚Äî tag or category applied to articles for navigation/SEO.
   237‚Üí   237‚Üí
   238‚Üí   238‚Üí**Invariants:**
   239‚Üí   239‚Üí- Article slugs are unique globally.
   240‚Üí   240‚Üí- Articles cannot be published without SEO metadata (LOCK-08).
   241‚Üí   241‚Üí- MediaAssets are referenced by URL; deletion is soft with orphan check.
   242‚Üí   242‚Üí
   243‚Üí   243‚Üí**Owns:** `articles`, `media_assets`, `taxonomy_terms`, `article_taxonomy`.
   244‚Üí   244‚Üí
   245‚Üí   245‚Üí### 5.4 Platform Operations Context (Admin)
   246‚Üí   246‚Üí
   247‚Üí   247‚Üí**Responsibility.** Operate the platform: feature flags, experiments, audit trail, system health, ad management.
   248‚Üí   248‚Üí
   249‚Üí   249‚Üí**Aggregates:**
   250‚Üí   250‚Üí- **FeatureFlag** (aggregate root) ‚Äî key, enabled state, targeting rules, rollout percentage.
   251‚Üí   251‚Üí- **Experiment** ‚Äî A/B test definition with variants and traffic allocation.
   252‚Üí   252‚Üí- **AuditEntry** ‚Äî immutable record of an admin action.
   253‚Üí   253‚Üí- **SystemHealthCheck** ‚Äî latest status of each infrastructure component.
   254‚Üí   254‚Üí- **Advertisement** ‚Äî ad placement definition with targeting.
   255‚Üí   255‚Üí
   256‚Üí   256‚Üí**Invariants:**
   257‚Üí   257‚Üí- AuditEntries are append-only; never edited or deleted.
   258‚Üí   258‚Üí- FeatureFlag changes are audited.
   259‚Üí   259‚Üí- Experiments cannot overlap on the same audience segment without explicit conflict resolution.
   260‚Üí   260‚Üí
   261‚Üí   261‚Üí**Owns:** `feature_flags`, `experiments`, `audit_entries`, `system_health_checks`, `advertisements`.
   262‚Üí   262‚Üí
   263‚Üí   263‚Üí### 5.5 Billing Context
   264‚Üí   264‚Üí
   265‚Üí   265‚Üí**Responsibility.** Manage subscriptions, plans, invoices, usage tracking against limits.
   266‚Üí   266‚Üí
   267‚Üí   267‚Üí**Aggregates:**
   268‚Üí   268‚Üí- **Subscription** (aggregate root) ‚Äî userId, planId, status, current period, renewal date.
   269‚Üí   269‚Üí- **Plan** ‚Äî defined tier (free, premium, enterprise) with limits and price.
   270‚Üí   270‚Üí- **Invoice** ‚Äî billing record from payment provider.
   271‚Üí   271‚Üí- **UsageRecord** ‚Äî monthly counter per user per limit type (tool_uses, ai_calls, storage_bytes).
   272‚Üí   272‚Üí
   273‚Üí   273‚Üí**Invariants:**
   274‚Üí   274‚Üí- A User has at most one active Subscription.
   275‚Üí   275‚Üí- UsageRecords reset monthly; rollover not supported in Phase 1.
   276‚Üí   276‚Üí- Plan changes are pro-rated per Stripe-standard logic.
   277‚Üí   277‚Üí
   278‚Üí   278‚Üí**Owns:** `subscriptions`, `plans`, `invoices`, `usage_records`.
   279‚Üí   279‚Üí
   280‚Üí   280‚Üí**Note:** Billing Context is Phase 2+. Phase 1 has a single implicit "free" plan; no billing tables needed.
   281‚Üí   281‚Üí
   282‚Üí   282‚Üí### 5.6 Analytics Context
   283‚Üí   283‚Üí
   284‚Üí   284‚Üí**Responsibility.** Capture, aggregate, and report on user behavior and tool performance.
   285‚Üí   285‚Üí
   286‚Üí   286‚Üí**Aggregates:**
   287‚Üí   287‚Üí- **Event** (aggregate root) ‚Äî raw event record (page_view, tool_started, tool_completed, etc.).
   288‚Üí   288‚Üí- **DailyAggregate** ‚Äî pre-aggregated metric per (date, toolSlug, eventType).
   289‚Üí   289‚Üí- **Funnel** ‚Äî defined sequence of events for conversion tracking.
   290‚Üí   290‚Üí
   291‚Üí   291‚Üí**Invariants:**
   292‚Üí   292‚Üí- Events are immutable once ingested.
   293‚Üí   293‚Üí- Aggregates are computed nightly from raw events; raw events retained for 90 days, aggregates indefinitely.
   294‚Üí   294‚Üí- Personally identifiable information in events is minimized (user ID only, no email/IP).
   295‚Üí   295‚Üí
   296‚Üí   296‚Üí**Owns:** `events`, `daily_aggregates`, `funnels`.
   297‚Üí   297‚Üí
   298‚Üí   298‚Üí**Note:** Analytics Context is the consumer of domain events from all other contexts. It does not publish events; it only subscribes.

   409‚Üí   409‚Üí
   410‚Üí   410‚ÜíWhere a context integrates with an external system, an Anti-Corruption Layer (ACL) translates external concepts to internal ones:
   411‚Üí   411‚Üí
   412‚Üí   412‚Üí- **Identity ACL:** Wraps Supabase Auth; translates Supabase user records to Identity Context's User aggregate.
   413‚Üí   413‚Üí- **Billing ACL:** Wraps Stripe; translates Stripe customer/subscription to Billing Context aggregates.
   414‚Üí   414‚Üí- **Analytics ACL:** Wraps external analytics provider (if used in Phase 2+); translates provider events to Analytics Context Events.
   415‚Üí   415‚Üí
   416‚Üí   416‚ÜíACLs prevent external schema changes from corrupting internal domain models.

   418‚Üí   418‚Üí## 8. Aggregate Consistency Rules
   419‚Üí   419‚Üí
   420‚Üí   420‚Üí### 8.1 Transaction Boundaries
   421‚Üí   421‚Üí- A single transaction modifies exactly ONE aggregate.
   422‚Üí   422‚Üí- Cross-aggregate consistency is achieved via domain events, eventually.
   423‚Üí   423‚Üí- No transaction spans multiple bounded contexts.
   424‚Üí   424‚Üí
   425‚Üí   425‚Üí### 8.2 Aggregate Size
   426‚Üí   426‚Üí- Aggregates are kept small. A Tool is its manifest. A User is its profile + role. A HistoryEntry is a single record.
   427‚Üí   427‚Üí- Large nested structures (e.g., User with all their HistoryEntries) are modeled as separate aggregates with references, not as one big aggregate.
   428‚Üí   428‚Üí
   429‚Üí   429‚Üí### 8.3 Reference Patterns
   430‚Üí   430‚Üí- Aggregates reference other aggregates by ID (slug, userId, articleId), never by direct object reference.
   431‚Üí   431‚Üí- Cross-context references use the foreign context's public identifier (e.g., HistoryEntry references `toolSlug`, not a Tools Context internal ID).
   432‚Üí   432‚Üí
   433‚Üí   433‚Üí## 9. Standards
   434‚Üí   434‚Üí
   435‚Üí   435‚Üí### 9.1 Context Ownership Standards
   436‚Üí   436‚Üí- Every code module MUST belong to exactly one bounded context (declared in module header comment).
   437‚Üí   437‚Üí- Every database table MUST belong to exactly one context (declared in migration header).
   438‚Üí   438‚Üí- Cross-context imports are flagged in code review and require explicit justification.
   439‚Üí   439‚Üí
   440‚Üí   440‚Üí### 9.2 Domain Event Standards
   441‚Üí   441‚Üí- Every domain event MUST be typed (TypeScript interface) and versioned (`event_version: 1`).
   442‚Üí   442‚Üí- Events MUST be idempotent (subscribers can safely receive duplicates).
   443‚Üí   443‚Üí- Events MUST NOT contain PII beyond userId (no emails, no file contents).
   444‚Üí   444‚Üí- Event payloads MUST be small (<10KB); large data referenced by ID.
   445‚Üí   445‚Üí
   446‚Üí   446‚Üí### 9.3 Schema Ownership Standards
   447‚Üí   447‚Üí- Each context owns a Postgres schema (e.g., `tools`, `identity`, `content`, `ops`, `billing`, `analytics`).
   448‚Üí   448‚Üí- No cross-schema foreign keys. References are by string ID.
   449‚Üí   449‚Üí- Migrations are scoped per context; no migration touches another context's schema.
   450‚Üí   450‚Üí
   451‚Üí   451‚Üí## 10. Best Practices
   452‚Üí   452‚Üí
   453‚Üí   453‚Üí### 10.1 When Adding a New Concept
   454‚Üí   454‚Üí1. Identify which bounded context it belongs to. If it doesn't fit any, you may need a new context (rare; requires Chief Architect approval).
   455‚Üí   455‚Üí2. Add the term to the ubiquitous language glossary (¬ß6).
   456‚Üí   456‚Üí3. Define the aggregate it belongs to and the invariants it must enforce.
   457‚Üí   457‚Üí4. Document any domain events it publishes or subscribes to.
   458‚Üí   458‚Üí
   459‚Üí   459‚Üí### 10.2 When Two Contexts Need to Share Data
   460‚Üí   460‚Üí- **Default:** publish a domain event from the owner; subscriber reacts.
   461‚Üí   461‚Üí- **Synchronous read needed:** owner exposes a published read API; subscriber calls it. Never read owner's tables directly.
   462‚Üí   462‚Üí- **Synchronous write needed:** this is a smell. Re-architect so writes are local to the owning context, triggered by event.
   463‚Üí   463‚Üí
   464‚Üí   464‚Üí### 10.3 When Renaming a Domain Concept
   465‚Üí   465‚Üí- Update the glossary first.
   466‚Üí   466‚Üí- Update all code that uses the term.
   467‚Üí   467‚Üí- Reject PRs that introduce synonyms for the renamed concept.
   468‚Üí   468‚Üí- Add a deprecation note in the old term's location pointing to the new term.
   469‚Üí   469‚Üí
   470‚Üí   470‚Üí### 10.4 When Modeling a New Tool
   471‚Üí   471‚Üí- The tool's domain concept is always a **Tool** (an aggregate in Tools Context).
   472‚Üí   472‚Üí- The tool's execution logic belongs in the Tool's Processing stage, not in a separate domain.
   473‚Üí   473‚Üí- The tool does NOT own user data; history of tool use belongs to Identity Context.
   474‚Üí   474‚Üí
   475‚Üí   475‚Üí## 11. Future Scalability
   476‚Üí   476‚Üí
   477‚Üí   477‚Üí### 11.1 Adding New Bounded Contexts
   478‚Üí   478‚ÜíPhase 3+ may introduce:
   479‚Üí   479‚Üí- **Collaboration Context** ‚Äî shared workspaces, team folders.
   480‚Üí   480‚Üí- **Marketplace Context** ‚Äî third-party plugin catalog, reviews, transactions.
   481‚Üí   481‚Üí- **API Context** ‚Äî public API keys, rate limits, usage billing for external API consumers.
   482‚Üí   482‚Üí
   483‚Üí   483‚ÜíAdding a context requires: (a) Chief Architect approval, (b) DDD amendment, (c) schema namespace allocation, (d) context map update.
   484‚Üí   484‚Üí
   485‚Üí   485‚Üí### 11.2 Context Splitting
   486‚Üí   486‚ÜíIf a context grows too large (e.g., Identity grows to include team management, organization management), split it. Split criteria: >15 aggregates, >20 tables, or team ownership boundary.
   487‚Üí   487‚Üí
   488‚Üí   488‚Üí### 11.3 Event Schema Evolution
   489‚Üí   489‚Üí- Events are versioned (`event_version` field).
   490‚Üí   490‚Üí- Subscribers handle all versions they care about; unknown versions are logged and skipped (not crashed).
   491‚Üí   491‚Üí- Breaking changes (field removal, semantic change) require a new event name (e.g., `ToolExecutionStarted` ‚Üí `ToolExecutionStartedV2`).
   492‚Üí   492‚Üí
   493‚Üí   493‚Üí### 11.4 Multi-Tenancy (Phase 3+)
   494‚Üí   494‚Üí- Each context's schema gains a `tenant_id` column.
   495‚Üí   495‚Üí- Aggregate roots enforce tenant isolation in queries.
   496‚Üí   496‚Üí- Platform Ops Context manages tenant administration.
