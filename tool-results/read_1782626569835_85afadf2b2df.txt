    10έΗΤ
    11έΗΤ## 1. Purpose
    12έΗΤ
    13έΗΤThis Folder Structure document defines the granular file and folder conventions for [PROJECT_NAME]. Where `05_ProjectStructure` defines the high-level layout (top-level directories, bounded contexts, layer boundaries, registry pattern), this document specifies the rules for files within those directories: naming, organization, grouping, and what belongs where.
    14έΗΤ
    15έΗΤThe reason this document exists separately from `05_ProjectStructure` is that folder structure failures happen at the file level, not the directory level. An engineer who creates `imageResize.ts` instead of `image-resize.ts`, or who puts a React component in `domain/`, or who creates `utils2.ts` because `utils.ts` got too big έΑΦ these are the failures that compound into an unmaintainable codebase. Without explicit rules, the file-level structure drifts in dozens of small ways that no single reviewer catches.
    16έΗΤ
    17έΗΤThis document enforces LOCK-04 (modularity), LOCK-05 (plugin-ready architecture via consistent tool structure), LOCK-06 (database optional via structural separation), EC-03 (component reuse first), EC-04 (tool template standard), and EC-01 (documentation first έΑΦ every folder has a reason recorded here).

    40έΗΤ## 3. Architectural Decisions
    41έΗΤ
    42έΗΤ### AD-01 έΑΦ File Naming: Three Conventions Based on File Type
    43έΗΤ
    44έΗΤ**Context.** Mixing naming conventions (some files kebab-case, some PascalCase, some camelCase) creates visual inconsistency and forces engineers to remember per-file rules. A clear convention based on file type reduces cognitive load.
    45έΗΤ
    46έΗΤ**Decision.** Three naming conventions, each tied to file type:
    47έΗΤ
    48έΗΤ| File Type | Convention | Example |
    49έΗΤ|-----------|-----------|---------|
    50έΗΤ| Non-component files (utils, hooks, schemas, configs, tests) | `kebab-case.ts` | `image-utils.ts`, `use-tool-history.ts` |
    51έΗΤ| React component files (single component per file) | `PascalCase.tsx` | `ToolCard.tsx`, `ImageUploader.tsx` |
    52έΗΤ| Test files | `[name].test.ts` or `[name].test.tsx` | `image-utils.test.ts`, `ToolCard.test.tsx` |
    53έΗΤ| Type definition files | `kebab-case.types.ts` or `[name].types.ts` | `tool.types.ts` |
    54έΗΤ| Index files (barrels) | `index.ts` or `index.tsx` | always `index` |
    55έΗΤ| Documentation | `kebab-case.md` | `architecture-decisions.md` |
    56έΗΤ| Configuration | `kebab-case.config.ts` | `tailwind.config.ts`, `drizzle.config.ts` |
    57έΗΤ
    58έΗΤ**Rationale:**
    59έΗΤ- `kebab-case` for non-components is the JavaScript ecosystem standard (Next.js, Vercel conventions).
    60έΗΤ- `PascalCase` for components matches the component name (`ToolCard` is in `ToolCard.tsx`), making imports visually obvious.
    61έΗΤ- Test suffix `.test.ts` colocates tests with their subjects, easily greppable.
    62έΗΤ
    63έΗΤ**Implements:** EC-04 (Tool Template Standard έΑΦ consistent naming).
    64έΗΤ
    65έΗΤ### AD-02 έΑΦ One Component Per File
    66έΗΤ
    67έΗΤ**Context.** Files with multiple components become hard to navigate, hard to test, and create import ambiguity.
    68έΗΤ
    69έΗΤ**Decision.** Each `.tsx` file in `presentation/` or `components/` contains exactly ONE exported React component. The file name matches the component name (PascalCase). Small "private" sub-components used only by the main component may live in the same file, but only if not exported.
    70έΗΤ
    71έΗΤ**Consequences.**
    72έΗΤ- έεΖ File name = component name = import path; no ambiguity.
    73έΗΤ- έεΖ Tests target one component per file.
    74έΗΤ- έγιΎ╕Π More files; mitigated by clear folder organization.
    75έΗΤ
    76έΗΤ**Implements:** EC-09 (Testing έΑΦ one component per file makes testing straightforward).
    77έΗΤ
    78έΗΤ### AD-03 έΑΦ Tool Folder Template (EC-04)
    79έΗΤ
    80έΗΤ**Context.** EC-04 mandates every tool follows an identical internal structure. The exact structure must be documented so tool authors have a template to copy.
    81έΗΤ
    82έΗΤ**Decision.** Every tool folder follows this exact structure:
    83έΗΤ
    84έΗΤ```
    85έΗΤsrc/tools/[category]/[tool-slug]/
    86έΗΤέΦεέΦΑέΦΑ manifest.ts              # ToolManifest έΑΦ the aggregate root
    87έΗΤέΦεέΦΑέΦΑ index.ts                 # Re-exports manifest as default
    88έΗΤέΦεέΦΑέΦΑ stages/
    89έΗΤέΦΓ   έΦεέΦΑέΦΑ input.ts             # InputStage implementation
    90έΗΤέΦΓ   έΦεέΦΑέΦΑ validation.ts        # ValidationStage (Zod schema + validator)
    91έΗΤέΦΓ   έΦεέΦΑέΦΑ processing.ts        # ProcessingStage (browser-side or server-side)
    92έΗΤέΦΓ   έΦεέΦΑέΦΑ preview.tsx          # PreviewStage (React component)
    93έΗΤέΦΓ   έΦεέΦΑέΦΑ download.ts          # DownloadStage (packaging logic)
    94έΗΤέΦΓ   έΦεέΦΑέΦΑ history.ts           # HistoryStage (optional)
    95έΗΤέΦΓ   έΦΦέΦΑέΦΑ share.ts             # ShareStage (optional)
    96έΗΤέΦεέΦΑέΦΑ components/              # Tool-specific React components
    97έΗΤέΦΓ   έΦεέΦΑέΦΑ InputForm.tsx        # Configuration form (if any)
    98έΗΤέΦΓ   έΦεέΦΑέΦΑ ProcessingProgress.tsx  # Progress UI
    99έΗΤέΦΓ   έΦΦέΦΑέΦΑ ResultDisplay.tsx    # Result preview UI
   100έΗΤέΦεέΦΑέΦΑ hooks/                   # Tool-specific hooks
   101έΗΤέΦΓ   έΦΦέΦΑέΦΑ use-processing.ts    # e.g., useProcessing()
   102έΗΤέΦεέΦΑέΦΑ lib/                     # Tool-specific pure utilities
   103έΗΤέΦΓ   έΦΦέΦΑέΦΑ image-ops.ts         # e.g., image manipulation functions
   104έΗΤέΦεέΦΑέΦΑ tests/
   105έΗΤέΦΓ   έΦεέΦΑέΦΑ stages.test.ts       # Unit tests per stage
   106έΗΤέΦΓ   έΦεέΦΑέΦΑ input-form.test.tsx  # Component tests
   107έΗΤέΦΓ   έΦΦέΦΑέΦΑ e2e.test.ts          # Playwright E2E test
   108έΗΤέΦΦέΦΑέΦΑ README.md                # Tool-specific docs (optional for simple tools)
   109έΗΤ```
   110έΗΤ
   111έΗΤ**Rules:**
   112έΗΤ- `manifest.ts` and `index.ts` are mandatory.
   113έΗΤ- `stages/` is mandatory with at least 5 files (input, validation, processing, preview, download).
   114έΗΤ- `stages/history.ts` and `stages/share.ts` are optional.
   115έΗΤ- `components/`, `hooks/`, `lib/` are optional; created only when needed.
   116έΗΤ- `tests/` is mandatory with at least `stages.test.ts`.
   117έΗΤ- `README.md` is mandatory for tools with non-obvious behavior; optional for trivial tools.
   118έΗΤ
   119έΗΤ**Implements:** EC-04 (Tool Template Standard), LOCK-05 (Plugin-Ready έΑΦ third-party tools follow same structure).
   120έΗΤ
   121έΗΤ### AD-04 έΑΦ Colocated Tests
   122έΗΤ
   123έΗΤ**Context.** Separating tests into a top-level `tests/` directory separates tests from the code they test, making it harder to keep them in sync.
   124έΗΤ
   125έΗΤ**Decision.** Tests are colocated with the code they test:
   126έΗΤ- Unit tests: `tests/` subdirectory within the module.
   127έΗΤ- Component tests: `tests/` subdirectory within the component's folder.
   128έΗΤ- E2E tests: project-level `tests/e2e/` directory (because they span modules).
   129έΗΤ- Integration tests: project-level `tests/integration/` directory.
   130έΗΤ
   131έΗΤ**Consequences.**
   132έΗΤ- έεΖ Tests are easy to find next to the code.
   133έΗΤ- έεΖ Deleting a module deletes its tests.
   134έΗΤ- έγιΎ╕Π E2E tests span modules; centralized for clarity.
   135έΗΤ
   136έΗΤ**Implements:** EC-09 (Testing Philosophy).
   137έΗΤ
   138έΗΤ### AD-05 έΑΦ Layer Subdirectory Pattern Within Contexts
   139έΗΤ
   140έΗΤ**Context.** `05_ProjectStructure` AD-02 mandates four layer subdirectories per context. Within each layer, files need further organization.
   141έΗΤ
   142έΗΤ**Decision.** Standard subdirectory pattern within each layer:
   143έΗΤ
   144έΗΤ```
   145έΗΤsrc/[context]/
   146έΗΤέΦεέΦΑέΦΑ presentation/
   147έΗΤέΦΓ   έΦεέΦΑέΦΑ components/          # Context-specific React components
   148έΗΤέΦΓ   έΦεέΦΑέΦΑ layouts/             # Layout components
   149έΗΤέΦΓ   έΦΦέΦΑέΦΑ pages/               # Page-level components (if not in app/)
   150έΗΤέΦεέΦΑέΦΑ application/
   151έΗΤέΦΓ   έΦεέΦΑέΦΑ actions/             # Server actions
   152έΗΤέΦΓ   έΦεέΦΑέΦΑ hooks/               # Application hooks
   153έΗΤέΦΓ   έΦεέΦΑέΦΑ services/            # Application services
   154έΗΤέΦΓ   έΦΦέΦΑέΦΑ use-cases/           # Use case orchestration
   155έΗΤέΦεέΦΑέΦΑ domain/
   156έΗΤέΦΓ   έΦεέΦΑέΦΑ entities/            # Domain entities
   157έΗΤέΦΓ   έΦεέΦΑέΦΑ value-objects/       # Value objects
   158έΗΤέΦΓ   έΦεέΦΑέΦΑ aggregates/          # Aggregate roots
   159έΗΤέΦΓ   έΦεέΦΑέΦΑ events/              # Domain events
   160έΗΤέΦΓ   έΦΦέΦΑέΦΑ schemas/             # Zod schemas for domain types
   161έΗΤέΦΦέΦΑέΦΑ infrastructure/
   162έΗΤ    έΦεέΦΑέΦΑ repositories/        # Drizzle-based repositories
   163έΗΤ    έΦεέΦΑέΦΑ clients/             # External API clients
   164έΗΤ    έΦΦέΦΑέΦΑ storage/             # File storage adapters
   165έΗΤ```
   166έΗΤ
   167έΗΤSubdirectories are created only when needed; an empty context may start with empty layer directories.
   168έΗΤ
   169έΗΤ**Implements:** LOCK-04 (Modular), `02_SAD` AD-01 (Layered Architecture).
   170έΗΤ
   171έΗΤ### AD-06 έΑΦ Shared Code Split Rule
   172έΗΤ
   173έΗΤ**Context.** `05_ProjectStructure` AD-05 defines `/packages` and `/src/shared` as two locations for shared code. The decision rule for which goes where must be explicit.
   174έΗΤ
   175έΗΤ**Decision.** Decision rule:
   176έΗΤ
   177έΗΤ| Code Characteristic | Location |
   178έΗΤ|---------------------|----------|
   179έΗΤ| Has zero dependencies on `src/` structure | `/packages` |
   180έΗΤ| Could theoretically be published to npm | `/packages` |
   181έΗΤ| Used by multiple bounded contexts | `/packages` or `/src/shared` (see below) |
   182έΗΤ| Depends on `src/` structure (e.g., imports from another context's published API) | `/src/shared` |
   183έΗΤ| App-specific configuration (env vars, feature flags client) | `/src/shared` |
   184έΗΤ| Cross-context hooks (useUser, useLocale) | `/src/shared` |
   185έΗΤ| Pure utilities (formatting, math) | `/packages/utils` |
   186έΗΤ| Type definitions shared across contexts | `/packages/types` |
   187έΗΤ| Zod schemas shared across contexts | `/packages/types` |
   188έΗΤ| UI primitives (Button, Input, Card) | `/packages/ui` |
   189έΗΤ| Tool Engine abstraction | `/packages/tool-engine` |
   190έΗΤ
   191έΗΤWhen uncertain, default to `/src/shared` έΑΦ promoting to `/packages` is easy; demoting is harder.
   192έΗΤ
   193έΗΤ**Implements:** EC-02 (One Source of Truth), EC-03 (Component Reuse First).
   194έΗΤ
   195έΗΤ### AD-07 έΑΦ Generated Files Convention
   196έΗΤ
   197έΗΤ**Context.** `05_ProjectStructure` AD-04 generates files via build-time codegen. Generated files must be clearly marked and never manually edited.
   198έΗΤ
   199έΗΤ**Decision.** All generated files:
   200έΗΤ- Live in `src/generated/` only.
   201έΗΤ- Start with the header comment `// AUTO-GENERATED. Do not edit manually.`
   202έΗΤ- Are committed to the repo (for type safety across the team).
   203έΗΤ- Are verified by CI (`scripts/verify-registry.ts` fails the build if generated files are out of sync with manifests).
   204έΗΤ
   205έΗΤ**Consequences.**
   206έΗΤ- έεΖ Type safety without running codegen before lint.
   207έΗΤ- έεΖ Diff visibility in PRs.
   208έΗΤ- έγιΎ╕Π Discipline required to never manually edit.
   209έΗΤ
   210έΗΤ**Implements:** LOCK-05 (Plugin-Ready via codegen), EC-01 (Documentation έΑΦ generated files are documented as such).
   211έΗΤ
   212έΗΤ### AD-08 έΑΦ Documentation File Placement
   213έΗΤ
   214έΗΤ**Context.** EC-01 mandates documentation first. Documentation files must be discoverable.
   215έΗΤ
   216έΗΤ**Decision.** Documentation placement rules:
   217έΗΤ
   218έΗΤ| Document Type | Location |
   219έΗΤ|--------------|----------|
   220έΗΤ| Project-level docs (architecture, conventions) | `/docs/*.md` |
   221έΗΤ| Tool-specific docs | `src/tools/[category]/[slug]/README.md` |
   222έΗΤ| Package docs | `/packages/[name]/README.md` |
   223έΗΤ| ADRs | `/docs/06_ArchitectureDecisionRecords.md` (consolidated) |
   224έΗΤ| API docs | `/docs/17_APIConvention.md` + inline JSDoc |
   225έΗΤ| README | Project root `README.md` (links to `/docs`) |
   226έΗΤ
   227έΗΤ**Implements:** EC-01 (Documentation First).

   229έΗΤ## 4. Design Principles
   230έΗΤ
   231έΗΤ### P1 έΑΦ Convention Over Configuration
   232έΗΤFollow the conventions in this document; don't invent new patterns. If a convention doesn't fit your case, propose an ADR rather than deviating silently.
   233έΗΤ
   234έΗΤ### P2 έΑΦ Colocation
   235έΗΤCode, tests, and docs for a module live together. Don't separate them by type.
   236έΗΤ
   237έΗΤ### P3 έΑΦ Discoverability
   238έΗΤAn engineer browsing the file tree should be able to find what they need without grepping. Folder and file names must be self-explanatory.
   239έΗΤ
   240έΗΤ### P4 έΑΦ Minimal Subdirectories
   241έΗΤDon't create subdirectories until you have 5+ files in a folder. Premature subdirectories create empty navigation.
   242έΗΤ
   243έΗΤ### P5 έΑΦ Naming Reflects Purpose
   244έΗΤFile and folder names reflect what they contain, not who created them or when. Avoid names like `utils-v2.ts` or `johns-stuff/`.
   245έΗΤ
   246έΗΤ### P6 έΑΦ Generated Files Are Sacred
   247έΗΤNever manually edit generated files. If a generated file is wrong, fix the source (manifest, codegen script) and regenerate.

   275έΗΤ## 6. Detailed Folder Rules
   276έΗΤ
   277έΗΤ### 6.1 `/src/app/` έΑΦ Next.js App Router
   278έΗΤ
   279έΗΤ```
   280έΗΤsrc/app/
   281έΗΤέΦεέΦΑέΦΑ (public)/                # Public route group (no auth required)
   282έΗΤέΦΓ   έΦεέΦΑέΦΑ tools/[category]/[slug]/page.tsx   # Single dynamic route for all tools
   283έΗΤέΦΓ   έΦεέΦΑέΦΑ tools/[category]/page.tsx          # Category landing
   284έΗΤέΦΓ   έΦΦέΦΑέΦΑ tools/page.tsx                     # All-tools hub
   285έΗΤέΦεέΦΑέΦΑ (auth)/                  # Auth-required route group
   286έΗΤέΦΓ   έΦεέΦΑέΦΑ login/page.tsx
   287έΗΤέΦΓ   έΦεέΦΑέΦΑ register/page.tsx
   288έΗΤέΦΓ   έΦεέΦΑέΦΑ dashboard/page.tsx
   289έΗΤέΦΓ   έΦΦέΦΑέΦΑ settings/page.tsx
   290έΗΤέΦεέΦΑέΦΑ (admin)/                 # Admin route group (RBAC required)
   291έΗΤέΦΓ   έΦεέΦΑέΦΑ admin/layout.tsx     # Admin layout with sidebar
   292έΗΤέΦΓ   έΦΦέΦΑέΦΑ admin/page.tsx       # Admin dashboard
   293έΗΤέΦεέΦΑέΦΑ (marketing)/             # Marketing pages
   294έΗΤέΦΓ   έΦεέΦΑέΦΑ page.tsx             # Homepage
   295έΗΤέΦΓ   έΦεέΦΑέΦΑ about/page.tsx
   296έΗΤέΦΓ   έΦεέΦΑέΦΑ pricing/page.tsx
   297έΗΤέΦΓ   έΦΦέΦΑέΦΑ blog/page.tsx        # Content Context articles
   298έΗΤέΦεέΦΑέΦΑ api/                     # API routes (server-side tools, webhooks)
   299έΗΤέΦΓ   έΦεέΦΑέΦΑ tools/[slug]/route.ts   # Server-side tool endpoints
   300έΗΤέΦΓ   έΦΦέΦΑέΦΑ webhooks/               # Webhook handlers
   301έΗΤέΦεέΦΑέΦΑ layout.tsx               # Root layout (HTML, body, providers)
   302έΗΤέΦεέΦΑέΦΑ globals.css              # Global styles (Tailwind imports, CSS custom properties)
   303έΗΤέΦεέΦΑέΦΑ not-found.tsx            # 404 page
   304έΗΤέΦΦέΦΑέΦΑ error.tsx                # Global error boundary
   305έΗΤ```
   306έΗΤ
   307έΗΤ**Rules:**
   308έΗΤ- Route groups `(name)` don't affect URL; used for organization and shared layouts.
   309έΗΤ- One `page.tsx` per route; no business logic in page files (delegate to context's application layer).
   310έΗΤ- `layout.tsx` files contain shared layout, navigation, and providers.
   311έΗΤ
   312έΗΤ### 6.2 `/src/tools/[category]/[slug]/` έΑΦ Tool Folder Template
   313έΗΤ
   314έΗΤSee AD-03 for the full template. Every tool folder follows this exact structure.
   315έΗΤ
   316έΗΤ**Category folders** (`image/`, `pdf/`, `developer/`, `text/`, `converters/`, `seo/`, `calculators/`) each contain their tool subfolders and an optional `_shared/` folder for category-specific shared code:
   317έΗΤ
   318έΗΤ```
   319έΗΤsrc/tools/image/
   320έΗΤέΦεέΦΑέΦΑ _shared/                 # Image-category-specific shared code
   321έΗΤέΦΓ   έΦεέΦΑέΦΑ lib/

   563έΗΤ## 8. Standards
   564έΗΤ
   565έΗΤ### 8.1 File Naming Standards
   566έΗΤ- Non-component TypeScript files: `kebab-case.ts`
   567έΗΤ- Component files: `PascalCase.tsx`
   568έΗΤ- Test files: `[name].test.ts` or `[name].test.tsx`
   569έΗΤ- Type files: `[name].types.ts`
   570έΗΤ- Documentation: `kebab-case.md`
   571έΗΤ- Config: `kebab-case.config.ts`
   572έΗΤ
   573έΗΤ### 8.2 Folder Naming Standards
   574έΗΤ- All folders: `kebab-case`
   575έΗΤ- Special prefixes: `_shared/` (sorts first, indicates internal shared)
   576έΗΤ- Route groups in `app/`: `(name)` per Next.js convention
   577έΗΤ
   578έΗΤ### 8.3 File Size Standards (per `00_Project_Charter` ┬π7.1)
   579έΗΤ- Soft limit: 300 lines per file
   580έΗΤ- Hard limit: 500 lines per file
   581έΗΤ- Files approaching 300 lines should be refactored into smaller modules
   582έΗΤ- Files exceeding 500 lines block PR merge
   583έΗΤ
   584έΗΤ### 8.4 Folder Cardinality Standards
   585έΗΤ- Don't create a subdirectory until parent has 5+ files
   586έΗΤ- Don't create more than 2 levels of nesting within a context's layer
   587έΗΤ- Single-file folders are acceptable (e.g., `stages/input.ts`)
   588έΗΤ
   589έΗΤ### 8.5 Index File Standards
   590έΗΤ- `index.ts` files are barrel files that re-export public APIs
   591έΗΤ- They do NOT contain logic
   592έΗΤ- They prevent deep import paths (`@/tools/image/image-resize` vs `@/tools/image/image-resize/manifest`)
   593έΗΤ
   594έΗΤ### 8.6 Test File Standards
   595έΗΤ- Every test file is named `[subject].test.ts` or `[subject].test.tsx`
   596έΗΤ- Test files live in `tests/` subdirectory of their subject's folder
   597έΗΤ- E2E tests live in project-level `tests/e2e/`
   598έΗΤ- Test fixtures live in `tests/fixtures/`
   599έΗΤ
   600έΗΤ## 9. Best Practices
   601έΗΤ
   602έΗΤ### 9.1 When Creating a New Tool
   603έΗΤ1. Run `pnpm new-tool [category] [slug]` to scaffold from template.
   604έΗΤ2. Edit `manifest.ts` with tool metadata.
   605έΗΤ3. Implement each stage in `stages/`.
   606έΗΤ4. Add components, hooks, lib as needed.
   607έΗΤ5. Write tests in `tests/`.
   608έΗΤ6. Run `pnpm gen:registry` to update generated files.
   609έΗΤ7. Verify with `pnpm test` and `pnpm lint`.
   610έΗΤ
   611έΗΤ### 9.2 When Adding a Component
   612έΗΤ1. Check `@packages/ui` for existing component (EC-03).
   613έΗΤ2. Check `src/shared/components/` for app-specific shared component.
   614έΗΤ3. If tool-specific, place in tool's `components/` folder.
   615έΗΤ4. Name file `PascalCase.tsx` matching component name.
   616έΗΤ5. One component per file (AD-02).
   617έΗΤ
   618έΗΤ### 9.3 When Adding a Utility Function
   619έΗΤ1. Check `@packages/utils` for existing utility (EC-03).
   620έΗΤ2. Check context's `domain/` or `application/` for context-specific utility.
   621έΗΤ3. If tool-specific, place in tool's `lib/` folder.
   622έΗΤ4. Name file `kebab-case.ts` describing the utility group (e.g., `image-ops.ts`, not `helpers.ts`).
   623έΗΤ
   624έΗΤ### 9.4 When Adding a Test
   625έΗΤ1. Place next to the code being tested (AD-04).
   626έΗΤ2. Name `[subject].test.ts`.
   627έΗΤ3. E2E tests go in `tests/e2e/` with descriptive name `[scenario].spec.ts`.
   628έΗΤ4. Use fixtures from `tests/fixtures/` for sample files.
   629έΗΤ
   630έΗΤ### 9.5 When Refactoring
   631έΗΤ1. Preserve file naming conventions.
   632έΗΤ2. Don't create temporary folders like `old/`, `deprecated/`, `v2/`.
   633έΗΤ3. If deleting a file, delete its tests too (colocated).
   634έΗΤ4. Update imports across the codebase; rely on TypeScript to catch misses.
   635έΗΤ
   636έΗΤ### 9.6 When Adding Documentation
   637έΗΤ1. Project-level docs go in `/docs/` with numbered filename.
   638έΗΤ2. Tool-specific docs go in tool's `README.md`.
   639έΗΤ3. Package docs go in package's `README.md`.
   640έΗΤ4. Inline code comments use JSDoc for public APIs.
   641έΗΤ

   644έΗΤ### 10.1 Scaling to 1,000+ Tools

   651έΗΤ- Each package remains self-contained; no cross-package imports except via published APIs.
   652έΗΤ
   653έΗΤ### 10.3 Plugin Directory (Phase 4)
   654έΗΤ- Add `/plugins/` directory for third-party tools.
   655έΗΤ- Plugin folder structure mirrors tool folder structure (AD-03).
   656έΗΤ- Plugins are signed and sandboxed per future ADR.
   657έΗΤ
   658έΗΤ### 10.4 Multi-Tenancy (Phase 3+)
   659έΗΤ- Each context's `infrastructure/` layer gains `tenancy/` subdirectory for tenant-aware repositories.
   660έΗΤ- No structural changes to top-level layout.
