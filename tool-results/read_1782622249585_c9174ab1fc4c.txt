    11тЖТ    11тЖТ## 1. Purpose
    12тЖТ    12тЖТ
    13тЖТ    13тЖТThis Tech Stack document defines every technology choice in [PROJECT_NAME] and the rationale behind it. It operationalizes the Architectural Locks in `00_Project_Charter` ┬з3 with concrete tools, libraries, and services. It also defines the **Dependency Policy** that governs future technology additions and the **Upgrade Path** for each choice as the platform scales from Phase 1 ($0 budget) to Phase 4 (revenue-funded).
    14тЖТ    14тЖТ
    15тЖТ    15тЖТThis document is the authoritative reference for "what technology do we use and why." Engineers consulting this doc should never need to ask "why did we choose X over Y?" тАФ the rationale is recorded here. New dependencies cannot be introduced without amending this document and getting Chief Architect approval (LOCK-09).
    16тЖТ    16тЖТ
    17тЖТ    17тЖТThe stack is deliberately conservative. We favor mature, well-maintained, free-tier-friendly technologies over novelty. Every choice must satisfy three filters: (1) does it serve an Architectural Lock? (2) does it have a viable free tier through Phase 1? (3) does it have a credible upgrade path should we outgrow the free tier? Technologies failing any filter are rejected.

    37тЖТ    37тЖТ## 3. Architectural Decisions
    38тЖТ    38тЖТ
    39тЖТ    39тЖТ### AD-01 тАФ Next.js 15+ App Router as the Unified Framework
    40тЖТ    40тЖТ
    41тЖТ    41тЖТ**Context.** The platform needs SSR/SSG for SEO (LOCK-08), Edge runtime for performance, API routes for server-side tools, and a component model for the Tool Engine UI. Using separate frameworks (e.g., Vite for SPA + Express for API) fragments the codebase and complicates deployment.
    42тЖТ    42тЖТ
    43тЖТ    43тЖТ**Decision.** Adopt Next.js 15+ with App Router as the single application framework. It provides:
    44тЖТ    44тЖТ- **Edge Runtime** for tool landing pages and middleware (LOCK-02 performance).
    45тЖТ    45тЖТ- **Node.js Runtime** for server-side tools and admin actions.
    46тЖТ    46тЖТ- **Server Components** for SEO-friendly initial render.
    47тЖТ    47тЖТ- **API Routes** for server-side tool endpoints.
    48тЖТ    48тЖТ- **File-based routing** that maps naturally to the tool registry (LOCK-05).
    49тЖТ    49тЖТ
    50тЖТ    50тЖТ**Consequences.**
    51тЖТ    51тЖТ- тЬЕ One framework to learn; one deployment target.
    52тЖТ    52тЖТ- тЬЕ Edge + Node runtimes available without separate config.
    53тЖТ    53тЖТ- тЬЕ Built-in code splitting supports lazy tool loading (LOCK-04).
    54тЖТ    54тЖТ- тЪая╕П Next.js version upgrades require care; major versions can break patterns.
    55тЖТ    55тЖТ- тЪая╕П Some libraries are incompatible with Edge Runtime; documented per-tool.
    56тЖТ    56тЖТ
    57тЖТ    57тЖТ**Implements:** LOCK-02 (Edge Runtime for browser-first page delivery), LOCK-05 (file-based routing aligns with registry).
    58тЖТ    58тЖТ
    59тЖТ    59тЖТ### AD-02 тАФ TypeScript Strict Mode End-to-End
    60тЖТ    60тЖТ
    61тЖТ    61тЖТ**Context.** The platform will scale to 1,000+ tools, each with its own manifest and stages. Without strict typing, runtime type errors will scale linearly with tool count, becoming unmaintainable.
    62тЖТ    62тЖТ
    63тЖТ    63тЖТ**Decision.** TypeScript 5+ with `strict: true` enabled. No `any` types allowed (ESLint-enforced). Every IO boundary (request, response, manifest, stage input/output) validated with Zod schemas that produce typed outputs.
    64тЖТ    64тЖТ
    65тЖТ    65тЖТ**Consequences.**
    66тЖТ    66тЖТ- тЬЕ Type errors caught at compile time, not runtime.
    67тЖТ    67тЖТ- тЬЕ Tool manifests are statically verifiable before registry codegen.
    68тЖТ    68тЖТ- тЬЕ Refactoring across 1,000+ tools is feasible with type-checked find/replace.
    69тЖТ    69тЖТ- тЪая╕П Slightly more verbose code; justified by maintainability.
    70тЖТ    70тЖТ
    71тЖТ    71тЖТ**Implements:** Type safety end-to-end standard (`00_Project_Charter` ┬з6.1).
    72тЖТ    72тЖТ
    73тЖТ    73тЖТ### AD-03 тАФ Supabase for Identity, Database, and Storage
    74тЖТ    74тЖТ
    75тЖТ    75тЖТ**Context.** The platform needs Postgres (relational integrity for users, history, content), auth (OAuth + email), and file storage (server-side tool uploads, media assets). Three separate services would triple integration cost.
    76тЖТ    76тЖТ
    77тЖТ    77тЖТ**Decision.** Adopt Supabase (free tier) as the unified backend for:
    78тЖТ    78тЖТ- **Postgres database** тАФ all persistent data across bounded contexts (`03_DDD`).
    79тЖТ    79тЖТ- **Auth** тАФ email/password, OAuth (Google, GitHub), JWT-based sessions.
    80тЖТ    80тЖТ- **Storage** тАФ server-side tool file uploads (temporary bucket, 1h TTL), media assets for Content Context.
    81тЖТ    81тЖТ
    82тЖТ    82тЖТ**Consequences.**
    83тЖТ    83тЖТ- тЬЕ One integration; one set of credentials; one dashboard.
    84тЖТ    84тЖТ- тЬЕ Row-Level Security (RLS) enforced at DB layer for defense-in-depth.
    85тЖТ    85тЖТ- тЬЕ Free tier covers Phase 1 (50k MAU, 500MB DB, 1GB storage).
    86тЖТ    86тЖТ- тЪая╕П Vendor lock-in for auth; mitigated by Supabase's JWT standard (can migrate to Auth0/Clerk if needed).
    87тЖТ    87тЖТ- тЪая╕П Free tier DB pauses after 1 week of inactivity; mitigated by scheduled pings.
    88тЖТ    88тЖТ
    89тЖТ    89тЖТ**Implements:** LOCK-06 (database optional тАФ Tools Context doesn't use Supabase, only Identity/Content/Platform Ops/Billing/Analytics do).
    90тЖТ    90тЖТ
    91тЖТ    91тЖТ### AD-04 тАФ Drizzle ORM for Type-Safe Database Access
    92тЖТ    92тЖТ
    93тЖТ    93тЖТ**Context.** Raw SQL loses type safety. Heavy ORMs (Prisma, TypeORM) add runtime overhead and complex migration workflows. The platform needs type-safe DB access that works in Edge Runtime (limited Node APIs).
    94тЖТ    94тЖТ
    95тЖТ    95тЖТ**Decision.** Adopt Drizzle ORM for all database access in the Infrastructure Layer. Drizzle provides:
    96тЖТ    96тЖТ- Type-safe schema definitions that match TypeScript types.
    97тЖТ    97тЖТ- SQL-like query builder (no magic; predictable SQL).
    98тЖТ    98тЖТ- Works in Edge Runtime (unlike Prisma's query engine).
    99тЖТ    99тЖТ- Migration files are plain SQL, reviewable in PRs.
   100тЖТ   100тЖТ
   101тЖТ   101тЖТ**Consequences.**
   102тЖТ   102тЖТ- тЬЕ Type safety from schema to query to result.
   103тЖТ   103тЖТ- тЬЕ Edge Runtime compatible.
   104тЖТ   104тЖТ- тЬЕ Migrations are version-controlled SQL, not auto-generated black boxes.
   105тЖТ   105тЖТ- тЪая╕П More verbose than Prisma; explicit query writing required.
   106тЖТ   106тЖТ- тЪая╕П Smaller community than Prisma; mitigated by excellent docs.
   107тЖТ   107тЖТ
   108тЖТ   108тЖТ**Implements:** LOCK-04 (modularity тАФ Drizzle schemas defined per bounded context).
   109тЖТ   109тЖТ
   110тЖТ   110тЖТ### AD-05 тАФ shadcn/ui for UI Components
   111тЖТ   111тЖТ
   112тЖТ   112тЖТ**Context.** Component libraries like MUI or Ant Design are heavy and opinionated. The platform needs a lightweight, accessible, customizable component system that aligns with the developer-first minimalism design (LOCK-10).
   113тЖТ   113тЖТ
   114тЖТ   114тЖТ**Decision.** Adopt shadcn/ui тАФ a collection of copy-into-your-codebase components built on Radix UI primitives and Tailwind CSS. Components are owned (not depended upon), enabling full customization.
   115тЖТ   115тЖТ
   116тЖТ   116тЖТ**Consequences.**
   117тЖТ   117тЖТ- тЬЕ No version lock-in; components live in our codebase.
   118тЖТ   118тЖТ- тЬЕ Built on Radix UI (accessibility-first, WCAG compliant тАФ supports `12_UDS`).
   119тЖТ   119тЖТ- тЬЕ Tailwind-based (aligns with `09_DesignSystem` token approach).
   120тЖТ   120тЖТ- тЬЕ Tree-shakeable тАФ only used components are bundled.
   121тЖТ   121тЖТ- тЪая╕П Updates require manual re-sync; mitigated by CLI tool.
   122тЖТ   122тЖТ
   123тЖТ   123тЖТ**Implements:** LOCK-10 (developer-first minimalism тАФ clean, accessible, customizable).
   124тЖТ   124тЖТ
   125тЖТ   125тЖТ### AD-06 тАФ Tailwind CSS for Styling
   126тЖТ   126тЖТ
   127тЖТ   127тЖТ**Context.** CSS-in-JS adds runtime overhead. Plain CSS lacks utility-first productivity. The platform needs a styling system that supports design tokens (LOCK-10) without runtime cost.
   128тЖТ   128тЖТ
   129тЖТ   129тЖТ**Decision.** Adopt Tailwind CSS 4+ for all styling. Design tokens (colors, spacing, typography) defined as CSS custom properties at the root, mapped to Tailwind theme extensions.
   130тЖТ   130тЖТ
   131тЖТ   131тЖТ**Consequences.**
   132тЖТ   132тЖТ- тЬЕ Zero runtime CSS-in-JS overhead.
   133тЖТ   133тЖТ- тЬЕ Token-driven theming enables light/dark mode (LOCK-10).
   134тЖТ   134тЖТ- тЬЕ Utility-first approach keeps styles colocated with components.
   135тЖТ   135тЖТ- тЪая╕П Class names can get long; mitigated by component extraction.
   136тЖТ   136тЖТ
   137тЖТ   137тЖТ**Implements:** LOCK-10 (token-driven, dark/light ready).
   138тЖТ   138тЖТ
   139тЖТ   139тЖТ### AD-07 тАФ Zod for Runtime Validation
   140тЖТ   140тЖТ
   141тЖТ   141тЖТ**Context.** TypeScript types exist only at compile time. Runtime data (user input, API responses, file metadata) needs validation. Without it, type mismatches cause runtime errors.
   142тЖТ   142тЖТ
   143тЖТ   143тЖТ**Decision.** Adopt Zod for all runtime validation. Every IO boundary (tool input, API request, manifest fields, form submissions) is validated with a Zod schema. Zod schemas are the single source of truth; TypeScript types are inferred from them.
   144тЖТ   144тЖТ
   145тЖТ   145тЖТ**Consequences.**
   146тЖТ   146тЖТ- тЬЕ Single source of truth тАФ no drift between types and validators.
   147тЖТ   147тЖТ- тЬЕ Tool manifests are runtime-verifiable.
   148тЖТ   148тЖТ- тЬЕ Form validation integrates via react-hook-form's Zod resolver.
   149тЖТ   149тЖТ- тЪая╕П Small runtime cost; negligible for validation-heavy use cases.
   150тЖТ   150тЖТ
   151тЖТ   151тЖТ**Implements:** Type safety end-to-end standard.
   152тЖТ   152тЖТ
   153тЖТ   153тЖТ### AD-08 тАФ React Hook Form + Zod for Forms
   154тЖТ   154тЖТ
   155тЖТ   155тЖТ**Context.** Tool configuration forms (e.g., image resize dimensions, PDF merge order) need performant, type-safe form management. Controlled React state is too slow for complex forms.
   156тЖТ   156тЖТ
   157тЖТ   157тЖТ**Decision.** Adopt React Hook Form for all form state, integrated with Zod via `@hookform/resolvers/zod`. Uncontrolled by default; re-renders minimized.
   158тЖТ   158тЖТ
   159тЖТ   159тЖТ**Consequences.**
   160тЖТ   160тЖТ- тЬЕ Performant forms with minimal re-renders.
   161тЖТ   161тЖТ- тЬЕ Type-safe form values via Zod schema inference.
   162тЖТ   162тЖТ- тЬЕ Works with shadcn/ui form components.
   163тЖТ   163тЖТ
   164тЖТ   164тЖТ**Implements:** LOCK-03 (Tool Engine input stage uses forms for configuration).
   165тЖТ   165тЖТ
   166тЖТ   166тЖТ### AD-09 тАФ Zustand for Client State
   167тЖТ   167тЖТ
   168тЖТ   168тЖТ**Context.** Some client state (theme, recent tools, in-progress tool state) doesn't belong in URL or server. Redux is overkill; React Context causes re-render storms.
   169тЖТ   169тЖТ
   170тЖТ   170тЖТ**Decision.** Adopt Zustand for global client state. Stores are small, typed, and composable. Server state (history, favorites) handled separately by TanStack Query (Phase 2+).
   171тЖТ   171тЖТ
   172тЖТ   172тЖТ**Consequences.**
   173тЖТ   173тЖТ- тЬЕ Minimal API; small bundle.
   174тЖТ   174тЖТ- тЬЕ No provider wrapper needed.
   175тЖТ   175тЖТ- тЬЕ Selectors prevent unnecessary re-renders.
   176тЖТ   176тЖТ
   177тЖТ   177тЖТ**Implements:** LOCK-04 (state management is local to modules where possible).
   178тЖТ   178тЖТ
   179тЖТ   179тЖТ### AD-10 тАФ Vercel for Hosting and Edge Delivery
   180тЖТ   180тЖТ
   181тЖТ   181тЖТ**Context.** The platform needs global CDN, Edge functions, serverless functions, and seamless Next.js integration. Separate providers for each would be operationally complex.
   182тЖТ   182тЖТ
   183тЖТ   183тЖТ**Decision.** Adopt Vercel as the unified hosting platform. Free tier covers Phase 1 (100GB bandwidth, unlimited Edge requests, 100GB-hours serverless).
   184тЖТ   184тЖТ
   185тЖТ   185тЖТ**Consequences.**
   186тЖТ   186тЖТ- тЬЕ Zero-config Next.js deployment.
   187тЖТ   187тЖТ- тЬЕ Edge Network included.
   188тЖТ   188тЖТ- тЬЕ Preview deployments per PR (improves review workflow).
   189тЖТ   189тЖТ- тЪая╕П Vendor lock-in for hosting config; mitigated by Next.js being portable to Netlify/self-host if needed.
   190тЖТ   190тЖТ
   191тЖТ   191тЖТ**Implements:** AD-06 in `02_SAD` (Edge-first deployment topology).
   192тЖТ   192тЖТ
   193тЖТ   193тЖТ### AD-11 тАФ pnpm as Package Manager
   194тЖТ   194тЖТ
   195тЖТ   195тЖТ**Context.** npm and yarn have inefficiencies in monorepo and large dependency trees. The platform will have many shared packages and tool modules.
   196тЖТ   196тЖТ
   197тЖТ   197тЖТ**Decision.** Adopt pnpm as the package manager. Uses a content-addressable store, saving disk space and install time. Strict about peer dependencies (prevents phantom deps).
   198тЖТ   198тЖТ
   199тЖТ   199тЖТ**Consequences.**
   200тЖТ   200тЖТ- тЬЕ Faster installs, less disk usage.
   201тЖТ   201тЖТ- тЬЕ Strict peer dep enforcement prevents subtle bugs.
   202тЖТ   202тЖТ- тЬЕ First-class monorepo support via workspaces.
   203тЖТ   203тЖТ- тЪая╕П Slightly different node_modules layout; rare compatibility issues.
   204тЖТ   204тЖТ
   205тЖТ   205тЖТ**Implements:** LOCK-04 (modularity тАФ pnpm workspaces enable clean package boundaries).

   207тЖТ   207тЖТ## 4. Design Principles
   208тЖТ   208тЖТ
   209тЖТ   209тЖТ### P1 тАФ Free-Tier-First
   210тЖТ   210тЖТEvery external service MUST have a viable free tier that covers Phase 1 (10k MAU, 30 tools). Paid services are introduced only when free tier is provably inadequate, documented in `01_BRD` ┬з8.1.
   211тЖТ   211тЖТ
   212тЖТ   212тЖТ### P2 тАФ Minimal Dependencies
   213тЖТ   213тЖТEvery dependency is a future maintenance liability. Reject dependencies that can be replaced with <100 lines of code. Prefer standard Web APIs over libraries where possible.
   214тЖТ   214тЖТ
   215тЖТ   215тЖТ### P3 тАФ Type Safety End-to-End
   216тЖТ   216тЖТTypeScript strict + Zod runtime validation at every IO boundary. No `any`. No untyped API responses.
   217тЖТ   217тЖТ
   218тЖТ   218тЖТ### P4 тАФ Edge-Compatible Where Possible
   219тЖТ   219тЖТCode that runs in Edge Runtime (auth middleware, page rendering) MUST avoid Node-only APIs. This constraint is documented per-module.
   220тЖТ   220тЖТ
   221тЖТ   221тЖТ### P5 тАФ Browser-Native First
   222тЖТ   222тЖТFor browser-side tools (LOCK-02), prefer native Web APIs (Canvas, Web Crypto, File System Access) over libraries. Use libraries only when native APIs are insufficient (e.g., pdf-lib for PDF manipulation).
   223тЖТ   223тЖТ
   224тЖТ   224тЖТ### P6 тАФ Upgrade Path Documented
   225тЖТ   225тЖТEvery external service dependency MUST have a documented upgrade path. "We'll figure it out when we hit the limit" is not acceptable. *Implements EC-12 (Enterprise Readiness).*

   313тЖТ   313тЖТ
   314тЖТ   314тЖТ### 6.1 Adding a New Dependency
   315тЖТ   315тЖТ
   316тЖТ   316тЖТAny new dependency (npm package or external service) requires:
   317тЖТ   317тЖТ
   318тЖТ   318тЖТ1. **Justification written in PR description** answering:
   319тЖТ   319тЖТ   - What problem does this solve?
   320тЖТ   320тЖТ   - Why can't it be solved with existing deps or <100 lines of code?
   321тЖТ   321тЖТ   - What is the bundle size / cost impact?
   322тЖТ   322тЖТ   - What is the free-tier adequacy?
   323тЖТ   323тЖТ   - What is the upgrade path?
   324тЖТ   324тЖТ
   325тЖТ   325тЖТ2. **Approval from Chief Architect.** LOCK-09 mandates no unapproved dependencies.
   326тЖТ   326тЖТ
   327тЖТ   327тЖТ3. **Update to this document** (`04_TechStack` ┬з5) recording the addition.
   328тЖТ   328тЖТ
   329тЖТ   329тЖТ4. **Update to `04_TechStack` revision history** noting the addition.
   330тЖТ   330тЖТ
   331тЖТ   331тЖТ### 6.2 Removing a Dependency
   332тЖТ   332тЖТ
   333тЖТ   333тЖТRemoving a dependency is encouraged when:
   334тЖТ   334тЖТ- It's used by only one tool and can be inlined.
   335тЖТ   335тЖТ- It's unmaintained (no commits in 12+ months).
   336тЖТ   336тЖТ- A native Web API now covers its functionality.
   337тЖТ   337тЖТ- It's been replaced by a more capable existing dependency.
   338тЖТ   338тЖТ
   339тЖТ   339тЖТRemoval PRs are fast-tracked; no Chief Architect approval needed.
   340тЖТ   340тЖТ
   341тЖТ   341тЖТ### 6.3 Versioning Strategy
   342тЖТ   342тЖТ
   343тЖТ   343тЖТ- **Patch updates** (e.g., 1.2.3 тЖТ 1.2.4): auto-merged by Renovate.

   345тЖТ   345тЖТ- **Major updates** (e.g., 1.2.3 тЖТ 2.0.0): manual review required; migration plan documented in PR.
   346тЖТ   346тЖТ- **Next.js major updates**: dedicated upgrade PR with full regression test.

   380тЖТ   380тЖТ## 8. Browser-First Technology Choices (LOCK-02)
   381тЖТ   381тЖТ
   382тЖТ   382тЖТThis section details the Web APIs and patterns that make browser-first processing possible. Tool authors consult this section when choosing how to implement a tool's Processing stage.
   383тЖТ   383тЖТ
   384тЖТ   384тЖТ### 8.1 Image Processing
   385тЖТ   385тЖТ
   386тЖТ   386тЖТ- **Canvas API** тАФ resize, crop, format convert (PNG/JPEG/WebP). Native, zero bundle.
   387тЖТ   387тЖТ- **OffscreenCanvas** тАФ image processing in Web Worker, doesn't block UI.
   388тЖТ   388тЖТ- **createImageBitmap** тАФ fast image decoding from Blob/File.
   389тЖТ   389тЖТ- **browser-image-compression** тАФ library for compression with quality control.
   390тЖТ   390тЖТ
   391тЖТ   391тЖТ### 8.2 PDF Processing
   392тЖТ   392тЖТ
   393тЖТ   393тЖТ- **pdf-lib** тАФ pure JS PDF creation, modification, merge, split, watermark. ~300KB but lazy-loaded only on PDF tool routes.
   394тЖТ   394тЖТ
   395тЖТ   395тЖТ### 8.3 Cryptography
   396тЖТ   396тЖТ
   397тЖТ   397тЖТ- **Web Crypto API** тАФ SHA-1, SHA-256, SHA-384, SHA-512, AES, HMAC. Native, zero bundle.
   398тЖТ   398тЖТ- **crypto-js** тАФ fallback for algorithms not in Web Crypto (MD5, RIPEMD). Only if needed.
   399тЖТ   399тЖТ
   400тЖТ   400тЖТ### 8.4 File System Access
   401тЖТ   401тЖТ
   402тЖТ   402тЖТ- **File System Access API** тАФ direct file read/write on supported browsers (Chrome, Edge). Falls back to download prompt on Safari/Firefox.
   403тЖТ   403тЖТ- **IndexedDB** тАФ store tool history client-side; persists across sessions.
   404тЖТ   404тЖТ- **Blob URLs** тАФ preview generated files without server round-trip.
   405тЖТ   405тЖТ
   406тЖТ   406тЖТ### 8.5 Concurrency
   407тЖТ   407тЖТ
   408тЖТ   408тЖТ- **Web Workers** тАФ offload heavy processing (image manipulation, PDF rendering) from main thread.
   409тЖТ   409тЖТ- **OffscreenCanvas** тАФ Canvas operations in Worker.
   410тЖТ   410тЖТ- **Comlink** тАФ thin wrapper simplifying Worker message passing.
   411тЖТ   411тЖТ
   412тЖТ   412тЖТ### 8.6 WASM (Phase 2+)
   413тЖТ   413тЖТ
   414тЖТ   414тЖТ- **Tesseract.js** тАФ OCR via WASM. Lazy-loaded only when OCR tool is used.
   415тЖТ   415тЖТ- **pdfium** тАФ PDF rendering via WASM (alternative to pdf-lib for some operations).
   416тЖТ   416тЖТ- **FFmpeg.wasm** тАФ video transcoding (Phase 3+).

   451тЖТ   451тЖТ## 10. Best Practices
   452тЖТ   452тЖТ
   453тЖТ   453тЖТ### 10.1 When Choosing a Library for a Tool
   454тЖТ   454тЖТ1. Check if a native Web API solves it (┬з8).
   455тЖТ   455тЖТ2. If not, search npm for maintained libraries with TypeScript types.
   456тЖТ   456тЖТ3. Evaluate bundle size; prefer <100KB. Lazy-load anything >200KB.
   457тЖТ   457тЖТ4. Verify license is permissive (MIT/Apache/BSD).
   458тЖТ   458тЖТ5. Add to ┬з5.3 table in this document via PR.
   459тЖТ   459тЖТ
   460тЖТ   460тЖТ### 10.2 When Hitting Edge Runtime Limitations
   461тЖТ   461тЖТ- Edge Runtime doesn't support: Node `fs`, `crypto` (use Web Crypto), `Buffer` (use Uint8Array), some `process` APIs.
   462тЖТ   462тЖТ- If a tool needs Node APIs, declare `execution: 'server'` (LOCK-02 decision matrix in `02_SAD` ┬зAD-04).
   463тЖТ   463тЖТ- Never silently fall back to Node runtime; explicit declaration required.
   464тЖТ   464тЖТ
   465тЖТ   465тЖТ### 10.3 When Upgrading Next.js
   466тЖТ   466тЖТ- Read the upgrade guide before starting.
   467тЖТ   467тЖТ- Create a dedicated branch; run full test suite.
   468тЖТ   468тЖТ- Manually test: every tool landing page, every tool workflow, admin panel, auth flows.
   469тЖТ   469тЖТ- Document any breaking changes in `19_DevelopmentGuideline`.
   470тЖТ   470тЖТ
   471тЖТ   471тЖТ### 10.4 When a Library Becomes Unmaintained
   472тЖТ   472тЖТ- Add to "deprecated" list in this doc.
   473тЖТ   473тЖТ- Open issue in backlog to find replacement.
   474тЖТ   474тЖТ- If critical, prioritize replacement in next sprint.
   475тЖТ   475тЖТ- Never remove without a replacement ready.
